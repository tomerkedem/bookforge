לפני שמריצים מודלים מ Hugging Face, כדאי להכין סביבת עבודה נקייה. בפרויקטי AI יש הרבה ספריות, גרסאות, קבצי הגדרות, tokens וקבצים כבדים. אם מתחילים להדביק קוד בתיקייה אקראית ולהתקין ספריות בלי סדר, מהר מאוד קשה להבין מה עובד, מה נשבר, ומה צריך לעלות לגיט.

המטרה בפרק הזה היא לבנות בסיס פשוט וברור:

סביבת Python נפרדת

קובץ התקנות מסודר

קובצי הגדרות שלא חושפים סודות

מבנה שאפשר להעלות לגיט בלי לכלוך מיותר

## יצירת virtual environment

ב Python לא כדאי להתקין ספריות ישירות על סביבת המחשב הכללית. לכל פרויקט עדיף ליצור סביבת עבודה נפרדת, כדי שהספריות של פרויקט אחד לא ישפיעו על פרויקט אחר.

ניצור virtual environment בשם .venv:

```bash
python -m venv .venv
```

הפקודה יוצרת תיקייה בשם venv. בתוך הפרויקט. התיקייה הזאת תכיל את סביבת Python המקומית של הפרויקט ואת הספריות שנחבר בהמשך.

עכשיו נפעיל את הסביבה:

```bash
.\.venv\Scripts\Activate.ps1
```

כאשר הסביבה פעילה, בדרך כלל נראה בתחילת שורת הפקודה:

```python
(.venv)
```

הסימון הזה חשוב. הוא אומר שהפקודות הבאות, כמו pip install, ירוצו בתוך סביבת הפרויקט ולא בסביבת Python הכללית של המחשב.

**אם PowerShell חוסם את ההפעלה**

לפעמים PowerShell חוסם הרצת scripts ומציג שגיאת הרשאות. במקרה כזה אפשר להריץ:

```bash
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

לאחר מכן מפעילים שוב:

```bash
.\.venv\Scripts\Activate.ps1
```

לא צריך להריץ את זה בכל פעם. בדרך כלל מספיק להגדיר פעם אחת עבור המשתמש הנוכחי.

**בדיקה שהסביבה פעילה**

כדי לוודא שאנחנו עובדים מתוך הסביבה הנכונה:

```bash
where python
```

או:

```bash
python --version
pip --version
```

אם הסביבה פעילה, הנתיב של Python או pip אמור להצביע לתוך תיקיית venv.

זו בדיקה קטנה, אבל היא חוסכת הרבה תקלות. הרבה פעמים מפתחים מתקינים ספרייה במקום אחד ומריצים Python ממקום אחר, ואז מקבלים שגיאה שהספרייה לא קיימת למרות שהיא “כבר הותקנה”.

**למה לא מעלים את .venv לגיט**

תיקיית .venv לא אמורה לעלות לגיט.

היא גדולה, מקומית, ותלויה במחשב, במערכת ההפעלה ובגרסת Python. אין טעם לשתף אותה. במקום זה מעלים את requirements.txt, שמאפשר לכל מי שמוריד את הפרויקט ליצור סביבת עבודה חדשה ולהתקין את אותן ספריות.

לכן בקובץ .gitignore נכניס בהמשך:

```python
.venv/
```

כך הקוד יישאר נקי, והסביבה המקומית תישאר מחוץ למאגר.



## התקנת הספריות המרכזיות

אחרי שיצרנו והפעלנו את סביבת העבודה, אפשר להתקין את הספריות שנשתמש בהן להרצת מודלים מ Hugging Face.

ניצור קובץ בשם:

requirements.txt

ובתוכו נכתוב:

```python
transformers
torch
accelerate
python-dotenv
```

לאחר מכן נריץ:

```bash
pip install -r requirements.txt
```

אפשר גם להתקין ישירות בלי קובץ, כך:

```bash
pip install transformers torch accelerate python-dotenv
```

אבל עדיף לעבוד עם requirements.txt, כי כך כל מי שיוריד את הפרויקט מהגיט יוכל לשחזר את סביבת העבודה בפקודה אחת.

**מה עושה כל ספרייה**

transformers היא הספרייה המרכזית של Hugging Face לעבודה עם מודלים. היא נותנת לנו מחלקות כמו AutoTokenizer, AutoModelForSequenceClassification ו AutoModelForCausalLM. בעזרתן אפשר לטעון מודל לפי שם, לטעון tokenizer מתאים, ולהריץ inference מתוך Python.

torch היא ספריית החישוב שעליה רצים הרבה מהמודלים. היא אחראית על tensors, חישובים מטריציוניים, הרצה על CPU או GPU, וחישוב יעיל של המודל. בקובץ sentiment.py, למשל, משתמשים ב torch.no_grad() כדי להריץ את המודל בלי לחשב gradients, כי בזמן inference אנחנו לא מאמנים את המודל אלא רק משתמשים בו.

accelerate עוזרת לטעון ולהריץ מודלים בצורה נוחה יותר, בעיקר כאשר עובדים עם GPU, מודלים גדולים או חלוקת עומס בין התקנים.

ההתקנה של accelerate היא חלק מהמעבר להרצת מודלים גדולים יותר.

python-dotenv מאפשרת לקרוא משתני סביבה מתוך קובץ .env. נשתמש בה כדי לשמור Hugging Face token מחוץ לקוד.

**בדיקה שהספריות הותקנו**

אחרי ההתקנה אפשר לבדוק שהכול נטען:

```bash
python
```

ואז בתוך Python:

```bash
import transformers
import torch
import accelerate

print("Transformers:", transformers.__version__)
print("Torch:", torch.__version__)
```

אם אין שגיאה, הספריות נטענו בהצלחה.

כדי לצאת מ Python:

```bash
exit()
```

## בדיקת GPU עם nvidia-smi

מודלים קטנים יכולים לרוץ גם על CPU, אבל מודלים גדולים יותר ירוצו הרבה יותר טוב עם GPU מתאים. אם יש במחשב כרטיס NVIDIA, אפשר לבדוק אותו בעזרת:

```bash
nvidia-smi
```

הפקודה מציגה את פרטי כרטיס המסך, כמות הזיכרון הגרפי, גרסת הדרייבר ותהליכים שמשתמשים ב GPU.

הפקודה הזאת מופיעה כחלק מבדיקת סביבת ההרצה המקומית לפני עבודה עם מודלים כבדים יותר.

אם הפקודה מחזירה מידע על הכרטיס, יש סיכוי טוב שאפשר להשתמש ב CUDA. אם היא לא מזוהה, זה יכול לקרות מכמה סיבות:

אין כרטיס NVIDIA במחשב.

הדרייבר לא מותקן או לא תקין.

הכרטיס קיים, אבל CUDA לא מוגדר כמו שצריך.

הטרמינל לא מזהה את הכלי.

**מה עושים אם אין GPU**

לא חייבים GPU כדי ללמוד את העקרונות. אפשר להריץ חלק מהמודלים הקטנים על CPU, אבל צריך לצפות לביצועים איטיים יותר.

הנקודה החשובה היא להתאים את הקוד לחומרה. למשל, בקובץ qwen.py מופיעה ההגדרה:

```python
device_map="cuda"
```

המשמעות היא שהקוד מניח שיש GPU עם CUDA. אם אין GPU, ההגדרה הזאת עלולה לגרום לשגיאה.

במחשב בלי GPU אפשר להתחיל ממודלים קטנים יותר, או לשנות את דרך הטעינה כך שתתאים ל CPU. לדוגמה, במקום להכריח cuda, אפשר לתת לספרייה לבחור התקן מתאים, או להריץ מודל קטן יותר.

בשלב לימודי, ההמלצה היא פשוטה:

- להתחיל ממודלים קטנים.

- לוודא שהקוד רץ.

- להבין את הזרימה.

ורק אחר כך לעבור למודלים גדולים יותר או להרצה מקומית עם Ollama.

**עדכון requirements בהמשך**

בשלב הזה requirements.txt מספיק להרצת הדוגמאות הראשונות עם Hugging Face.

בהמשך, כשנגיע ל Ollama ולשכבת התאימות של Anthropic, נוסיף גם:

```python
ollama
anthropic
```

הקובץ הסופי יוכל להיראות כך:

```python
transformers
torch
accelerate
python-dotenv
ollama
anthropic
```

אבל אין צורך למהר להוסיף הכול מיד. עדיף להתקין לפי שלבי הלמידה, כדי להבין למה כל ספרייה נכנסת לפרויקט.



## יצירת Hugging Face Access Token

חלק מהמודלים ב Hugging Face פתוחים לגמרי, וחלקם דורשים הרשאה, אישור תנאי שימוש או זיהוי משתמש. לכן כדאי ליצור Access Token כבר בתחילת העבודה.

ה Token מאפשר לקוד שלנו להזדהות מול Hugging Face כאשר הוא מוריד מודלים, טוען קבצים או ניגש למודלים שמחייבים הרשאה.

חשוב להבין: Token הוא סוד. הוא דומה למפתח גישה. לכן לא כותבים אותו ישירות בתוך קובץ Python.

לא כך:

```python
HF_TOKEN = "hf_xxxxxxxxxxxxxxxxx"
```

הבעיה ברורה: אם הקובץ עולה לגיט, ה Token נחשף. גם אם מוחקים אותו אחר כך, הוא עלול להישאר בהיסטוריית commits.

**שמירת Token בקובץ .env**

הדרך הנכונה היא לשמור את ה Token בקובץ מקומי בשם:

```python
.env
```

ובתוכו:

```python
HF_TOKEN=your_real_hugging_face_token_here
```

הקובץ הזה נשאר רק במחשב המקומי. הוא לא עולה לגיט.

כדי לקרוא אותו מתוך Python, נשתמש בספרייה python-dotenv שהתקנו במנה הקודמת.

דוגמה בסיסית:

```python
import os
from dotenv import load_dotenv

load_dotenv()

hf_token = os.getenv("HF_TOKEN")

if not hf_token:
    raise RuntimeError("HF_TOKEN is missing. Please define it in your .env file.")

print("HF_TOKEN loaded:", bool(hf_token))
```

הרעיון פשוט: הקוד יודע שצריך משתנה בשם HF_TOKEN, אבל הערך האמיתי נשמר מחוץ לקוד.

**יצירת .env.example**

כדי שמי שיוריד את הפרויקט מהגיט יבין אילו משתנים צריך להגדיר, יוצרים קובץ נוסף:

```python
.env.example
```

ובתוכו:

```bash
HF_TOKEN=put_your_hugging_face_token_here
```

את הקובץ הזה כן מעלים לגיט, כי אין בו סוד אמיתי. הוא משמש כתבנית.

כל מי שמוריד את הפרויקט יוכל להעתיק אותו:

```bash
copy .env.example .env
```

ואז להכניס לתוך .env את ה Token האמיתי שלו.

**למה אסור להעלות Tokens לגיט**

Tokens לא אמורים להופיע בקוד, בקובצי README, בצילומי מסך, בלוגים או בקבצים שעולים לגיט.

ברגע ש Token נחשף, מישהו אחר יכול להשתמש בו. גם אם מדובר בפרויקט פרטי, זו עדיין פרקטיקה לא נכונה. בפרויקטים אמיתיים, חשיפת tokens עלולה לגרום לשימוש לא מורשה, עלויות לא צפויות, גישה למשאבים, או צורך לבטל ולהחליף מפתחות.

לכן הכלל פשוט:

קוד עולה לגיט.

תבניות עולות לגיט.

סודות לא עולים לגיט.

**מבנה מומלץ ל .gitignore**

ניצור קובץ בשם:

```python
.gitignore
```

ובתוכו:

```python
.venv/
.env
__pycache__/
*.pyc
.cache/
```

המשמעות:

.venv/ לא עולה לגיט, כי זו סביבת עבודה מקומית.

.env לא עולה לגיט, כי הוא מכיל secrets.

__pycache__/ ו *.pyc לא עולים לגיט, כי אלה קבצים זמניים שפייתון יוצר.

.cache/ לא עולה לגיט, כי ספריות AI יכולות לשמור שם קבצי cache גדולים.

בהמשך, אם נוריד מודלים כבדים או נשמור תוצרים זמניים, אפשר להוסיף גם:

models/ 
outputs/

לא תמיד חייבים להתעלם מ models/. אם יש שם קבצים קטנים וחשובים לפרויקט, אפשר להחליט אחרת. אבל ברוב המקרים קובצי מודלים כבדים לא שייכים לגיט רגיל.

**בדיקה שהכול מוכן**

בסיום ההכנה, מבנה הפרויקט אמור להיראות בערך כך:

```bash
lesson-07-open-source-models-local-llms/
  README.md
  requirements.txt
  .env
  .env.example
  .gitignore

  examples/
  homework/
```

כאשר נעלה לגיט, הקובץ .env ותיקיית .venv לא יופיעו ב commit.

אפשר לבדוק בעזרת:

```bash
git status
```

אם רואים את .env ברשימת הקבצים להעלאה, סימן ש .gitignore לא מוגדר נכון או שהקובץ כבר נכנס בעבר ל tracking. במקרה כזה צריך לעצור ולא להעלות את הקוד עד שמתקנים.

**סביבת העבודה מוכנה עכשיו להרצת המודלים הראשונים.**

יש לנו תיקיית פרויקט מסודרת, סביבת Python נפרדת, קובץ התקנות, בדיקת GPU, ניהול Token דרך .env, וקובץ .gitignore שמונע העלאה של קבצים רגישים או מיותרים.

מכאן אפשר לעבור לקוד הראשון: מודל סיווג רגשות עם Transformers, שבו נראה איך טקסט נכנס למודל, איך tokenizer מכין את הקלט, איך המודל מחזיר logits, ואיך הופכים אותם לתווית כמו Positive או Negative.
