# Ollama: הרצת מודלים מקומית

אחרי שראינו איך להריץ מודלים דרך Transformers, מגיעים לכלי שמפשט מאוד את העבודה עם מודלים מקומיים: Ollama.

ב Transformers אנחנו עובדים ישירות מתוך Python. אנחנו טוענים tokenizer, טוענים מודל, דואגים ל device, מתמודדים עם tensors, ומריצים את המודל מתוך הקוד. זו דרך מצוינת ללמידה, כי היא חושפת מה באמת קורה מאחורי הקלעים.

אבל כאשר רוצים להריץ LLM מקומי בצורה נוחה יותר, לא תמיד רוצים לנהל את כל זה ידנית. לפעמים רוצים פקודה פשוטה שמורידה מודל, מריצה אותו, ופותחת לנו ממשק עבודה מקומי.

כאן Ollama נכנס לתמונה.

## מהו Ollama

Ollama הוא כלי שמאפשר להוריד, להריץ ולנהל מודלי שפה מקומיים בצורה פשוטה יחסית.

במקום לחפש קבצים, להבין פורמטים, לכתוב קוד טעינה, להגדיר שרת, ולבנות בעצמנו API סביב המודל, Ollama נותן דרך קצרה יותר:

```bash
ollama run llama3
```

או למשל:

```bash
ollama run qwen2.5
```

הפקודה מורידה את המודל אם הוא עדיין לא קיים במחשב, טוענת אותו, ומאפשרת להתחיל לדבר איתו ישירות מהטרמינל.

זו חוויה שונה מאוד מהרצה דרך Transformers. במקום שאנחנו נכתוב את כל קוד הטעינה, Ollama מנהל עבורנו חלק גדול מהשכבה התפעולית.

**למה Ollama דומה ל Docker for LLMs**

דרך טובה להבין את Ollama היא לחשוב עליו כעל משהו שדומה ל Docker for LLMs.

ההשוואה אינה מושלמת, אבל היא מאוד שימושית.

ב Docker אנחנו יכולים להריץ שירותים בצורה מבודדת יחסית בעזרת פקודה אחת. במקום להתקין ידנית את כל התלויות של Redis, PostgreSQL או אפליקציה מסוימת, אנחנו מריצים image מוכן.

באופן דומה, Ollama מאפשר להריץ מודלים בצורה ארוזה ונוחה. במקום להתעסק ידנית עם כל פרטי המודל, הפורמט, ההרצה וה API המקומי, אנחנו מבקשים מ Ollama להריץ מודל מסוים.

לדוגמה:

```bash
ollama run gemma3:4b
```

המשמעות היא: תוריד את המודל אם צריך, תטען אותו, ותן לי להתחיל לעבוד מולו.

כמו Docker, גם Ollama לא “מעלים” את המורכבות לגמרי. עדיין צריך להבין איזה מודל בוחרים, כמה הוא גדול, האם החומרה מספיקה, ומה מגבלות האיכות שלו. אבל הוא מוריד הרבה מאוד חסמים טכניים בתחילת הדרך.

## איך מורידים ומריצים מודל בפקודה אחת

הפקודה המרכזית היא:

```bash
ollama run <model-name>
```

לדוגמה:

```bash
ollama run llama3
```

או:

```bash
ollama run mistral
```

או:

```bash
ollama run qwen2.5
```

אם המודל לא נמצא במחשב, Ollama יוריד אותו. אם הוא כבר קיים, Ollama ישתמש בעותק המקומי.

לאחר ההרצה, בדרך כלל נקבל prompt בטרמינל ונוכל להתחיל לכתוב למודל:

```bash
>>> Explain what a local LLM is in one paragraph.
```

המודל יענה מתוך המחשב המקומי, בלי שנצטרך לכתוב קוד Python ובלי לפנות ל API חיצוני.

אפשר גם להוריד מודל בלי להריץ אותו מיד:

```bash
ollama pull llama3
```

ולראות אילו מודלים קיימים אצלנו מקומית:

```bash
ollama list
```

אם רוצים למחוק מודל שכבר לא צריך:

```bash
ollama rm llama3
```

הפקודות האלה הופכות את ניהול המודלים לפשוט בהרבה, במיוחד בשלבי ניסוי ולמידה.



## מה היתרון של local execution

היתרון המרכזי של Ollama הוא local execution, כלומר הרצת המודל על המחשב או השרת שלנו.

כאשר המודל רץ מקומית, הקלט לא חייב להישלח לספק חיצוני. זה חשוב כאשר עובדים עם טקסטים פנימיים, קוד מקור, מסמכים ארגוניים, תהליכי פיתוח או מידע שלא רוצים להוציא החוצה.

יתרון נוסף הוא שליטה. אנחנו יודעים איזה מודל הורדנו, איזו גרסה רצה, ואפשר להפעיל אותו שוב ושוב באותה סביבה. זה לא מבטיח שהתשובות תמיד יהיו זהות, אבל זה נותן שליטה טובה יותר מאשר שירות חיצוני שמשתנה לפי החלטות הספק.

יש גם יתרון בעלויות. אחרי שהמודל הורד ורץ מקומית, לא משלמים לפי כל טוקן לספק חיצוני. עדיין יש עלות חומרה, חשמל, זמן ריצה ותחזוקה, אבל מבנה העלות שונה. עבור ניסויים רבים, בדיקות פנימיות או שימוש מקומי, זה יכול להיות נוח מאוד.

יתרון נוסף הוא מהירות פיתוח. קל לבדוק מודלים שונים בלי לבנות תשתית מסובכת. מריצים פקודה, בודקים, מחליפים מודל, משווים תוצאות.

לדוגמה:

```bash
ollama run llama3
ollama run mistral
ollama run gemma3:4b
ollama run qwen2.5
```

כך אפשר לקבל תחושה ראשונית לגבי התנהגות של כמה מודלים, לפני שמחליטים איזה מהם מתאים יותר להמשך עבודה.

**מה חשוב לזכור**

Ollama מקל מאוד על הרצה מקומית, אבל הוא לא פוטר אותנו מחשיבה הנדסית.

עדיין צריך לשאול:

- האם המודל מתאים למשימה?

- האם הוא מספיק איכותי?

- האם הוא רץ מספיק מהר על החומרה שלנו?

- האם יש מספיק זיכרון?

- האם הרישיון מתאים?

- האם הוא מתאים לשפה ולדומיין שלנו?

כלומר, Ollama פותר הרבה מהקושי של ההרצה, אבל לא מחליף את תהליך הבחירה והבדיקה של המודל.



## איך Ollama פותח שרת מקומי

אחד הדברים החשובים ב Ollama הוא שהוא לא רק מריץ מודל בתוך הטרמינל. מאחורי הקלעים הוא גם פותח שירות מקומי, כלומר שרת שרץ על המחשב שלנו ומאפשר לאפליקציות לדבר עם המודל דרך HTTP.

ברירת המחדל הנפוצה היא ש Ollama מאזין בכתובת מקומית:

http://localhost:11434

המשמעות היא שהמודל רץ אצלנו במחשב, אבל אפשר לגשת אליו כמו אל API. לא חייבים להישאר בתוך חלון הטרמינל. אפשר לכתוב סקריפט Python, אפליקציית backend, כלי פנימי, או Agent, ולשלוח בקשות למודל המקומי.

זה שינוי חשוב מאוד.

במקום שמודל מקומי יהיה רק כלי אינטראקטיבי לשיחה בטרמינל, הוא הופך לרכיב שאפשר לחבר לאפליקציה. מבחינת הארכיטקטורה, Ollama מתפקד כשכבת הרצה מקומית למודלים.

אפשר לבדוק שהשרת עובד באמצעות פקודה כמו:

```bash
ollama list
```

ואפשר גם להריץ מודל:

```bash
ollama run llama3
```

כאשר מודל רץ, Ollama יודע לקבל בקשות, להריץ את המודל המתאים, ולהחזיר תשובה.

## איך אפליקציה יכולה לדבר עם מודל מקומי

ברגע שיש שרת מקומי, אפליקציה יכולה לשלוח אליו בקשות.

הרעיון דומה לקריאה ל API חיצוני, רק שהכתובת היא מקומית. במקום לשלוח טקסט לספק ענן, שולחים אותו ל Ollama שרץ על המחשב או השרת שלנו.

דוגמה רעיונית:היתרון הוא שהאפליקציה לא חייבת לדעת איך המודל נטען, איפה הקבצים שלו, או באיזה פורמט הוא שמור. היא רק שולחת בקשה ומקבלת תשובה.
<img src="/Lesson-7-open-source-models-local-llms/assets/image-02.png" alt="image-02.png" width="710" height="112" />


ב Python אפשר לעבוד מול Ollama בכמה דרכים. דרך אחת היא להשתמש בספריית ollama:

```python
import ollama

response = ollama.chat(
    model="llama3",
    messages=[
        {
            "role": "user",
            "content": "Explain what a local LLM is in one short paragraph."
        }
    ]
)

print(response["message"]["content"])
```

הקוד הזה דומה מאוד לעבודה עם מודלי chat אחרים. יש model, יש רשימת messages, ויש תשובה שמכילה את תוכן ההודעה של המודל.

ההבדל הוא שהמודל רץ מקומית.

**למה זה חשוב מבחינה הנדסית**

היכולת לדבר עם מודל מקומי דרך API מקומי נותנת לנו גמישות.

אפשר לבנות אפליקציה שמדברת עם Ollama בזמן פיתוח, ובשלב אחר להחליף את שכבת המודל לספק חיצוני. אפשר גם לעשות להפך: להתחיל עם API חיצוני, ובהמשך להעביר חלק מהמשימות למודל מקומי.

הדבר הנכון הוא לא לפזר קריאות ישירות ל Ollama בכל מקום בקוד, אלא לבנות שכבת שירות פנימית.

לדוגמה:

```python
CodeReviewService
SummaryService
ClassificationService
LocalLLMClient
```

כך שאר האפליקציה לא צריכה לדעת אם מאחורי הקלעים רץ llama3, gemma, qwen, או מודל אחר. היא רק מבקשת פעולה: סכם, בדוק קוד, ענה על שאלה, צור טקסט.

גישה כזאת מקלה להחליף מודל בהמשך, להריץ בדיקות, למדוד איכות, ולעבור בין מודלים בלי לשכתב את כל המערכת.

## מודלים נתמכים

Ollama תומך במגוון משפחות מודלים פופולריות. בין השמות הנפוצים אפשר למצוא:

- Llama

- Mistral

- Gemma

- Phi

- Qwen

- DeepSeek

- Code Llama

המודלים מגיעים בגדלים שונים ובגרסאות שונות. לפעמים נראה שם כמו: gemma3:4b או: llama3:8b

המספרים האלה מתארים בדרך כלל את גודל המודל במיליארדי פרמטרים. ככל שהמודל גדול יותר, הוא יכול להיות חזק יותר, אבל הוא גם דורש יותר זיכרון וחומרה טובה יותר.

לכן לא בוחרים מודל רק לפי השם. צריך לבדוק אם הוא רץ בצורה סבירה על המחשב שלנו.

לדוגמה:

```bash
ollama run gemma3:4b
```

יכול להתאים יותר למחשב מקומי מאשר מודל גדול בהרבה.

מצד שני, מודל קטן יותר לא תמיד יספק את אותה איכות במשימות מורכבות. לכן הבחירה היא תמיד איזון בין איכות, מהירות, זיכרון וחומרה.

**בדיקת מודלים שונים במהירות**

אחד היתרונות הגדולים של Ollama הוא שאפשר להשוות מודלים בצורה מהירה יחסית.

לדוגמה, אפשר להריץ:

```python
ollama run llama3
```

ואז לשאול:

```python
Explain what RAG is in simple terms.
```

לאחר מכן להריץ:

```python
ollama run mistral
```

ולשאול את אותה שאלה.

ואז:

```python
ollama run qwen2.5
```

ולבדוק שוב.

כך אפשר לקבל תחושה ראשונית לגבי סגנון התשובה, איכות ההסבר, מהירות ההרצה, ויכולת המודל להבין את ההוראה.

זו עדיין לא בדיקה מסודרת כמו benchmark פנימי, אבל זו דרך טובה לסינון ראשוני. לאחר שמוצאים מועמדים טובים, אפשר להריץ אותם על סט שאלות קבוע ולמדוד בצורה עקבית יותר.

**איך זה משתלב עם מה שראינו קודם**

בפרקים הקודמים עבדנו עם Transformers.

ראינו את הפרטים הפנימיים: tokenizer, מודל, tensors, device, generation.

Ollama מסתיר חלק מהמורכבות הזאת. הוא לא מבקש מאיתנו לכתוב בכל פעם קוד טעינה ידני. במקום זה הוא נותן ממשק פשוט יותר להרצת מודלים מקומיים.

אבל ההבנה של Transformers עדיין חשובה. היא עוזרת להבין מה קורה מאחורי הקלעים:

- למה מודל צריך tokenizer

- למה מודל גדול דורש זיכרון

- למה יצירת טקסט איטית יותר מסיווג

- למה אורך תשובה משפיע על זמן ריצה

- למה חשוב לבחור מודל שמתאים למשימה

כלומר, Ollama עושה את העבודה נוחה יותר, אבל לא מבטל את הצורך להבין מודלים.

## מתי Ollama מתאים

Ollama מתאים במיוחד כאשר רוצים להתחיל לעבוד עם מודלים מקומיים בלי להסתבך יותר מדי עם קבצי weights, פורמטים, טעינת tokenizer, הגדרות device וקוד הרצה ידני.

במקום לכתוב הרבה קוד רק כדי לגרום למודל לעלות, אפשר להריץ פקודה אחת:

```bash
ollama run gemma3:4b
```

ולהתחיל לבדוק את המודל מיד.

זו נקודת פתיחה מצוינת ללמידה, ניסויים, בדיקות פנימיות, עבודה עם prompts, השוואת מודלים, ובניית כלים קטנים שמדברים עם LLM מקומי.

Ollama מתאים במיוחד במקרים כאלה:

- כאשר רוצים לבדוק מודל במהירות.

- כאשר רוצים להריץ מודל בלי לשלוח מידע לספק חיצוני.

- כאשר רוצים לחבר אפליקציה מקומית למודל דרך API פשוט.

- כאשר רוצים לבדוק כמה מודלים שונים באותה סביבה.

- כאשר רוצים לבנות כלי פנימי קטן, למשל מסכם טקסטים, עוזר קוד, או צ'אט מקומי.

- כאשר רוצים ללמוד איך מודלים מקומיים מתנהגים בפועל.

היתרון הגדול הוא מהירות ההגעה להרצה ראשונה. לא צריך לבנות תשתית מלאה. לא צריך לכתוב שרת. לא צריך להבין את כל פרטי הפורמט של המודל כבר מההתחלה. Ollama נותן שכבת נוחות שמאפשרת להתמקד בהתנהגות של המודל.

## מתי Ollama פחות מתאים

למרות הנוחות, Ollama אינו פתרון קסם לכל מצב.

הוא פחות מתאים כאשר צריך שליטה עמוקה מאוד בתהליך ההרצה. למשל, אם רוצים לשלוט באופן מדויק בשכבות המודל, לבצע אופטימיזציות מיוחדות, לנהל inference מתקדם, להריץ מודלים בצורה מבוזרת, או לשלב תשתית production מורכבת, ייתכן שיהיה צורך בכלים אחרים.

הוא גם פחות מתאים כאשר הארגון צריך זמינות גבוהה, scaling, הרשאות מורכבות, ניטור מתקדם, תורים, עומסים גדולים, או פריסה מבוקרת בין סביבות. אפשר לבנות סביב Ollama שכבות כאלה, אבל זה כבר דורש עבודה ארכיטקטונית נוספת.

בנוסף, איכות המודל עדיין תלויה במודל עצמו. העובדה שמודל רץ דרך Ollama לא הופכת אותו לחכם יותר. אם בחרנו מודל קטן מדי, חלש מדי, או לא מתאים לשפה ולמשימה, נקבל תוצאות בהתאם.

גם חומרה נשארת שיקול משמעותי. מודל גדול דורש יותר RAM או VRAM. אם אין מספיק זיכרון, המודל ירוץ לאט מאוד או לא ירוץ בכלל. גם עם Ollama, אי אפשר לעקוף לגמרי את המגבלות הפיזיות של המחשב.

**Ollama מול Transformers**

כדאי להבין את ההבדל בין עבודה עם Transformers לבין עבודה עם Ollama.

Transformers נותן שליטה עמוקה יותר מתוך Python. אנחנו רואים את ה tokenizer, את המודל, את tensors, את device, את הקריאה ל generate, ואת הפלט הגולמי. זו דרך מצוינת ללמוד איך הדברים עובדים באמת.

Ollama נותן שכבת הרצה נוחה יותר. הוא מסתיר חלק מהמורכבות ומאפשר להריץ מודלים בפקודות קצרות או דרך API מקומי.

אפשר לחשוב על זה כך:

- **Transformers:** יותר שליטה, יותר קוד, יותר חשיפה לפרטים הפנימיים.

- **Ollama:** פחות קוד, הרצה מקומית נוחה, API פשוט, התחלה מהירה.

אין כאן מנצח מוחלט. אלה כלים שונים לצרכים שונים.

אם רוצים ללמוד לעומק איך מודל נטען ומייצר טקסט, Transformers מצוין.

אם רוצים להריץ מודלים מקומיים מהר ולחבר אותם לאפליקציה, Ollama נוח מאוד.

Ollama כחלק מארכיטקטורת AI

הדרך הנכונה לחשוב על Ollama היא לא כתחליף לכל עולם ה AI, אלא כרכיב אפשרי בארכיטקטורה.

לדוגמה, אפשר לבנות מערכת שבה:

מודל מקומי דרך Ollama מטפל במשימות פשוטות או רגישות.

מודל חיצוני חזק מטפל במשימות מורכבות יותר.

מערכת RAG מספקת הקשר מתוך מסמכים פנימיים.

שכבת backend מחליטה לאיזה מודל לפנות לפי סוג המשימה.

בצורה כזו, Ollama הופך לאחד ממנועי ההרצה האפשריים של המערכת.

הקוד העסקי לא חייב לדעת אם מאחורי הקלעים רץ gemma, llama, qwen או מודל חיצוני. עדיף לבנות שכבת שירות פנימית שמסתירה את פרטי המודל ומחזירה לאפליקציה תשובה במבנה קבוע.

לדוגמה:

```bash
User Request
-> Application Service
-> LLM Client
-> Ollama Local API
-> Local Model
-> Response
```

גישה כזו מאפשרת להחליף מודל בלי לשכתב את כל האפליקציה.

**דוגמת זרימה מעשית**

נניח שאנחנו רוצים לבנות כלי מקומי שמסביר קטעי קוד.

במקום לשלוח את הקוד לספק חיצוני, אפשר להריץ מודל מקומי: ollama run qwen2.5

ואז מתוך Python לשלוח לו הודעה:

```python
import ollama

response = ollama.chat(
    model="qwen2.5",
    messages=[
        {
            "role": "user",
            "content": "Explain this C# method in simple terms:\n\npublic int Add(int a, int b) { return a + b; }"
        }
    ]
)

print(response["message"]["content"])
```

התוצאה מתקבלת מהמודל המקומי. אין כאן קריאה לספק חיצוני. זה לא אומר שהפתרון מתאים אוטומטית לכל מערכת, אבל זה מדגים את הכוח של הרצה מקומית: קל לחבר מודל לקוד ולבדוק רעיונות מהר.

**מה צריך לבדוק לפני שימוש אמיתי**

לפני שמכניסים Ollama לתהליך עבודה רציני, כדאי לבדוק כמה דברים:

- האם המודל רץ מספיק מהר על החומרה שלנו.

- האם איכות התשובות מספיקה למשימה.

- האם המודל מתאים לשפה ולדומיין.

- האם הרישיון מתאים לשימוש המתוכנן.

- האם יש צורך בהרשאות וגישה מבוקרת לשרת המקומי.

- האם יש צורך בלוגים, ניטור ותיעוד.

- האם יש דרך לבדוק רגרסיות כאשר מחליפים מודל.

כלומר, Ollama מקל על ההרצה, אבל לא מבטל את עבודת הבחירה, המדידה והאחריות.
