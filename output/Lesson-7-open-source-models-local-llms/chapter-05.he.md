## טעינת tokenizer

השלב הראשון בקוד הוא לייבא את המחלקות שנצטרך מתוך ספריית transformers:

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import torch
```

שתי המחלקות המרכזיות כאן הן:

```python
AutoTokenizer
AutoModelForSequenceClassification
```

השם Auto חשוב. הוא אומר לספרייה: קחי את שם המודל, בדקי ב Hugging Face איזה tokenizer ואיזה מבנה מודל מתאימים לו, וטעני את המחלקה הנכונה באופן אוטומטי.

לאחר מכן מוגדר שם המודל:

```python
model_name = "tabularisai/robust-sentiment-analysis"
```

שם המודל הוא למעשה הנתיב של המודל ב Hugging Face. החלק הראשון הוא שם הארגון או המשתמש שפרסם את המודל, והחלק השני הוא שם המודל עצמו.

לאחר מכן טוענים את ה tokenizer:

```python
tokenizer = AutoTokenizer.from_pretrained(model_name)
```

כאן חשוב להבין משהו בסיסי: מודל לא עובד ישירות עם טקסט כמו שאנחנו קוראים אותו. הוא לא מקבל משפט רגיל ומבין מילים בצורה אנושית. לפני שהטקסט נכנס למודל, צריך להפוך אותו לייצוג מספרי.

זו בדיוק העבודה של ה tokenizer.

ה tokenizer מקבל טקסט כמו:

```python
I absolutely loved this movie!
```

ומפרק אותו ל tokens. לאחר מכן כל token מומר למספר מזהה. המודל מקבל את המספרים האלה, לא את המילים עצמן.

כלומר, ה tokenizer הוא שכבת התרגום בין שפה אנושית לבין קלט מספרי שהמודל יודע לעבד.

חשוב להשתמש ב tokenizer שמגיע עם אותו מודל בדיוק. המודל אומן עם צורת פירוק מסוימת של טקסט. אם נשתמש ב tokenizer אחר, הטקסט עלול להתפרק אחרת, המספרים יהיו שונים, והמודל יקבל קלט שלא תואם למה שהוא ראה בזמן האימון.

לכן השורה הזאת:

```python
AutoTokenizer.from_pretrained(model_name)
```

אינה רק טעינה טכנית. היא מבטיחה שהקלט שנכין יתאים למודל שבחרנו.

## טעינת מודל Sequence Classification

אחרי שטענו את ה tokenizer, טוענים את המודל עצמו:

```python
model = AutoModelForSequenceClassification.from_pretrained(model_name)
```

כאן כבר ברור איזה סוג מודל אנחנו טוענים: מודל לסיווג רצף טקסט.

Sequence פירושו רצף, ובמקרה שלנו הרצף הוא משפט או קטע טקסט.

Classification פירושו סיווג, כלומר בחירה של מחלקה מתוך רשימה מוגדרת.

המודל הזה לא נועד לכתוב תשובה חופשית. הוא לא אמור להמשיך טקסט, לנהל שיחה או לנסח הסבר. הוא אמור לקבל טקסט ולהחזיר ציונים עבור כמה מחלקות אפשריות.

במקרה של הקובץ שלנו, המחלקות הן:

```python
{
    0: "Very Negative",
    1: "Negative",
    2: "Neutral",
    3: "Positive",
    4: "Very Positive"
}
```

כלומר, המודל מחזיר הערכה מספרית לכל אחת מחמש האפשרויות, והקוד בוחר את האפשרות שקיבלה את הציון הגבוה ביותר.

זו הסיבה שהמחלקה המתאימה היא:

```python
AutoModelForSequenceClassification
```

ולא למשל:

```python
AutoModelForCausalLM
```

AutoModelForCausalLM מתאים למודלים שמייצרים טקסט, כמו בקובץ qwen.py. שם המודל מקבל prompt ומייצר המשך. כאן אנחנו לא רוצים המשך. אנחנו רוצים סיווג.

ההבדל הזה חשוב מאוד:

Sequence Classification: מקבל טקסט ומחזיר קטגוריה

Causal Language Model: מקבל טקסט ומייצר טקסט חדש

לכן הקובץ sentiment.py הוא דוגמה טובה להתחלה. הוא ממוקד, צפוי יותר ממודל שיחה, ומאפשר להבין בצורה נקייה את השלבים הבסיסיים של inference.

## מה קורה בשורת tokenizer(...)

בתוך הפונקציה predict_sentiment(text) מופיעה אחת השורות החשובות בקובץ:

```python
inputs = tokenizer(
    text.lower(),
    return_tensors="pt",
    truncation=True,
    padding=True,
    max_length=512
)
```

זו השורה שבה הטקסט האנושי הופך לקלט שהמודל באמת יודע לקבל.

הפונקציה מקבלת מחרוזת רגילה, למשל:

```python
I absolutely loved this movie!
```

אבל המודל לא עובד ישירות עם המילים האלה. הוא צריך לקבל מספרים. לכן ה tokenizer מבצע כמה פעולות:

הוא מנקה או מכין את הטקסט לפי הכללים של המודל.

הוא מפרק את הטקסט ל tokens.

הוא ממיר כל token למזהה מספרי.

הוא מוסיף מידע עזר שהמודל צריך, למשל אילו tokens הם חלק אמיתי מהטקסט ואילו הם padding.

הוא מחזיר מבנה נתונים ש PyTorch יודע להעביר למודל.

בקוד שלנו יש שימוש ב:

```python
text.lower()
```

כלומר, לפני השליחה ל tokenizer, הטקסט מומר לאותיות קטנות. למשל:

```python
I absolutely loved this movie!
```

יהפוך ל:

```python
i absolutely loved this movie!
```

במודלים מסוימים זה יכול לעזור להפחית שונות בין מילים כמו Movie ו movie. מצד שני, לא תמיד נכון להוריד אותיות גדולות. במודלים מסוימים האותיות הגדולות יכולות לשאת משמעות. לכן זה לא כלל אוניברסלי, אלא החלטה שנלקחה בקוד הזה. בקוד לימודי זה בסדר, אבל בפרויקט אמיתי כדאי לבדוק אם ההמרה ל lowercase באמת משפרת תוצאות או פוגעת בהן.

**מה המשמעות של return_tensors="pt"**

הפרמטר:

```python
return_tensors="pt"
```

אומר ל tokenizer להחזיר את הקלט בפורמט של PyTorch tensors.

Tensor הוא מבנה נתונים מספרי. אפשר לחשוב עליו כהכללה של מספר, וקטור, מטריצה או מערך רב ממדי. מודלים לא מקבלים רשימות מילים. הם מקבלים tensors שמכילים מספרים.

כאשר כותבים "pt", הכוונה היא PyTorch. לכן הפלט של tokenizer יתאים ישירות למודל שנטען עם transformers ו torch.

בפועל, inputs יהיה מבנה שמכיל בדרך כלל שדות כמו:

```python
input_ids
attention_mask
```

input_ids הם המזהים המספריים של ה tokens.

attention_mask אומר למודל אילו מקומות הם טקסט אמיתי ואילו מקומות הם padding.

**מה המשמעות של truncation=True**

הפרמטר:

```python
truncation=True
```

אומר ל tokenizer: אם הטקסט ארוך מדי, חתוך אותו לאורך המקסימלי שמותר.

בקוד שלנו האורך המקסימלי מוגדר כך:

```python
max_length=512
```

כלומר, אם הטקסט מתפרק ליותר מ 512 tokens, החלק שמעבר לכך ייחתך.

זה חשוב כי לכל מודל יש מגבלת אורך. מודל סיווג כזה לא יכול לקבל טקסט באורך בלתי מוגבל. אם נשלח לו מסמך ארוך מדי בלי חיתוך, נקבל שגיאה או התנהגות לא רצויה.

אבל צריך לשים לב: חיתוך טקסט יכול להשפיע על התוצאה. אם החלק החשוב נמצא בסוף הטקסט ונחתך, המודל יסווג לפי מידע חלקי בלבד. לכן במסמכים ארוכים לא תמיד מספיק להשתמש ב truncation=True. לפעמים צריך לחלק את הטקסט לחלקים, להריץ את המודל על כל חלק, ואז לאחד תוצאות.

**מה המשמעות של padding=True**

הפרמטר:

```python
padding=True
```

אומר ל tokenizer להשלים טקסטים קצרים לאורך אחיד.

בדוגמה שלנו אנחנו מעבירים בכל פעם טקסט אחד, ולכן padding פחות דרמטי. אבל אם מעבירים כמה משפטים יחד ב batch, המשפטים יכולים להיות באורכים שונים. מודלים אוהבים לקבל batch בצורה מלבנית, כלומר אותו אורך לכל הדוגמאות. לכן מוסיפים padding למשפטים הקצרים.

לדוגמה, אם משפט אחד מתפרק ל 10 tokens ומשפט אחר ל 20 tokens, המשפט הקצר יקבל tokens ריקים מיוחדים עד לאורך 20. ה attention_mask יגיד למודל אילו tokens אמיתיים ואילו רק padding.

כך המודל יודע להתעלם מהחלקים המלאכותיים.

**מה המשמעות של max_length=512**

הפרמטר:

```python
max_length=512
```

קובע את האורך המקסימלי של הקלט אחרי tokenization.

חשוב לשים לב: מדובר ב tokens, לא במילים. Token יכול להיות מילה שלמה, חלק ממילה, סימן פיסוק או יחידה אחרת לפי ה tokenizer של המודל.

לכן 512 tokens אינם בהכרח 512 מילים.

במודל סיווג רגשות, 512 tokens בדרך כלל מספיקים למשפטים, ביקורות קצרות או פסקאות. אבל אם נרצה לסווג מסמכים ארוכים, נצטרך אסטרטגיה אחרת.

**למה משתמשים ב torch.no_grad()**

אחרי הכנת הקלט, הקוד מריץ את המודל:

```python
with torch.no_grad():
    outputs = model(**inputs)
```

החלק החשוב כאן הוא:

```python
torch.no_grad()
```

כדי להבין למה משתמשים בו, צריך להבחין בין שני מצבים:

אימון 
שימוש במודל

בזמן אימון, המודל צריך לחשב gradients. אלה הערכים שמשמשים לעדכון המשקלים של המודל. זה תהליך יקר מבחינת זיכרון וחישוב.

אבל כאן אנחנו לא מאמנים את המודל. אנחנו רק משתמשים בו כדי לקבל תחזית. זה נקרא inference.

בזמן inference אין צורך לחשב gradients, ולכן משתמשים ב:

```python
with torch.no_grad():
```

היתרון הוא שההרצה קלה יותר, צורכת פחות זיכרון, ובדרך כלל מהירה יותר.

זו פרקטיקה נכונה כמעט בכל פעם שמריצים מודל רק לצורך חיזוי ולא לצורך אימון.

בקובץ sentiment.py זה מופיע בדיוק סביב קריאת המודל, כדי לוודא שההרצה מתבצעת במצב חיזוי ולא במצב אימון.

**מה עושה model(**inputs)**

השורה:

```python
outputs = model(**inputs)
```

מעבירה את הקלטים שה tokenizer הכין אל המודל.

הסימון:

```python
**inputs
```

פותח את המילון שה tokenizer החזיר ומעביר את הערכים כפרמטרים למודל.

אם inputs כולל למשל:

```python
{
    "input_ids": ...,
    "attention_mask": ...
}
```

אז הקריאה:

```python
model(**inputs)
```

דומה לכתיבה:

```python
model(
    input_ids=...,
    attention_mask=...
)
```

זו צורת כתיבה נפוצה בפייתון כאשר רוצים להעביר מילון של פרמטרים לפונקציה.

## מהם logits

אחרי שהמודל רץ, הקוד לוקח מתוך הפלט את:

```python
logits = outputs.logits
```

logits הם הציונים הגולמיים שהמודל נותן לכל מחלקה.

חשוב להבין: logits הם לא הסתברויות.

הם יכולים להיות מספרים חיוביים, שליליים, גדולים או קטנים. הם מייצגים את מידת ההתאמה של הקלט לכל מחלקה, אבל הם עדיין לא מנורמלים לסכום של 100 אחוז.

במקרה שלנו יש חמש מחלקות:

```python
Very Negative
Negative
Neutral
Positive
Very Positive
```

לכן עבור טקסט אחד, המודל יחזיר חמישה ציונים. לדוגמה, באופן סכמטי:

```bash
[-2.1, -0.8, 0.3, 1.7, 3.2]
```

המספר הגבוה ביותר כאן הוא 3.2, והוא נמצא במקום שמתאים למחלקה Very Positive.

אבל לפני שבוחרים מחלקה, בדרך כלל נרצה להפוך את הציונים האלה להסתברויות. את זה עושים עם softmax, שנראה בהמשך.

**מה קורה עד עכשיו בזרימה**

עד השלב הזה עברנו מסלול ברור:

טקסט רגיל נכנס לפונקציה.

ה tokenizer הופך אותו ל tokens ולמספרים.

המספרים נארזים כ PyTorch tensors.

המודל מקבל את הקלט.

המודל מחזיר logits.

כלומר, עדיין לא קיבלנו תשובה מילולית כמו Positive. קיבלנו ציונים מספריים לכל מחלקה.



## איך softmax הופך logits להסתברויות

ראינו שהמודל לא מחזיר מיד תווית כמו Positive או Negative. הוא מחזיר logits, כלומר ציונים גולמיים לכל מחלקה.

לדוגמה, עבור משפט מסוים, המודל יכול להחזיר משהו שנראה רעיונית כך:

```bash
Very Negative: -2.1
Negative:      -0.8
Neutral:        0.3
Positive:       1.7
Very Positive:  3.2
```

המספרים האלה מראים שהמודל נוטה יותר ל Very Positive, כי זה הציון הגבוה ביותר. אבל אלה עדיין לא הסתברויות. אי אפשר לומר ש 3.2 פירושו 3.2 אחוז, או ש 1.7 פירושו 1.7 אחוז. אלה רק ציונים פנימיים.

כדי להפוך אותם לערכים שקל יותר לפרש, משתמשים ב softmax:

```bash
probabilities = torch.nn.functional.softmax(logits, dim=-1)
```

softmax לוקח את כל הציונים וממיר אותם להתפלגות הסתברויות. כלומר, כל מחלקה מקבלת ערך בין 0 ל 1, וכל הערכים יחד מסתכמים ל 1.

לדוגמה, אחרי softmax, הפלט יכול להיראות רעיונית כך:

```bash
Very Negative: 0.02
Negative:      0.05
Neutral:       0.10
Positive:      0.23
Very Positive: 0.60
```

עכשיו קל יותר להבין את התוצאה: המודל מעריך שהמחלקה הסבירה ביותר היא Very Positive, עם ביטחון של בערך 60 אחוז.

חשוב לדייק: זו לא “אמת מוחלטת”. זו הערכת ביטחון של המודל לפי החישוב שלו. אם הדאטה לא מתאים, אם המשפט עמום, או אם המודל חלש בדומיין מסוים, גם confidence גבוה יכול להיות מטעה. ועדיין, עדיף לקבל confidence מאשר להחזיר רק label בלי שום מידע נוסף.

בפרמטר dim=-1 אנחנו אומרים ל PyTorch לבצע את הפעולה על הממד האחרון של ה Tensor. במקרה של מודל סיווג, הממד האחרון הוא רשימת המחלקות. לכן softmax(..., dim=-1) הופך את ציוני המחלקות להסתברויות, ו argmax(..., dim=-1) בוחר את המחלקה בעלת ההסתברות הגבוהה ביותר.



## איך argmax בוחר את המחלקה

אחרי שיש לנו הסתברויות, צריך לבחור איזו מחלקה היא התוצאה הסופית.

בקוד המקורי מופיעה פעולה בסגנון הזה:

```python
predicted_class = torch.argmax(probabilities, dim=-1).item()
```

argmax מחפש את המיקום שבו נמצא הערך הגבוה ביותר.

אם ההסתברויות הן: [0.02, 0.05, 0.10, 0.23, 0.60]

אז הערך הגבוה ביותר הוא 0.60, והוא נמצא באינדקס 4.

לכן:

```python
predicted_class = 4
```

אבל המספר 4 עדיין לא אומר לקורא הרבה. הוא רק אינדקס. כדי להפוך אותו לתוצאה קריאה, צריך מיפוי.

## מיפוי מספרים לתוויות מילוליות

בקובץ sentiment.py יש מילון שממפה מספרים לתוויות:

```python
sentiment_map = {
    0: "Very Negative",
    1: "Negative",
    2: "Neutral",
    3: "Positive",
    4: "Very Positive"
}
```

אם argmax החזיר 4, הקוד יחזיר:

```bash
"Very Positive"
```

אם הוא החזיר 1, הקוד יחזיר:

```bash
"Negative"
```

זה השלב שבו הפלט המספרי של המודל הופך לתוצאה אנושית.

כל הזרימה נראית כך:

```python
Text
-> Tokenizer
-> input_ids + attention_mask
-> Model
-> logits
-> softmax
-> probabilities
-> argmax
-> class index
-> sentiment_map
-> readable label
```

זו בדיוק הצורה שבה הרבה מודלי סיווג עובדים: הם לא “כותבים תשובה”, אלא מחשבים ציונים למחלקות, ואז הקוד בוחר את המחלקה הסבירה ביותר.

**מה חסר בקוד המקורי**

הקוד המקורי מחזיר רק את התווית:

```python
return sentiment_map[predicted_class]
```

כלומר, נקבל משהו כמו:

```bash
Positive
```

אבל לא נדע עד כמה המודל בטוח בזה.

זו מגבלה חשובה. יש הבדל גדול בין מצב שבו המודל מחזיר Positive עם confidence של 96 אחוז, לבין מצב שבו הוא מחזיר Positive עם confidence של 38 אחוז.

בשני המקרים ה label זהה, אבל המשמעות שונה מאוד.

במערכת אמיתית, confidence יכול לעזור לנו להחליט מה לעשות:

אם הביטחון גבוה, אפשר להשתמש בתוצאה אוטומטית.

אם הביטחון בינוני, אפשר לסמן לבדיקה.

אם הביטחון נמוך, אפשר להחזיר Uncertain, לבקש בדיקה אנושית, או להריץ מודל אחר.

לכן השיפור הראשון שכדאי לבצע בקוד הוא להחזיר גם את רמת הביטחון.

## גרסה משופרת של predict_sentiment

במקום להחזיר רק label, נוכל להחזיר גם confidence:

```python
def predict_sentiment(text):
    inputs = tokenizer(
        text.lower(),
        return_tensors="pt",
        truncation=True,
        padding=True,
        max_length=512
    )

    with torch.no_grad():
        outputs = model(**inputs)

    logits = outputs.logits
    probabilities = torch.nn.functional.softmax(logits, dim=-1)

    predicted_class = torch.argmax(probabilities, dim=-1).item()
    confidence = probabilities[0][predicted_class].item()

    sentiment_map = {
        0: "Very Negative",
        1: "Negative",
        2: "Neutral",
        3: "Positive",
        4: "Very Positive"
    }

    return {
        "label": sentiment_map[predicted_class],
        "confidence": confidence
    }
```

עכשיו הפונקציה מחזירה מילון:

```python
{
    "label": "Very Positive",
    "confidence": 0.92
}
```

זה הרבה יותר שימושי. במקום לקבל רק תשובה מילולית, אנחנו מקבלים גם מדד שמאפשר להבין עד כמה המודל בטוח בבחירה שלו.

**הדפסה משופרת של התוצאה**

כדי להדפיס את התוצאה בצורה נוחה:

```python
for text in texts:
    result = predict_sentiment(text)

    print(f"Text: {text}")
    print(f"Sentiment: {result['label']}")
    print(f"Confidence: {result['confidence']:.2%}")
    print()
```

אם confidence הוא 0.9234, ההדפסה תהיה:

```bash
Confidence: 92.34%
```

זו צורה קריאה יותר לבני אדם.

**למה confidence חשוב במיוחד**

במודל סיווג, קל להתפתות לחשוב שהתווית הסופית היא כל הסיפור. אבל במערכות אמיתיות, התווית לבד לא תמיד מספיקה.

נניח שמערכת מסווגת פניות לקוחות לפי טון:

```bash
Negative
```

אם הביטחון הוא 95 אחוז, ייתכן שנרצה להעלות את הפנייה בעדיפות גבוהה.

אבל אם הביטחון הוא 41 אחוז, אולי הטקסט עמום, אולי הוא מעורב, ואולי לא כדאי לקבל החלטה אוטומטית.

אותו רעיון נכון גם לעולם פיננסי. אם מודל מחזיר Positive על משפט פיננסי עם ביטחון נמוך, לא נכון לבנות עליו פעולה אוטומטית. בשיעורי הבית, למשל, נשתמש ברעיון דומה עם FinBERT: רק כאשר הביטחון גבוה מסף מסוים, נבצע פעולה מדומה כמו BUY או SELL. אחרת נחזיר HOLD.
