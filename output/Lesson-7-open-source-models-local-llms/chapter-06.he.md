# מודל יוצר טקסט עם Qwen

אחרי שראינו מודל סיווג רגשות, אפשר לעבור לסוג אחר של מודל: מודל שמייצר טקסט.

בקובץ sentiment.py המודל קיבל משפט והחזיר קטגוריה מתוך רשימה מוגדרת. זו משימת Classification.

בקובץ qwen.py המטרה שונה: המודל מקבל הודעה בפורמט של chat, ומייצר תשובה חדשה. זו משימת Text Generation.

לא מדובר בבחירה בין Positive, Negative או Neutral. המודל צריך להמשיך טקסט, לבנות תשובה, לנסח משפטים, ולעיתים גם לבצע תהליך פנימי של חשיבה לפני התשובה הסופית.

הקובץ qwen.py מדגים עבודה עם המודל:

```bash
Qwen/Qwen3-0.6B
```

זהו מודל קטן יחסית ממשפחת Qwen, שמתאים ללמידה ולהרצה מקומית בסיסית יותר ממודלים גדולים. המטרה אינה להראות את המודל החזק ביותר, אלא להבין את תהליך העבודה עם מודל יוצר טקסט דרך Transformers.

## איך מוצאים את קוד ההרצה באתר Hugging Face

גם כאן לא מתחילים בכתיבת קוד מאפס. מתחילים מעמוד המודל ב Hugging Face.

במקרה שלנו אפשר לחפש ישירות את שם המודל:

Qwen/Qwen3-0.6B

או לחפש לפי משפחת המודלים:

Qwen3

לאחר שנכנסים לעמוד המודל, קודם קוראים בקצרה את ה Model Card. המטרה היא לוודא שזה באמת מודל שמתאים ליצירת טקסט, ולא מודל סיווג, embeddings או משימה אחרת.

אחר כך מחפשים בעמוד אזור כמו:

How to use

או דוגמת קוד שמראה איך לטעון את המודל עם transformers.

במודל יוצר טקסט, נצפה לראות בדרך כלל שימוש ברכיבים כאלה:

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
```

כלומר, לא AutoModelForSequenceClassification, אלא:

```python
AutoModelForCausalLM
```

זו האינדיקציה החשובה: מדובר במודל שמייצר טקסט, לא במודל שמחזיר קטגוריה.

דוגמת הקוד בדרך כלל תכלול טעינת tokenizer, טעינת מודל, בניית הודעה, המרה לפורמט שהמודל מבין, ואז קריאה ל:

```python
model.generate(...)
```

את הקוד מעתיקים לקובץ חדש בפרויקט:

```bash
examples/qwen.py
```

או בצורה פשוטה יותר:

```bash
qwen.py
```

ואז מריצים:

```bash
python .\examples\qwen.py
```

או:

```bash
python qwen.py
```

כמו בפרק הקודם, דוגמת הקוד באתר היא נקודת התחלה. היא מראה איך להריץ את המודל, אבל לא תמיד מסבירה מה כל שורה עושה, איך להתאים את הקוד לחומרה שלנו, איך לשלוט באורך התשובה, ואיך להבין את מבנה הפלט. מכאן מתחיל החלק החשוב: לפרק את הקוד ולהבין אותו.

## מטרת הקובץ qwen.py

מטרת הקובץ qwen.py היא להריץ מודל שפה יוצר טקסט מתוך Python.

הקוד טוען את המודל:

```python
model_name = "Qwen/Qwen3-0.6B"
```

לאחר מכן הוא טוען tokenizer ומודל:

```python
tokenizer = AutoTokenizer.from_pretrained(model_name)

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="cuda"
)
```

בהמשך הקוד בונה הודעת chat, מעביר אותה דרך apply_chat_template, מריץ generation בעזרת model.generate, ואז מפריד בין שני חלקים בפלט:

```bash
thinking content
content
```

כלומר, הקוד לא רק מקבל תשובה רגילה. הוא גם מטפל במצב שבו המודל מחזיר חלק של חשיבה פנימית וחלק של תשובה סופית. זה מופיע בקובץ qwen.py, שבו יש הפרדה בין thinking_content לבין content.

## מה ההבדל בין Classification לבין Text Generation

כדי להבין את qwen.py, צריך קודם להבין את ההבדל בין שני סוגי המודלים שראינו.

במודל סיווג, כמו sentiment.py, הקלט הוא טקסט והפלט הוא מחלקה:

```bash
Input:
I loved this movie!

Output:
Positive
```

המודל לא כותב תשובה חופשית. הוא מחשב ציונים למחלקות מוגדרות מראש, ואז הקוד בוחר את המחלקה שקיבלה את הציון הגבוה ביותר.

במודל יוצר טקסט, כמו Qwen, הקלט הוא prompt או הודעת chat, והפלט הוא טקסט חדש:

```bash
Input:
Write a short explanation of what an LLM is.

Output:
A large language model is...
```

אין כאן רשימת מחלקות קבועה. בכל צעד המודל בוחר את הטוקן הבא, ואז את הטוקן הבא אחריו, וכך נבנית התשובה.

לכן Text Generation הוא תהליך פתוח יותר. הוא גמיש יותר, אבל גם דורש יותר שליטה:

- צריך להגדיר prompt טוב.

- צריך לשלוט באורך התשובה.

- צריך להיזהר מתשובות ארוכות מדי.

- צריך להבין שהפלט יכול להשתנות בין הריצות.

- צריך לוודא שהמודל מחזיר תשובה בפורמט שמתאים לנו.

במודל סיווג אנחנו שואלים:

- איזו מחלקה הכי מתאימה לטקסט?

במודל יוצר טקסט אנחנו שואלים:

- איזה טקסט המודל צריך לייצר עכשיו?

זו קפיצה משמעותית מבחינה הנדסית.



## טעינת Qwen/Qwen3-0.6B

הקוד מתחיל מהגדרת שם המודל:

```python
model_name = "Qwen/Qwen3-0.6B"
```

השם הזה מתאר מודל שנמצא ב Hugging Face תחת הארגון או המשפחה Qwen, בשם Qwen3-0.6B.

החלק 0.6B מציין בקירוב את גודל המודל: כ 0.6 מיליארד פרמטרים. זה קטן יחסית לעולם ה LLMs, ולכן מתאים יותר ללמידה ולהרצה מקומית ממודלים גדולים בהרבה.

אבל גם מודל “קטן” כזה עדיין כבד יותר ממודל סיווג רגשות פשוט. הוא צריך לטעון משקלים, לנהל context, ולייצר טוקנים אחד אחרי השני. לכן כבר בקובץ הזה מתחילים לראות חשיבות לחומרה, במיוחד כאשר משתמשים ב:

```python
device_map="cuda"
```

שנפרק במנה הבאה.

## שימוש ב AutoTokenizer

כמו בפרק הקודם, גם כאן צריך tokenizer:

```python
tokenizer = AutoTokenizer.from_pretrained(model_name)
```

גם מודל יוצר טקסט לא מקבל מחרוזת רגילה ישירות. הוא מקבל tokens מספריים.

אבל כאן ל tokenizer יש תפקיד נוסף: הוא לא רק מפרק טקסט למספרים, אלא גם עוזר לבנות את הפורמט שהמודל מצפה לקבל בשיחת chat.

מודלי chat לא תמיד מקבלים prompt פשוט כמו:

```python
Write a short introduction to LLMs.
```

לעיתים הם מצפים למבנה של הודעות, למשל:

```python
messages = [
    {"role": "user", "content": "Write a short introduction to LLMs."}
]
```

ואז tokenizer יודע להפוך את מבנה ה chat הזה לטקסט פנימי בפורמט שהמודל אומן עליו. את זה נראה בהמשך עם:

```python
apply_chat_template
```



## שימוש ב AutoModelForCausalLM

טעינת המודל נעשית כך:

```python
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="cuda"
)
```

החלק החשוב כאן הוא:

```python
AutoModelForCausalLM
```

CausalLM הוא קיצור של Causal Language Model.

מודל כזה מיועד לחיזוי הטוקן הבא על בסיס הטוקנים הקודמים. כלומר, הוא מקבל רצף טוקנים ומחשב מה הטוקן הבא הסביר ביותר. לאחר מכן הוא מוסיף את הטוקן הזה לרצף, ושוב מחשב את הבא. כך נוצרת תשובה שלמה.

זה שונה מ:

```python
AutoModelForSequenceClassification
```

שם המודל מחזיר ציונים למחלקות.

אפשר לסכם את ההבדל כך:

```python
AutoModelForSequenceClassification
```

מתאים לסיווג טקסט למחלקות מוגדרות.

AutoModelForCausalLM

מתאים ליצירת טקסט חדש, טוקן אחרי טוקן.

הבחירה במחלקה הנכונה ב transformers אינה פרט טכני קטן. היא מגדירה איזה סוג פעולה אנחנו מבקשים מהמודל לבצע.

בהמשך נפרק את שני הפרמטרים החשובים בטעינת המודל:

```python
torch_dtype="auto"
device_map="cuda"
```

ונראה איך להתאים את הקוד למחשב שאין בו GPU.



## המשך טעינת המודל

בקובץ qwen.py טעינת המודל מתבצעת כך:

```python
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="cuda"
)
```

לאחר שראינו ש AutoModelForCausalLM מתאים למודל שמייצר טקסט. עכשיו צריך להבין את שני הפרמטרים החשובים שנוספו כאן:

```python
torch_dtype="auto"
device_map="cuda"
```

אלה לא פרטים שוליים. הם קובעים איך המודל ייטען לזיכרון, באיזה סוג מספרים הוא ישתמש, ועל איזו חומרה הוא ירוץ.

## מה המשמעות של torch_dtype="auto"

הפרמטר:

```python
torch_dtype="auto"
```

אומר ל transformers לבחור באופן אוטומטי את סוג הנתונים המתאים למשקלים של המודל.

כדי להבין למה זה חשוב, צריך לזכור שמודל הוא אוסף גדול מאוד של מספרים. המספרים האלה הם המשקלים של המודל. כאשר טוענים מודל לזיכרון, צריך להחליט באיזה פורמט מספרי לשמור אותם.

לדוגמה, משקלים יכולים להיטען בפורמטים כמו:

```python
float32
float16
bfloat16
```

float32 מדויק יותר, אבל צורך יותר זיכרון.

float16 צורך פחות זיכרון ויכול להיות מהיר יותר על GPU מתאים.

bfloat16 נפוץ בהרצה ואימון על חומרות מסוימות, ונותן איזון טוב בין יעילות ליציבות.

כאשר כותבים:

```python
torch_dtype="auto"
```

אנחנו לא מכריחים את המודל להיטען בפורמט מסוים. אנחנו נותנים לספרייה לבחור את הפורמט שמתאים למודל ולמטא דאטה שלו.

במודלים גדולים, הבחירה הזאת יכולה להשפיע מאוד על צריכת הזיכרון. טעינה ב float32 יכולה להכפיל כמעט את כמות הזיכרון לעומת טעינה ב float16. לכן בפרויקטים של LLMs, סוג הנתונים הוא שיקול מעשי מאוד.

במילים פשוטות: 
torch_dtype="auto" אומר לספרייה לטעון את המודל בפורמט מספרי שמתאים לו, במקום שאנחנו נקבע ידנית.

**למה זה חשוב יותר במודל יוצר טקסט**

במודל סיווג כמו sentiment.py, המודל קטן יחסית והתהליך קצר: שולחים טקסט, מקבלים logits, בוחרים מחלקה.

במודל יוצר טקסט, המצב כבד יותר. המודל לא מחשב פלט אחד בלבד. הוא מייצר טוקן אחרי טוקן. בכל שלב הוא צריך לחשב מה הטוקן הבא, להוסיף אותו לרצף, ואז להמשיך. ככל שהתשובה ארוכה יותר, יש יותר חישובים.

לכן כל החלטה שקשורה לזיכרון ולחומרה משמעותית יותר.

מודל כמו:

```python
Qwen/Qwen3-0.6B
```

הוא קטן יחסית לעולם ה LLMs, אבל עדיין גדול בהרבה ממודל סיווג פשוט. לכן כבר כאן רואים פרמטרים שמתייחסים להרצה יעילה יותר.

## מה המשמעות של device_map="cuda"

הפרמטר:

```python
device_map="cuda"
```

אומר למודל להיטען ולהריץ את החישובים על GPU שתומך ב CUDA.

CUDA היא סביבת החישוב של NVIDIA. כאשר יש במחשב כרטיס NVIDIA מתאים וגרסת PyTorch שתומכת ב CUDA, אפשר להריץ חישובים כבדים על כרטיס המסך במקום על המעבד הראשי.

למה זה חשוב?

כי מודלים גדולים מבצעים הרבה מאוד חישובים מטריציוניים. GPU מתאים לבצע חישובים כאלה בצורה מקבילית ויעילה הרבה יותר מ CPU רגיל. לכן מעבר מ CPU ל GPU יכול לשפר משמעותית את מהירות ההרצה.

בקובץ qwen.py, הבחירה ב:

```python
device_map="cuda"
```

משמעותה שהקוד מניח שיש GPU זמין. אם במחשב אין GPU מתאים, הקוד עלול להיכשל בזמן טעינת המודל.

**איך לבדוק אם יש CUDA זמין**

אפשר לבדוק מתוך Python:

```python
import torch

print(torch.cuda.is_available())

if torch.cuda.is_available():
    print(torch.cuda.get_device_name(0))
```

אם התוצאה היא:

```bash
True
```

אז PyTorch מזהה GPU עם CUDA.

אם התוצאה היא:

```bash
False
```

אז הקוד לא יכול להשתמש ב CUDA בסביבה הנוכחית.

אפשר גם לבדוק מבחוץ באמצעות PowerShell:

```bash
nvidia-smi
```

אם הפקודה מציגה את כרטיס המסך, גרסת הדרייבר וצריכת זיכרון, זה סימן טוב שכרטיס NVIDIA מזוהה במערכת.

**איך להתאים את הקוד למחשב בלי GPU**

אם אין GPU, לא כדאי להשאיר:

```python
device_map="cuda"
```

כי הקוד מבקש במפורש להשתמש ב CUDA.

אפשרות פשוטה יותר היא להסיר את הפרמטר:

```python
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto"
)
```

במצב כזה המודל ייטען בדרך כלל ל CPU, אם אין התקן אחר זמין.

אפשרות אחרת היא לציין במפורש CPU:

```python
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto",
    device_map="cpu"
)
```

אבל צריך לקחת בחשבון שהרצה על CPU תהיה איטית יותר, במיוחד כאשר מייצרים טקסט ארוך.

במחשב בלי GPU כדאי גם להקטין את אורך התשובה שנבקש מהמודל. כלומר, להשתמש בערך נמוך יותר ב:

```python
max_new_tokens
```

**גרסה גמישה יותר של טעינת המודל**

אפשר לכתוב את הקוד כך שיבחר אוטומטית בין GPU ל CPU:

```python
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM

model_name = "Qwen/Qwen3-0.6B"

tokenizer = AutoTokenizer.from_pretrained(model_name)

device = "cuda" if torch.cuda.is_available() else "cpu"

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto"
)

model = model.to(device)

print(f"Model loaded on: {device}")
```

כאן אנחנו בודקים אם CUDA זמינה. אם כן, משתמשים ב GPU. אם לא, משתמשים ב CPU.

עם זאת, חשוב לשים לב: לא כל תצורת טעינה מתאימה לכל מודל ולכל גודל. במודלים גדולים יותר, מעבר פשוט עם:

```python
model.to(device)
```

לא תמיד יהיה הפתרון הכי טוב, כי ייתכן שהמודל גדול מדי לזיכרון של ההתקן. אבל עבור מודל קטן יחסית כמו Qwen3-0.6B, זו גישה טובה ללמידה.

**גרסה שמעדיפה CUDA כאשר קיימת**

אם רוצים לשמור על רעיון דומה לקוד המקורי, אבל בלי להישבר במחשב שאין בו GPU, אפשר לכתוב:

```python
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM

model_name = "Qwen/Qwen3-0.6B"

tokenizer = AutoTokenizer.from_pretrained(model_name)

if torch.cuda.is_available():
    model = AutoModelForCausalLM.from_pretrained(
        model_name,
        torch_dtype="auto",
        device_map="cuda"
    )
else:
    model = AutoModelForCausalLM.from_pretrained(
        model_name,
        torch_dtype="auto"
    )

print("CUDA available:", torch.cuda.is_available())
```

היתרון כאן הוא שהקוד שומר על שימוש ב CUDA כאשר היא זמינה, אבל לא מניח שהיא תמיד קיימת.

**למה מודל יוצר טקסט רגיש יותר לחומרה**

מודל יוצר טקסט עובד בצורה איטרטיבית. הוא לא מחזיר את כל התשובה בפעולה אחת פשוטה. הוא מייצר טוקן, ואז עוד טוקן, ואז עוד טוקן, עד שהוא מגיע לסוף התשובה או למגבלת האורך שהגדרנו.

כל טוקן חדש דורש חישוב נוסף.

לכן יש קשר ישיר בין:

- אורך התשובה

- גודל המודל

- כמות הזיכרון

- סוג החומרה

- מהירות ההרצה

אם נגדיר למודל לייצר הרבה מאוד טוקנים, ההרצה יכולה להיות איטית, במיוחד על CPU. אם המודל גדול מדי, הוא יכול לא להיטען בכלל. אם אין מספיק VRAM בכרטיס המסך, נקבל שגיאת זיכרון.

לכן בפרויקטים אמיתיים לא מספיק לשאול אם המודל איכותי. צריך לשאול גם אם הוא רץ בצורה סבירה על החומרה שיש לנו.

בקובץ qwen.py, שני הפרמטרים האלה חשובים במיוחד:

```python
torch_dtype="auto"
device_map="cuda"
```

torch_dtype="auto" עוזר לטעון את המודל בפורמט מספרי מתאים.

device_map="cuda" אומר להריץ את המודל על GPU עם CUDA.

אם יש GPU מתאים, זו דרך טובה להריץ מהר יותר. אם אין GPU, צריך להתאים את הקוד כדי שלא ייכשל.

## בניית הודעת chat

אחרי שטענו את ה tokenizer ואת המודל, צריך להכין את הקלט שנשלח למודל.

במודל סיווג כמו sentiment.py, שלחנו טקסט פשוט לפונקציה:

```python
predict_sentiment("I loved this movie!")
```

אבל במודל chat כמו Qwen, הקלט בדרך כלל נבנה כמערך של הודעות. בקובץ qwen.py מופיע מבנה כזה:

```python
messages = [
    {"role": "user", "content": "Give me a short introduction to large language model."}
]
```

המבנה הזה אומר למודל:

יש כאן הודעה אחת.

מי ששלח אותה הוא המשתמש.

תוכן ההודעה הוא בקשה קצרה להסבר על Large Language Model.

הפורמט הזה חשוב, כי מודלי chat לא מקבלים רק טקסט גולמי. הם אומנו לעבוד עם מבנה שיחה: משתמש, עוזר, לפעמים system message, ולעיתים כמה הודעות ברצף. לכן במקום לשלוח רק מחרוזת, אנחנו בונים רשימת הודעות עם תפקידים.

**מה המשמעות של role**

בכל הודעה יש שדה:

```python
"role": "user"
```

השדה role אומר למודל מי כתב את ההודעה.

בדרך כלל נראה תפקידים כמו:

```python
system
user
assistant
```

system משמש להגדרת התנהגות כללית של המודל, למשל: “ענה בקצרה” או “פעל כמומחה Python”.

user הוא ההודעה של המשתמש.

assistant הוא תשובה קודמת של המודל, אם ממשיכים שיחה קיימת.

בקובץ שלנו יש רק הודעת user, כי זו דוגמה פשוטה. אבל במערכות אמיתיות, רשימת messages יכולה לכלול היסטוריית שיחה שלמה.

לדוגמה:

```python
messages = [
    {"role": "system", "content": "You are a concise AI engineering tutor."},
    {"role": "user", "content": "Explain what an LLM is."}
]
```

כאן ה system מגדיר את סגנון התשובה, וה user שואל את השאלה עצמה.

**למה לא שולחים פשוט string**

אפשר לשאול למה לא לשלוח ישירות:

```python
"Give me a short introduction to large language model."
```

הסיבה היא שמודל chat מצפה לפורמט מסוים. בזמן האימון, הוא ראה שיחות במבנה מסוים, עם סימונים פנימיים שמבדילים בין משתמש לבין עוזר. אם נשלח לו רק טקסט רגיל, ייתכן שהוא עדיין יענה, אבל לא בהכרח בצורה שתואמת את הדרך שבה הוא אומן.

לכן משתמשים במנגנון שממיר את רשימת ההודעות לפורמט הפנימי שהמודל מכיר.

זה התפקיד של:

```python
apply_chat_template
```

## שימוש ב apply_chat_template

בקובץ qwen.py מופיעה קריאה בסגנון הזה:

```python
text = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,
    enable_thinking=True
)
```

זו אחת השורות החשובות בקובץ.

היא לוקחת את רשימת ההודעות:

```python
messages = [
    {"role": "user", "content": "Give me a short introduction to large language model."}
]
```

והופכת אותה לטקסט בפורמט שהמודל מצפה לקבל.

כלומר, אנחנו לא צריכים לדעת בעצמנו מהם הסימונים הפנימיים של Qwen, איך מסמנים תחילת הודעת משתמש, איך מסמנים תחילת תשובת עוזר, ואיפה המודל אמור להתחיל לייצר טקסט. ה tokenizer יודע לבנות את התבנית הנכונה עבור המודל הזה.

זו הסיבה שמשתמשים ב tokenizer שמגיע עם אותו מודל. הוא לא רק מפרק טקסט ל tokens. במקרה של מודל chat, הוא גם יודע איך לעצב את השיחה בפורמט המתאים למודל.

**מה עושה tokenize=False**

בקריאה ל apply_chat_template מופיע:

```python
tokenize=False
```

המשמעות היא: בשלב הזה אל תחזיר עדיין tokens מספריים. תחזיר טקסט מוכן בפורמט chat.

כלומר, אחרי הקריאה הזאת המשתנה text עדיין מכיל טקסט, לא tensors.

למה עושים את זה בשני שלבים?

כי הקוד רוצה קודם לבנות את טקסט ה chat, ורק לאחר מכן להעביר אותו ל tokenizer רגיל:

```python
model_inputs = tokenizer([text], return_tensors="pt").to(model.device)
```

זו הפרדה נוחה ללמידה:

קודם בונים את prompt בפורמט הנכון.

אחר כך הופכים אותו ל tensors שהמודל יודע לקבל.

**מה עושה add_generation_prompt=True**

הפרמטר:

```python
add_generation_prompt=True
```

אומר ל tokenizer להוסיף בסוף הטקסט סימון שמודיע למודל: עכשיו הגיע התור שלך לענות.

במילים פשוטות, רשימת ההודעות כוללת את שאלת המשתמש. אבל המודל צריך להבין איפה מתחילה תשובת העוזר. add_generation_prompt=True מוסיף את החלק הפנימי שמסמן למודל להתחיל לייצר תשובה.

בלי זה, המודל עלול לא להבין בצורה מדויקת שהקלט הסתיים ושעכשיו עליו להשלים את תשובת ה assistant.

זה דומה להכנת במה נכונה למודל: הנה השיחה עד עכשיו, ועכשיו תייצר את ההודעה הבאה.

**מה עושה enable_thinking=True**

הפרמטר:

```python
enable_thinking=True
```

קשור ליכולת של Qwen להחזיר פלט שמופרד לשני חלקים:

```python
thinking content
content
```

כלומר, המודל יכול לייצר חלק שמייצג תהליך חשיבה פנימי, ולאחריו תשובה סופית.

בקובץ qwen.py רואים שהקוד מתייחס לכך במפורש: הוא מנסה להפריד בין thinking_content לבין content, ומדפיס את שניהם בנפרד.

חשוב להבין את המשמעות המעשית: לא בכל אפליקציה נרצה להציג למשתמש את חלק החשיבה. במקרים רבים נרצה להציג רק את התשובה הסופית. לכן הקוד מדגים איך לזהות את ההפרדה ולשלוט במה שמודפס.

בפרויקט אמיתי, צריך להחליט במפורש מה עושים עם חלק כזה:

האם מציגים רק תשובה סופית?

האם שומרים את החשיבה לצורכי Debug?

האם מתעלמים ממנה לחלוטין?

האם בכלל מכבים את האפשרות אם היא לא נחוצה?

בדרך כלל, למשתמש הקצה נציג רק את התוכן הסופי.

**הפיכת טקסט ה chat לקלט מספרי**

אחרי ש apply_chat_template יצר את הטקסט בפורמט הנכון, הקוד הופך אותו לקלט מספרי:

```python
model_inputs = tokenizer([text], return_tensors="pt").to(model.device)
```

כאן חוזרים לרעיון שכבר הכרנו בפרק הקודם: המודל לא מקבל טקסט רגיל. הוא מקבל tensors.

ההבדל הוא שכאן לא מעבירים ל tokenizer את השאלה המקורית בלבד, אלא את טקסט ה chat המלא שנוצר מה template.

שימו לב גם לסוגריים המרובעים:

```python
[text]
```

הם יוצרים רשימה עם פריט אחד. כלומר, מבחינת tokenizer, אנחנו שולחים batch בגודל 1. זה שימושי כי רוב ממשקי המודלים עובדים באופן טבעי עם batch, גם אם יש בו רק דוגמה אחת.

החלק:

```python
.to(model.device)
```

מעביר את הקלט לאותו התקן שבו נמצא המודל. אם המודל נטען ל GPU, גם הקלט צריך להיות על GPU. אם המודל על CPU, גם הקלט צריך להיות על CPU.

אם המודל והקלט נמצאים על התקנים שונים, נקבל שגיאה.

**למה זה חשוב**

במודלים יוצרי טקסט, איכות התשובה תלויה לא רק במודל עצמו, אלא גם בדרך שבה מכינים את ההודעה.

אם שולחים prompt לא בפורמט שהמודל מצפה לו, התשובה יכולה להיות פחות יציבה. אם לא מוסיפים generation prompt, המודל לא תמיד יבין נכון איפה להתחיל לענות. אם לא מעבירים את הקלט לאותו device, הקוד ייכשל טכנית.

לכן החלק הזה בקובץ qwen.py הוא קריטי: הוא מחבר בין מבנה שיחה אנושי לבין קלט מספרי תקין למודל.

## הפעלת model.generate

אחרי שהקוד בנה את הודעת ה chat, המיר אותה לפורמט שהמודל מבין, והעביר את הקלט לאותו device שבו נמצא המודל, מגיע שלב יצירת התשובה.

בקובץ qwen.py הקריאה נראית כך:

```python
generated_ids = model.generate(
    **model_inputs,
    max_new_tokens=32768
)
```

זו השורה שבה המודל מתחיל לייצר טקסט.

כדאי לשים לב להבדל לעומת מודל הסיווג מהפרק הקודם. במודל סיווג, הקריאה למודל החזירה logits, כלומר ציונים למחלקות. כאן אנחנו לא רוצים ציונים למחלקות. אנחנו רוצים שהמודל ייצור רצף חדש של tokens.

הפעולה generate עושה בדיוק את זה: היא לוקחת את הקלט, ואז מייצרת טוקן חדש, ועוד טוקן, ועוד טוקן, עד שהיא מגיעה לעצירה טבעית או עד שהיא מגיעה למגבלה שהגדרנו.

כלומר, מודל יוצר טקסט לא “כותב את כל התשובה במכה אחת”. הוא בונה אותה בהדרגה, טוקן אחרי טוקן.

## מהו max_new_tokens ולמה צריך להזהר ממנו

הפרמטר:

```python
max_new_tokens=32768
```

קובע כמה tokens חדשים המודל רשאי לייצר לכל היותר.

חשוב להבין את המילה **new**. הכוונה אינה לאורך הכולל של הקלט והתשובה, אלא לכמות הטוקנים החדשים שהמודל יכול להוסיף אחרי ה prompt.

אם שלחנו למודל שאלה קצרה, וכתבנו:

```python
max_new_tokens=200
```

המודל יוכל לייצר עד 200 tokens חדשים בתשובה.

אם כתבנו:

```python
max_new_tokens=32768
```

אנחנו מאפשרים לו לייצר תשובה ארוכה מאוד.

**למה צריך להיזהר מ max_new_tokens גדול מדי**

ערך גדול מדי עלול לגרום לכמה בעיות.

ראשית, ההרצה יכולה להיות איטית מאוד. כל טוקן חדש דורש חישוב נוסף. אם מאפשרים למודל לייצר עשרות אלפי tokens, ההרצה יכולה להימשך זמן רב, במיוחד על CPU או על GPU חלש.

שנית, צריכת הזיכרון יכולה לגדול. ככל שהרצף ארוך יותר, המודל צריך לנהל יותר הקשר פנימי בזמן generation.

שלישית, במקרים מסוימים המודל עלול להמשיך לייצר טקסט הרבה מעבר למה שהתכוונו. למשל, ביקשנו “הסבר קצר”, אבל נתנו לו מגבלת אורך עצומה. אם אין עצירה טבעית טובה, הוא עלול לייצר יותר מדי.

לכן בפרויקט לימודי או במחשב מקומי כדאי להתחיל מערך קטן יותר, למשל:

```python
max_new_tokens=256
```

או:

```python
max_new_tokens=512
```

ורק אם באמת צריך תשובה ארוכה, להגדיל בהדרגה.

במילים פשוטות: max_new_tokens הוא בלם בטיחות. הוא מונע מהמודל לייצר בלי גבול.

**חיתוך הפלט החדש מתוך generated_ids**

אחרי הקריאה ל generate, מתקבל רצף tokens שכולל גם את הקלט המקורי וגם את הטקסט החדש שהמודל יצר.

לכן בקובץ יש שלב שמפריד בין מה שכבר היה בקלט לבין מה שהמודל הוסיף:

```python
output_ids = generated_ids[0][len(model_inputs.input_ids[0]):].tolist()
```

השורה הזאת אומרת:

קח את הפלט הראשון ב batch.

התעלם מה tokens שהיו חלק מהקלט המקורי.

השאר רק את ה tokens החדשים שהמודל יצר.

נפרק את זה:

```python
generated_ids[0]
```

מתייחס לתוצאה הראשונה, כי שלחנו batch בגודל 1.

```python
len(model_inputs.input_ids[0])
```

מחזיר את אורך הקלט המקורי.

ואז:

```python
generated_ids[0][len(model_inputs.input_ids[0]):]
```

חותך את הרצף ומחזיר רק את החלק שנוצר אחרי הקלט.

למה זה חשוב?

כי אם נפענח את כל generated_ids, נקבל גם את prompt המקורי וגם את התשובה. בדרך כלל אנחנו רוצים להציג למשתמש רק את מה שהמודל יצר, לא את כל הקלט שנשלח אליו.

**פענוח tokens חזרה לטקסט**

אחרי שחילצנו את הטוקנים החדשים, צריך להפוך אותם חזרה לטקסט קריא:

```python
response = tokenizer.decode(output_ids, skip_special_tokens=True)
```

כאן ה tokenizer עושה את הפעולה ההפוכה ממה שראינו קודם.

בהתחלה הוא הפך טקסט ל tokens מספריים. 
עכשיו הוא הופך tokens מספריים חזרה לטקסט.

הפרמטר:

```python
skip_special_tokens=True
```

אומר לו לא להציג tokens מיוחדים של המודל, כמו סימוני התחלה, סוף, או סימונים פנימיים של chat.

התוצאה היא מחרוזת רגילה שאפשר להדפיס למסך.

## הפרדה בין thinking_content לבין content

בקובץ qwen.py יש טיפול מיוחד בפלט של המודל: הפרדה בין תוכן חשיבה לבין התשובה הסופית.

הקוד מחפש token או סימון שמפריד בין שני חלקים, ואז מגדיר:

```python
thinking_content
content
```

הרעיון הוא שהמודל יכול להחזיר שני אזורים שונים:

thinking_content - חלק שמייצג תהליך חשיבה פנימי או reasoning.

Content - התשובה הסופית שאותה נרצה להציג למשתמש.

## איך להתאים את הקוד למחשב בלי GPU

אם במחשב אין GPU מתאים, יש שתי התאמות חשובות.

הראשונה היא טעינת המודל בלי להכריח CUDA:

```python
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype="auto"
)
```

או בחירה דינמית:

```python
device = "cuda" if torch.cuda.is_available() else "cpu"

model = AutoModelForCausalLM.from_pretrained(
```

**מה הקובץ qwen.py מלמד**

הקובץ qwen.py מדגים מעבר חשוב מאוד.

בפרק הקודם עבדנו עם מודל סיווג. שם הפלט היה מחלקה מתוך רשימה. כאן אנחנו עובדים עם מודל שפה יוצר טקסט. הפלט כבר פתוח, ארוך יותר, גמיש יותר, וגם מורכב יותר לשליטה.

הזרימה הכללית היא:

```bash
Model name
-> AutoTokenizer
-> AutoModelForCausalLM
-> messages
-> apply_chat_template
-> tokenizer
-> model.generate
-> generated token ids
-> decode
-> thinking content + final content
```

זו אחת הזרימות המרכזיות בעבודה עם LLMs מקומיים דרך Transformers.
