# עבודה מול APIs של מודלי LLM

## המעבר מצ'אט למערכות אמיתיות

עד עכשיו רוב האנשים מכירים מודלי AI דרך ממשקי צ'אט. פותחים אתר, כותבים שאלה, ומקבלים תשובה. אבל המרצה הסביר שבשלב הזה אנחנו מתחילים לעבור לעולם האמיתי של AI Engineering, שבו מודל השפה מפסיק להיות "צ'אט" והופך לחלק ממערכת תוכנה.

זו נקודת מעבר מאוד משמעותית.

כאשר עובדים דרך אתר כמו ChatGPT, רוב המורכבות מוסתרת מאחורינו:

- אין צורך לנהל שרתים

- לא חושבים על APIs

- לא מתעסקים עם Authentication

- לא מנהלים עלויות

- ולא בונים ארכיטקטורה

אבל ברגע שרוצים לשלב AI בתוך מערכת אמיתית, הכול משתנה.

המרצה הסביר שהמטרה שלנו היא ללמוד כיצד תוכנות מתקשרות עם מודלי שפה בצורה אוטומטית דרך קוד. במקום שאדם יכתוב Prompt ידנית, האפליקציה עצמה תשלח בקשות למודל ותקבל תשובות באופן מלא אוטומטי.

זו בדיוק הנקודה שבה AI מתחיל להפוך לחלק אינטגרלי ממערכות תוכנה.

**לדוגמה:**

- מערכת שירות לקוחות יכולה לשלוח שיחות למודל לצורך סיכום

- מערכת CRM יכולה לייצר לידים אוטומטית

- מערכת פנימית יכולה לנתח מסמכים

- אפליקציה יכולה לבצע תרגום בזמן אמת

- Agent יכול לבצע שרשרת פעולות מורכבת

- ומערכת קוד יכולה להיעזר במודל לצורך השלמת קוד או ניתוח לוגים

כלומר, מודל השפה הופך להיות רכיב תוכנה לכל דבר.

המרצה גם הדגיש שכאן מתחילים להופיע אתגרים חדשים שלא קיימים בשימוש רגיל בצ'אט:

- איך שולחים Requests?

- איך עובדים עם API Keys?

- איך מנהלים עלויות?

- איך שומרים Context?

- איך מטפלים ב-Errors?

- איך עובדים עם Latency?

- ואיך מונעים מהמערכת לקרוס תחת עומס?

זו בדיוק הסיבה שעולם ה-LLMs כבר אינו רק "Prompt", אלא תחום הנדסי שלם שמשלב:

- תוכנה

- תשתיות

- APIs

- אבטחת מידע

- ארכיטקטורה

- וניהול מערכות מבוזרות

**המרצה ניסה להעביר כאן רעיון מרכזי:**

ברגע שמודל AI נכנס לתוך קוד אמיתי, הוא מפסיק להיות הדגמה טכנולוגית והופך לשירות תוכנה שחייב לעבוד בצורה יציבה, מאובטחת ויעילה.

## מהו API של מודל שפה

לאחר שהבנו שמודל AI הופך לחלק ממערכת תוכנה אמיתית, המרצה עבר להסביר את אחד המושגים המרכזיים ביותר בעולם הזה: API.

המונח API הוא קיצור של:

Application Programming Interface

בפשטות, API הוא הדרך שבה שתי מערכות תוכנה מתקשרות אחת עם השנייה.

במקרה שלנו, האפליקציה שלנו אינה "מדברת" ישירות עם המודל עצמו, אלא שולחת בקשה לשירות חיצוני שמפעיל את המודל ומחזיר תשובה.

המרצה הסביר שכאשר עובדים מול מודלי LLM, כמעט הכול מתבצע דרך APIs.

במקום לפתוח אתר ולהקליד Prompt ידנית, התוכנה עצמה שולחת בקשת HTTP לשרת של ספק ה-AI.

לדוגמה:

Application API LLM Response

**הבקשה כוללת בדרך כלל:**

- איזה מודל להפעיל

- מה ההודעה שנשלחת

- כמה Tokens להחזיר

- הגדרות כמו temperature

- ולעיתים גם Context קודם

לאחר מכן השרת של ספק ה-AI:

- מקבל את הבקשה

- מעביר אותה למודל

- מריץ inference

- ומחזיר תשובה בפורמט מסודר

המרצה הדגיש שזהו שינוי תפיסתי חשוב מאוד.

מודל השפה אינו "קסם" שיושב בתוך האפליקציה. 
ברוב המקרים מדובר בשירות מרוחק שפועל בענן, והתוכנה שלנו מתקשרת איתו בדיוק כמו שהיא מתקשרת עם:

- Database

- מערכת תשלומים

- שירות אימיילים

- או כל API אחר

זו גם הסיבה שעבודה עם LLMs מתחברת באופן טבעי לעולם ה-Backend והארכיטקטורה.

ברגע שמבינים שהמודל הוא שירות API, מתחילים להופיע מושגים מוכרים מאוד מעולם התוכנה:

- Authentication

- Rate Limits

- Timeouts

- Retries

- Monitoring

- Logging

- Error Handling

- Latency

- Billing

כלומר, AI אינו עולם נפרד מעולם התוכנה.

הוא פשוט מוסיף שכבת יכולות חדשה מעל עקרונות תוכנה קיימים.

בנוסף, המרצה הסביר שלא תמיד עובדים ישירות מול HTTP Requests גולמיים. רוב הספקים מספקים גם SDKs רשמיים שמקלים על העבודה מול ה-API.

לדוגמה:

- OpenAI SDK

- Anthropic SDK

- Gemini SDK

ה-SDK מטפל עבורנו בחלק גדול מהמורכבות:

- יצירת Requests

- Authentication

- Serialization

- Parsing

- ולעיתים גם Retry Logic

אבל מאחורי הקלעים, גם SDK בסופו של דבר שולח בקשות API רגילות לשרת של ספק ה-AI.

המרצה ניסה להעביר כאן רעיון בסיסי אך חשוב מאוד:

ברגע שמבינים שמודל שפה הוא שירות API, הרבה מהעולם של AI הופך לפחות מסתורי והרבה יותר הנדסי.



## איך מודלי LLM מתקשרים עם תוכנות

לאחר שהבנו מהו API, המרצה עבר להסביר כיצד התקשורת בפועל מתבצעת בין האפליקציה שלנו לבין מודל השפה.

הוא הדגיש שמודל LLM אינו "רץ בתוך הקוד שלנו" ברוב המקרים.

במקום זאת, האפליקציה שלנו שולחת בקשה דרך האינטרנט אל שרת מרוחק שמפעיל את המודל, והשרת מחזיר תשובה.

בפועל, זה דומה מאוד לאופן שבו מערכות מתקשרות עם שירותים אחרים בעולם התוכנה.

לדוגמה:

- אפליקציה שמדברת עם Database

- אתר שמבצע סליקה מול חברת אשראי

- מערכת שמתחברת ל-Google Maps

- או שירות ששולח אימיילים

בכל המקרים האלו קיימת תקשורת בין מערכות שונות דרך Requests ו-Responses.

גם בעולם ה-LLMs זה בדיוק אותו רעיון.

**המרצה תיאר את התהליך הכללי כך:**

Application SDK/API AI Provider Model Response

התהליך מתחיל מהאפליקציה שלנו.

האפליקציה:

- בונה Request

- מוסיפה Authentication

- שולחת Prompt

- וממתינה לתשובה

הבקשה נשלחת דרך HTTP אל ספק ה-AI, לדוגמה:

- OpenAI

- Anthropic

- Google

- Azure OpenAI

- AWS Bedrock

השרת של הספק מקבל את הבקשה ומעביר אותה למודל המתאים.

בשלב הזה מתבצע תהליך inference, כלומר המודל קורא את ה-Prompt, מחשב את ההמשך ההסתברותי המתאים ביותר, ומייצר תשובה Token אחרי Token.

לאחר מכן התשובה חוזרת בחזרה דרך ה-API אל האפליקציה שלנו.

אבל המרצה הדגיש שהסיפור לא מסתיים רק ב"הבקשה חזרה".

במערכות אמיתיות האפליקציה צריכה גם להתמודד עם:

- Timeout

- Errors

- Rate Limits

- Latency

- Parsing

- Streaming

- Logging

- Retry Logic

- ו-Monitoring

כלומר, העבודה מול מודל שפה דומה הרבה יותר לעבודה מול מערכת מבוזרת מאשר לעבודה מול פונקציה רגילה בקוד.

בנוסף, המרצה הסביר שהמודל עצמו בדרך כלל אינו מחזיר "טקסט חופשי" בלבד. 
התגובה מגיעה בפורמט structured שמכיל:

- תוכן התשובה

- Metadata

- Usage

- Token Counts

- Model Information

- ולעיתים גם מידע נוסף על הבקשה

לדוגמה:

```python
{
  "model": "claude-haiku",
  "usage": {
    "input_tokens": 120,
    "output_tokens": 45
  },
  "content": [
    {
      "text": "Hello!"
    }
  ]
}
```

זו נקודה חשובה מאוד משום שמרגע שעובדים מול APIs, אנחנו כבר לא "משוחחים עם AI".

אנחנו מתחילים לעבוד מול מערכת תוכנה מורכבת שדורשת:

- ניהול Requests

- טיפול בתשובות

- ניתוח נתונים

- בקרת עלויות

- ואינטגרציה מלאה עם שאר המערכת

המרצה ניסה להעביר כאן רעיון חשוב:

LLM הוא לא רק מודל חכם. 
הוא שירות רשת מלא שחייב להשתלב בצורה נכונה בתוך הארכיטקטורה של האפליקציה.

## מבנה בסיסי של בקשת API

אחרי שהמרצה הסביר כיצד האפליקציה מתקשרת עם מודל השפה דרך API, הוא עבר להראות איך נראית בפועל בקשה שנשלחת למודל.

זו נקודה חשובה מאוד, משום שכמעט כל העבודה מול LLMs מבוססת בסופו של דבר על אותו רעיון בסיסי:

האפליקציה שולחת Request מובנה, והמודל מחזיר Response מובנה.

למרות שלכל ספק יש API מעט שונה, רוב הבקשות בעולם ה-LLMs דומות מאוד זו לזו.

בדרך כלל בקשה למודל כוללת כמה רכיבים מרכזיים:

<div dir="rtl">

| **רכי**ב | **תפקיד** |
| --- | --- |
| **model** | איזה מודל להפעיל |
| **messages** | תוכן השיחה |
| **max_tokens** | גודל תשובה מקסימלי |
| **temperature** | רמת אקראיות |
| **system prompt** | הוראות התנהגות למודל |
| **metadata** | מידע נוסף על הבקשה |

</div>

המרצה הדגיש שבקשת API למודל אינה "משפט אחד".

זו למעשה פקודת הפעלה מלאה שמגדירה למודל:

- מי הוא

- מה הוא צריך לעשות

- איך הוא צריך לענות

- וכמה מידע להחזיר

דוגמה בסיסית:

```python
message = client.messages.create(
    model="claude-haiku-4-5",
    max_tokens=1000,
    temperature=0,
    messages=[
        {
            "role": "user",
            "content": "say hello to me"
        }
    ]
)
```

במבט ראשון הקוד נראה פשוט, אבל המרצה הסביר שבתוך כמה שורות בודדות מסתתרות הרבה מאוד החלטות חשובות.

לדוגמה:

**model**

קובע איזה מודל יריץ את הבקשה.

**messages**

מכיל את השיחה עצמה ואת ה-Context.

**temperature**

משפיע על רמת היצירתיות והאקראיות.

**max_tokens**

משפיע על אורך התשובה וגם על העלות.

כלומר, כל בקשה למודל היא למעשה שילוב של:

- Prompt Engineering

- Cost Management

- Behavior Control

- ו-Context Management

המרצה גם הדגיש שרוב ספקי ה-AI עובדים כיום במבנה של Messages ולא רק Prompt בודד.

במקום לשלוח:

"Explain Python"

שולחים מבנה מסודר של שיחה:

```python
[
  {
    "role": "system",
    "content": "You are a senior Python engineer"
  },
  {
    "role": "user",
    "content": "Explain decorators"
  }
]
```

זה מאפשר למודל להבין:

- מי המשתמש

- מה תפקיד המערכת

- ומה ההקשר של השיחה

בנוסף, המרצה הסביר שבמערכות אמיתיות הבקשות האלו עלולות להפוך למורכבות מאוד.

לעיתים Request אחד כולל:

- הוראות מערכת

- היסטוריית שיחה

- מסמכים

- תוצאות חיפוש

- Tool Calls

- Metadata

- ו-Context דינמי

ולכן ניהול נכון של Requests הופך לחלק מרכזי ב-AI Engineering.

המרצה ניסה להעביר כאן רעיון חשוב:

כאשר עובדים עם LLMs, איכות התוצאה אינה תלויה רק במודל עצמו.

היא תלויה בצורה שבה בונים ושולחים את הבקשה למודל.

## messages - מבנה השיחה עם המודל

לאחר שהמרצה הסביר כיצד בוחרים מודל, הוא עבר לאחד הרכיבים המרכזיים ביותר בכל בקשת API:

ה-messages.

זהו למעשה הלב של התקשורת עם מודל השפה.

בניגוד למערכות ישנות יותר שעבדו עם Prompt בודד, רוב מודלי ה-LLM המודרניים עובדים במבנה של שיחה. כלומר, במקום לשלוח רק טקסט אחד, שולחים למודל רצף מסודר של הודעות עם תפקידים שונים.

לדוגמה:

```python
messages=[
    {
        "role": "user",
        "content": "say hello to me"
    }
]
```

במבט ראשון זה נראה פשוט מאוד, אבל המרצה הסביר שהמבנה הזה הוא אחת הסיבות המרכזיות לכך שמודלים מודרניים מסוגלים לנהל שיחות מורכבות ולהחזיק Context לאורך זמן.

כל הודעה במערך כוללת בדרך כלל שני רכיבים מרכזיים:

<div dir="rtl">

| **רכי**ב | **משמעות** |
| --- | --- |
| **role** | מי "מדבר" |
| **content** | תוכן ההודעה |

</div>

ה-role מאפשר למודל להבין מה ההקשר של כל הודעה בתוך השיחה.

ברוב המודלים קיימים שלושה Roles מרכזיים:

<div dir="rtl">

| **Role** | **תפקיד** |
| --- | --- |
| **system** | הוראות מערכת |
| **user** | הודעת המשתמש |
| **assistant** | תשובות קודמות של המודל |

</div>

המרצה הסביר שהמודל אינו "זוכר" שיחות כמו בני אדם. 
בכל Request שולחים לו מחדש את כל ה-Context הרלוונטי בתוך מערך ה-messages.

כלומר, כאשר המשתמש שואל שאלה נוספת, המערכת בדרך כלל שולחת מחדש:

- את הוראות המערכת

- את היסטוריית השיחה

- ואת ההודעה החדשה

לדוגמה:

כך המודל מצליח להבין את רצף השיחה.

**המרצה הדגיש שזו נקודה קריטית מאוד ב-AI Engineering:**

המודל אינו מחזיק state אמיתי בין Requests.

האחריות על ניהול ה-Context נמצאת אצל האפליקציה שלנו.

זו אחת הסיבות שמערכות AI אמיתיות צריכות לדעת:

- לשמור היסטוריה

- לקצר Context

- למחוק מידע ישן

- לבצע Summarization

בנוסף, המרצה הסביר שה-messages אינם רק "צ'אט".

המרצה גם רמז לכך שסדר ההודעות משפיע מאוד על התוצאה.

שינוי קטן ב-Context או במיקום של הודעה מסוימת יכול לשנות לחלוטין את התשובה שהמודל יחזיר.

לכן בניית מערך messages נכון הפכה לאחד החלקים החשובים ביותר ב-Prompt Engineering וב-AI Systems Design.

המסר המרכזי של הסעיף היה ברור:

בונים שיחה שלמה שהמודל משתמש בה כדי להבין את העולם שבו הוא פועל כרגע.

## system prompt מול user prompt

לאחר שהמרצה הסביר את מבנה ה-messages, הוא עבר להבחנה חשובה מאוד בעולם ה-LLMs:

ההבדל בין system prompt לבין user prompt.

במבט ראשון שניהם נראים כמו טקסט רגיל שנשלח למודל, אבל בפועל הם ממלאים תפקידים שונים לחלוטין.

המרצה הסביר שאפשר לחשוב על ה-system prompt כעל "חוקי העולם" של המודל בתוך אותה שיחה.

זה המקום שבו המערכת מגדירה:

- מי המודל

- איך הוא צריך להתנהג

- מה הסגנון שלו

- מה מותר ואסור לו לעשות

- ואיך עליו לענות

לעומת זאת, ה-user prompt הוא הבקשה עצמה של המשתמש.

לדוגמה:

```python
[
  {
    "role": "system",
    "content": "You are a senior backend architect"
  },
  {
    "role": "user",
    "content": "Explain API Gateway"
  }
]
```

במקרה הזה:

- ה-system prompt מגדיר את האישיות והתפקיד

- וה-user prompt מגדיר את המשימה

המרצה הדגיש שה-system prompt הוא אחד הכלים החזקים ביותר בעולם ה-AI Engineering.

באמצעותו אפשר להשפיע על:

- סגנון התשובה

- רמת פירוט

- פורמט

- שפה

- טון כתיבה

- מבנה נתונים

- ואפילו behavior כללי של המודל

לדוגמה, אפשר להגדיר שהמודל:

- יענה כמו מורה

- יענה בצורה קצרה

- יחזיר JSON בלבד

- יסביר ברמה למתחילים

- לא ימציא מידע

- או יענה כמו מומחה בתחום מסוים

המרצה גם הסביר שבמערכות אמיתיות ה-system prompt הופך לעיתים לחלק קריטי בארכיטקטורה של המערכת.

מערכות גדולות משקיעות עבודה רבה בכתיבת system prompts יציבים, משום שהם משפיעים ישירות על:

- איכות התשובות

- יציבות ה-behavior

- עקביות

- ויכולת השליטה במודל

בנוסף, ה-system prompt משמש לעיתים גם כמנגנון הגנה.

לדוגמה:

- מניעת תשובות מסוימות

- הגבלת behavior

- שמירה על Policy

- או מניעת Prompt Injection בסיסי

אבל המרצה הדגיש נקודה חשובה מאוד:

מודלי LLM אינם "מצייתים" בצורה מוחלטת.

גם system prompt חזק מאוד אינו מבטיח שליטה מלאה במודל.

לעיתים:

- המשתמש מצליח לעקוף הוראות

- Context חדש משנה behavior

- או שהמודל פשוט מפרש את ההוראות בצורה שונה מהמצופה

זו בדיוק הסיבה שעולם ה-Prompt Engineering הפך לתחום שלם בפני עצמו.

בנוסף, המרצה הסביר שהפרדה נכונה בין system prompt לבין user prompt עוזרת מאוד בבניית מערכות מסודרות.

לדוגמה:

<div dir="rtl">

| **רכי**ב | **אחריות** |
| --- | --- |
| **system prompt** | חוקי המערכת |
| **user prompt** | בקשת המשתמש |
| **assistant messages** | Context קודם |

</div>

כך המערכת יכולה לשלוט בצורה טובה יותר בהתנהגות המודל בלי לערבב בין לוגיקת המערכת לבין תוכן המשתמש.

המרצה ניסה להעביר כאן רעיון מרכזי:

כאשר עובדים עם מודלי LLM, לא מספיק לחשוב רק על "מה המשתמש שאל".

צריך לחשוב גם על איזה עולם המערכת בונה סביב המודל לפני שהוא בכלל מתחיל לענות.

## max_tokens - שליטה בגודל התשובה

אחד הפרמטרים החשובים ביותר בכל בקשת API למודל הוא:

max_tokens

המרצה הסביר שבמבט ראשון זה נראה כמו הגדרה טכנית פשוטה, אבל בפועל לפרמטר הזה יש השפעה ישירה על:

- עלויות

- ביצועים

- Latency

- יציבות

- וחוויית המשתמש

כדי להבין את max_tokens צריך קודם להבין מהו בכלל Token.

מודלי LLM אינם "קוראים מילים" כמו בני אדם.

הם עובדים עם יחידות קטנות יותר שנקראות Tokens.

Token יכול להיות:

- מילה שלמה

- חלק ממילה

- סימן פיסוק

- מספר

- או אפילו רווח

לדוגמה:

Hello world

עלול להתפצל לכמה Tokens שונים.

כאשר שולחים בקשה למודל, גם ה-Input וגם ה-Output נמדדים לפי Tokens.

כלומר:

- משלמים על Tokens שנשלחים

- ומשלמים גם על Tokens שהמודל מחזיר

וכאן נכנס max_tokens.

הפרמטר הזה קובע מהו האורך המקסימלי של התשובה שהמודל רשאי להחזיר.

לדוגמה:

```python
max_tokens=1000
```

פירושו:

"המודל יכול לייצר עד 1000 Tokens בתשובה."

המרצה הדגיש שזה לא אומר שהמודל תמיד יחזיר 1000 Tokens.

זו רק מגבלה עליונה.

אם התשובה מסתיימת קודם, המודל יעצור מוקדם יותר.

אבל אם התשובה דורשת יותר Tokens מהמגבלה שהוגדרה, המודל פשוט ייחתך באמצע.

זו אחת הבעיות הנפוצות ביותר אצל מפתחים חדשים.

לדוגמה:

- תשובה שנקטעת באמצע משפט

- JSON שנשבר באמצע

- קוד לא שלם

- או הסבר שנעצר לפני הסוף

ולכן בחירת max_tokens צריכה להתאים למשימה.

לדוגמה:

<div dir="rtl">

| **סוג משימ**ה | **max_tokens מתאים** |
| --- | --- |
| **תשובת צ'אט קצרה** | קטן יחסית |
| **סיכום מסמך** | בינוני |

| **Agent מורכב** | גדול משמעותית |

</div>

בנוסף, המרצה הסביר של-max_tokens יש השפעה ישירה גם על העלות.

ככל שהמודל מייצר יותר Tokens:

- הקריאה יקרה יותר

- זמן התגובה ארוך יותר

- והעומס על המערכת גדל

זו הסיבה שבמערכות אמיתיות לא נהוג פשוט להגדיר ערכים עצומים "ליתר ביטחון".

צריך למצוא איזון נכון בין:

- איכות תשובה

- עלות

- מהירות

- וגודל Response

המרצה גם רמז לכך ש-max_tokens קשור ישירות ל-Context Window.

לכל מודל קיימת מגבלה כוללת של Tokens:

Input Tokens + Output Tokens

כלומר, אם שולחים Context ארוך מאוד, נשאר פחות מקום לתשובה עצמה.

זו אחת הסיבות שמערכות AI גדולות צריכות לבצע:

- Context Compression

- Summarization

- Trimming

- Token Budgeting

כדי לנהל נכון את ה-Context.

בנוסף, המרצה הדגיש שבמערכות Production חשוב מאוד לנטר Usage של Tokens.

ספקי AI בדרך כלל מחזירים מידע כמו:

```python
"usage": {

  "output_tokens": 340
}
```

המידע הזה חשוב עבור:

- Billing

- Optimization

- Monitoring

- Cost Analysis

- ו-Capacity Planning

**המרצה ניסה להעביר כאן רעיון חשוב:**

בעולם ה-LLMs, Tokens הם המשאב המרכזי של המערכת.

ולכן שליטה נכונה ב-max_tokens היא לא רק עניין טכני קטן, אלא חלק משמעותי מהארכיטקטורה, העלויות והביצועים של מערכת AI אמיתית.

</div>

כאשר temperature נמוך, המודל נוטה לבחור את ה-Tokens עם ההסתברות הגבוהה ביותר.

כאשר temperature גבוה יותר, המודל מוכן "להסתכן" ולבחור גם Tokens פחות צפויים.

המרצה הדגיש שזו אחת הסיבות לכך שאותה שאלה יכולה להחזיר תשובות שונות.

- לנסח אחרת

- לחשוב בצורה שונה

- או לייצר תשובות יצירתיות יותר

אבל כאן נכנס Tradeoff חשוב מאוד.

ככל שה-temperature גבוה יותר:

- היצירתיות עולה

- אבל גם הסיכון לטעויות והזיות עולה

זו נקודה קריטית במערכות אמיתיות.

המרצה הסביר שבמערכות Production רבות נהוג להשתמש ב-temperature נמוך יחסית כדי לקבל:

## top_p - שליטה מתקדמת בתהליך ה-Sampling

לאחר שהמרצה הסביר על temperature, הוא עבר לפרמטר מתקדם יותר שנקרא:

top_p

המרצה ציין שבמערכות בסיסיות הרבה פעמים מספיק לעבוד עם temperature בלבד, אבל בעולם מקצועי יותר מתחילים להופיע מנגנוני Sampling מתקדמים שמאפשרים שליטה עדינה יותר על האופן שבו המודל בוחר Tokens.

כדי להבין את **top_p** צריך לזכור שמודל שפה עובד בצורה הסתברותית.

בכל שלב המודל מחשב רשימה עצומה של Tokens אפשריים יחד עם הסתברות לכל אחד מהם.

לדוגמה:

<div dir="rtl">

| **Token** | **Probability** |
| --- | --- |
| **blue** | 0.52 |
| **clear** | 0.18 |
| **cloudy** | 0.11 |
| **beautiful** | 0.07 |
| **dark** | 0.03 |

</div>

temperature משנה את "רמת האקראיות" של הבחירה.

לעומת זאת, top_p עובד בצורה אחרת.

במקום לשנות את ההסתברויות עצמן, הוא מגביל את קבוצת ה-Tokens שהמודל בכלל רשאי לבחור מתוכה.

זה נקרא:

Nucleus Sampling

הרעיון הוא שהמודל סוכם Tokens לפי הסתברות עד שמגיעים לערך top_p שהוגדר.

לדוגמה:

```python
top_p = 0.9
```

פירושו:

"המודל רשאי לבחור רק מתוך קבוצת ה-Tokens שמכסה יחד 90% מההסתברות הכוללת."

כל Tokens עם הסתברות נמוכה מאוד פשוט נחתכים החוצה.

זו דרך חכמה למנוע מהמודל לבחור Tokens "מוזרים" או אקראיים מדי.

המרצה הדגיש ש-top_p מאפשר שליטה עדינה יותר מאשר temperature בלבד.

לדוגמה:

<div dir="rtl">

| **top_p** | **Behavior** |
| --- | --- |
| **נמוך** | תשובות צפויות וממוקדות |
| **בינוני** | איזון בין יציבות לגיוון |
| **גבוה** | מגוון רחב יותר של תשובות |

</div>

בפועל, הרבה מערכות משתמשות ב-top_p כדי לשמור על איכות ויציבות גם כאשר רוצים מעט יצירתיות.

בנוסף, המרצה הסביר שבדרך כלל לא נהוג "לשחק" חזק גם עם temperature וגם עם top_p באותו זמן, משום ששני הפרמטרים משפיעים על אותו שלב בתהליך ה-Sampling.

במערכות רבות:

- משתמשים בעיקר ב-temperature

- או עובדים עם top_p קטן יחסית ו-temperature יציב

המרצה גם ציין שבמערכות Production לא תמיד רוצים מקסימום יצירתיות.

לפעמים דווקא חשוב:

- לקבל תשובות יציבות

- להקטין hallucinations

- לשמור על consistency

- ולהפחית variability

במקרים כאלה top_p יכול לעזור מאוד.

לדוגמה:

<div dir="rtl">
