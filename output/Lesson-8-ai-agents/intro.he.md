# פתיחה - ממודל שעונה למערכת שפועלת

למה בכלל צריך Agent?

בשנים האחרונות התרגלנו לעבוד עם מודלי שפה גדולים שמסוגלים לענות על שאלות, להסביר מושגים, לכתוב קוד, לסכם טקסטים ולנהל שיחה טבעית. זו יכולת חזקה מאוד, אבל היא עדיין לא מספיקה לכל מערכת אמיתית. בעולם האמיתי אנחנו לא תמיד רוצים רק תשובה. לפעמים אנחנו רוצים שמערכת תבדוק מידע, תחפש במסמכים, תפעיל כלי, תקבל החלטה בתוך תהליך, ותבצע פעולה בצורה מבוקרת.

כאן נכנס הרעיון של AI Agent. Agent אינו רק מודל שמחזיר טקסט. הוא שכבת מערכת שמחברת בין מודל שפה, הנחיות, זיכרון, כלים ותהליך עבודה. במילים פשוטות, הוא מנסה להפוך את ה-LLM ממנוע תשובות לרכיב בתוך מערכת שיכולה לפעול.

## למה מודל שפה לבדו לא מספיק

מודל שפה גדול יודע לבצע דבר מרשים מאוד: לקבל טקסט ולהחזיר טקסט. המשתמש כותב שאלה, בקשה או הוראה, והמודל מייצר תשובה שנראית טבעית, ברורה ולעיתים גם מאוד חכמה.

לדוגמה, אפשר לשאול מודל שפה:

```bash
Explain what RAG means in simple words.
```

והוא יחזיר הסבר. אפשר לבקש ממנו:

```bash
Write a Python function that calculates the average of a list.
```

והוא יכתוב קוד. אפשר גם לתת לו טקסט ארוך ולבקש סיכום.

אבל במערכות אמיתיות יש מגבלה חשובה: המודל עצמו אינו באמת “יוצא לעולם”. הוא לא קורא קובץ חדש מהמחשב, לא ניגש למסד נתונים, לא מפעיל API, לא בודק מחיר מניה בזמן אמת, לא יוצר רשומה במערכת ארגונית, ולא יודע בוודאות מה השתנה אחרי שהמידע שעליו אומן כבר לא עדכני.

כלומר, LLM לבדו טוב מאוד בעיבוד שפה, אבל מוגבל כאשר צריך לעבוד עם מידע חיצוני או לבצע פעולה.

ניקח דוגמה פשוטה. משתמש שואל:

```bash
What is the current stock price of Apple?
```

אם המערכת היא רק מודל שפה ללא כלי חיצוני, יש לה בעיה. מחיר מניה הוא נתון משתנה. תשובה שמבוססת רק על ידע פנימי של המודל עלולה להיות לא עדכנית או פשוט לא נכונה. במקרה כזה, המערכת צריכה להבין שהשאלה דורשת מידע חי, להפעיל כלי שמביא נתוני שוק, לקבל את התוצאה, ורק אז לנסח תשובה למשתמש.

זו בדיוק הנקודה שבה מתחילים לחשוב על Agent.

במקום לראות את המודל כמערכת שלמה, אנחנו מתחילים לראות אותו כרכיב אחד בתוך מערכת רחבה יותר:

```bash
User request
     ↓
LLM understands the request
     ↓
The system decides whether external information or action is needed
     ↓
A tool is used if needed
     ↓
The final answer is returned to the user
```

ההבדל הזה חשוב מאוד. מודל שפה רגיל בעיקר **עונה**. 
Agent יכול, בתכנון נכון, גם **לבדוק**, **לשלוף**, **להפעי**ל, **להחליט ולבצע.**

עם זאת, חשוב לא לקפוץ מהר מדי למסקנה שכל דבר צריך להיות Agent. לפעמים מספיק Prompt טוב. לפעמים מספיק תהליך קבוע. לפעמים מספיק RAG פשוט. Agent נהיה רלוונטי כאשר יש צורך בשילוב של הבנה, מידע חיצוני, כלים, החלטות ופעולות.

## ההבדל בין Chatbot רגיל, RAG Bot, Workflow ו-Agent

כדי להבין AI Agents בצורה טובה, כדאי לא להתחיל ישר מהמערכת הכי מורכבת. עדיף לראות את זה כרצף של יכולות. בכל שלב המערכת מקבלת עוד יכולת אחת: קודם היא רק עונה, אחר כך היא גם שולפת מידע, אחר כך היא פועלת לפי תהליך, ובהמשך היא יכולה לקבל החלטות בתוך התהליך.

אפשר לראות את הרצף כך:

```bash
Chatbot → RAG Bot → Workflow → Agentic Workflow → AI Agent
```

זה לא רצף רשמי או חד-משמעי, אלא דרך נוחה לחשוב על רמות שונות של מערכת.

**Chatbot רגיל**

Chatbot רגיל הוא מערכת שמקבלת הודעה מהמשתמש ומחזירה תשובה. בדרך כלל היא מתבססת על הידע הפנימי של המודל ועל מה שנמצא בפרומפט הנוכחי.

לדוגמה:

```bash
User: Explain what embeddings are.
Assistant: Embeddings are numerical representations of text...
```

זה שימושי מאוד להסברים, כתיבה, סיכום, ניסוח, תרגום, עזרה בקוד ושיחה כללית. אבל ה-Chatbot הרגיל לא בהכרח יודע לגשת למידע פרטי, למסמכים פנימיים או לנתונים עדכניים.

אם נשאל אותו על מסמך שנמצא אצלנו במחשב, והוא לא קיבל את המסמך כחלק מהקלט, אין לו דרך אמיתית לדעת מה כתוב בו. אם נשאל אותו על נתון שמשתנה בזמן אמת, כמו מחיר מניה או סטטוס משלוח, הוא יצטרך מקור מידע חיצוני.

כאן נכנס השלב הבא.

**RAG Bot**

RAG Bot הוא Chatbot שמחובר למקור ידע חיצוני. במקום לענות רק מהידע הפנימי של המודל, הוא קודם מחפש מידע רלוונטי במסמכים, ואז שולח את המידע הזה למודל כחלק מה-context.

הזרימה הבסיסית נראית כך:

User question 
 ↓ 
Search relevant documents 
 ↓ 
Retrieve context 
 ↓ 
Send question + context to the LLM 
 ↓ 
Return grounded answer

לדוגמה, אם המשתמש שואל:

```bash
What does our internal document say about ChromaDB?
```

המערכת לא צריכה לנחש. היא יכולה לחפש במסמכים, לשלוף את הקטעים הרלוונטיים, ורק אז לבקש מהמודל לענות לפי המידע שנמצא.

זה כבר שיפור גדול לעומת Chatbot רגיל, אבל עדיין לא בהכרח מדובר ב-Agent. בהרבה מערכות RAG, התהליך קבוע מאוד: תמיד מקבלים שאלה, תמיד מחפשים במסמכים, תמיד שולחים context למודל, ותמיד מחזירים תשובה.

כלומר, RAG מוסיף יכולת שליפה, אבל לא בהכרח מוסיף יכולת החלטה רחבה.

**Workflow**

Workflow הוא תהליך מסודר עם שלבים ידועים מראש. במקום לתת למודל להחליט מה לעשות בכל רגע, אנחנו מגדירים את המסלול.

לדוגמה, תהליך ליצירת תשובה איכותית יכול להיראות כך:

Generate draft 
 ↓ 
Review draft 
 ↓ 
Improve wording 
 ↓ 
Return final answer

או בעברית פשוטה:

1. כתוב טיוטה.

2. בדוק אם היא ברורה.

3. שפר את הניסוח.

4. החזר תשובה סופית.

היתרון של Workflow הוא צפיות. קל יותר להבין מה המערכת עושה, קל יותר לבדוק אותה, וקל יותר לתקן שלב מסוים אם הוא לא עובד טוב.

זה מתאים במיוחד לתהליכים שבהם אנחנו יודעים מראש מה סדר הפעולות הרצוי. לדוגמה: סיכום מסמך, בדיקת קוד, יצירת מייל, ניתוח טקסט, או הפקת דוח.

אבל Workflow רגיל פחות מתאים למצבים שבהם לא יודעים מראש מה הצעד הבא. אם כל בקשה דורשת תהליך אחר, או אם המערכת צריכה לבחור בין כמה כלים לפי המצב, אנחנו מתקרבים לעולם Agentic יותר.

**Agentic Workflow**

Agentic Workflow הוא עדיין תהליך יחסית מסודר, אבל יש בו נקודות שבהן המודל מקבל החלטה.

```bash
לדוגמה:
User question
     ↓
Classify request type
     ↓
Choose route
     ↓
Run the matching workflow
     ↓
Return answer
```

במקרה כזה, לא כל התהליך חופשי. יש מסגרת ברורה. אבל בתוך המסגרת יש החלטה חכמה: לאיזה מסלול לשלוח את הבקשה.

נניח שיש לנו מערכת תמיכה:

```bash
Billing question    → Billing flow
Technical question  → Technical flow
Document question   → RAG flow
General question    → General assistant
```

כאן המודל לא עושה את הכול לבד. הוא בעיקר עוזר לבחור מסלול. אחרי הבחירה, כל מסלול יכול להיות מוגדר היטב.

זו גישה מעשית מאוד, כי היא נותנת איזון טוב: מצד אחד יש גמישות, מצד שני המערכת עדיין מבוקרת.

**AI Agent**

AI Agent הוא מערכת שבה יש יותר יכולת החלטה בזמן ריצה. הוא יכול להתבונן במצב, לחשוב מה הצעד הבא, להפעיל כלי, לבדוק את התוצאה, ואז להחליט אם להמשיך, לשנות כיוון או להחזיר תשובה.

אפשר לתאר את זה כך:

```bash
observe → think → act → observe → think → act
```

לדוגמה, משתמש מבקש:

```bash
Check if Apple stock looks stronger or weaker this year and explain why.
```

מערכת פשוטה אולי תנסה מיד. 
מערכת Agentic יותר יכולה לעבוד בשלבים:

1. להבין שמדובר בשאלה על מניה.

2. לזהות שצריך נתונים היסטוריים.

3. להפעיל כלי שמביא נתוני מניה.

4. לבדוק מחיר בתחילת השנה מול מחיר נוכחי.

5. אולי להפעיל כלי נוסף לחדשות או המלצות אנליסטים.

6. לאחד את המידע.

7. להחזיר תשובה זהירה וברורה.

הנקודה החשובה היא שה-Agent לא רק עובר במסלול קבוע. הוא יכול להחליט מה לעשות לפי המצב.

עם זאת, יותר אוטונומיה פירושה גם יותר סיכון. ככל שהמערכת מחליטה יותר לבד, קשה יותר לצפות כל צעד מראש. לכן במערכות אמיתיות לא תמיד נרצה Agent חופשי לגמרי. לעיתים נעדיף Workflow ברור עם נקודות החלטה מוגבלות.

הבחירה הנכונה אינה “כמה שיותר Agent”, אלא כמה אוטונומיה באמת דרושה לבעיה. מערכת טובה היא לא בהכרח המערכת החופשית ביותר, אלא המערכת שמבצעת את המשימה בצורה הכי אמינה, ברורה ובטוחה.

## דוגמה פשוטה: שאלה על מחיר מניה

כדי להפוך את ההבדל למוחשי, נשתמש בדוגמה קטנה שחוזרת הרבה בעולם של Agents: משתמש שואל על מחיר מניה.

נניח שהמשתמש כותב:

```bash
What is the current stock price of Apple?
```

זו נראית כמו שאלה פשוטה, אבל מבחינת מערכת AI יש כאן פרט חשוב: מחיר מניה הוא נתון משתנה. הוא יכול להשתנות במהלך יום המסחר, להשתנות בין מקורות מידע, להיות מושפע ממטבע, בורסה, זמן עדכון, ואפילו מהשאלה האם השוק פתוח או סגור.

אם נשלח את השאלה הזאת למודל שפה רגיל, הוא יכול להבין שמדובר ב-Apple, להבין שהכוונה היא כנראה ל-AAPL, ולנסח תשובה יפה. אבל בלי כלי חיצוני אין לו דרך אמינה לדעת את המחיר הנוכחי.

הבעיה אינה שהמודל “לא חכם”. הבעיה היא שהוא לא מחובר למקור הנתונים הנדרש.

מערכת פשוטה מדי עלולה להחזיר תשובה שנשמעת בטוחה:

```bash
Apple stock is currently around...
```

אבל זו בדיוק הבעיה. אם הנתון לא נבדק בזמן אמת, התשובה יכולה להיות לא נכונה גם אם היא מנוסחת מצוין.

במערכת Agentic, הזרימה צריכה להיות אחרת:

```bash
User asks about Apple stock
        ↓
The system understands this requires current market data
        ↓
The system maps Apple to the ticker AAPL
        ↓
The system calls a stock information tool
        ↓
The tool returns the current data
        ↓
The LLM explains the result clearly
```

כאן ה-LLM עדיין חשוב מאוד, אבל הוא לא עובד לבד. הוא עוזר להבין את הבקשה, לבחור את הפעולה המתאימה, ולנסח את התוצאה למשתמש. את הנתון עצמו מביא כלי חיצוני.

זו נקודה מרכזית בהבנת Agents:

המודל לא חייב לדעת הכול בעצמו. במקום זאת, הוא צריך לדעת מתי להשתמש במערכת חיצונית.

אפשר לחשוב על זה כמו עובד מקצועי שמקבל שאלה. אם שואלים אותו שאלה כללית, הוא יכול לענות מהידע שלו. אבל אם שואלים אותו על נתון עדכני, הוא לא אמור לנחש. הוא אמור לבדוק במערכת המתאימה ואז להסביר את התוצאה.

אותו עיקרון נכון גם בדוגמאות אחרות:

```bash
What is the status of shipment 12345?
```

כאן צריך כנראה כלי שמחובר למערכת משלוחים.

```bash
What does the company policy say about remote work?
```

כאן צריך כנראה חיפוש במסמכים או RAG.

```bash
Create a support ticket for this issue.
```

כאן צריך כלי שיוצר רשומה במערכת חיצונית.

```bash
Check this code file and suggest improvements.
```

כאן ייתכן שצריך לקרוא קובץ, לנתח אותו, אולי להריץ בדיקות, ואז להחזיר המלצות.

בכל הדוגמאות האלה, תשובה טקסטואלית בלבד אינה תמיד מספיקה. המערכת צריכה להבין מה המשתמש מבקש, לזהות איזה מידע חסר, לבחור דרך להשיג אותו, ורק אז להחזיר תשובה.

זה ההבדל בין מערכת שמדברת על פעולה לבין מערכת שמסוגלת להשתתף בפעולה.

חשוב לשים לב לעוד נקודה: גם Agent לא צריך לפעול בלי גבולות. אם המשתמש שואל על מחיר מניה, אפשר להפעיל כלי שמחזיר מידע. אבל אם המשתמש מבקש לבצע פעולה כספית, לשלוח מייל, למחוק קובץ או לעדכן נתונים במערכת, נצטרך לחשוב על הרשאות, אישור משתמש, תיעוד, ובדיקות נוספות.

כלומר, Agent טוב אינו רק מערכת שמסוגלת להפעיל כלים. Agent טוב הוא מערכת שיודעת מתי להפעיל כלי, איך לפרש את התוצאה, ומתי לעצור ולבקש אישור.



## מה נלמד בשיעור

אחרי שהבנו למה מודל שפה לבדו לא תמיד מספיק, אפשר לראות את המטרה של השיעור בצורה ברורה יותר: ללמוד איך בונים סביב ה-LLM מערכת שלמה.

לא נתחיל מהקוד מיד. קודם נבנה את ההבנה: מהו Agent, אילו רכיבים יש לו, למה הוא צריך זיכרון, איך הוא משתמש בכלים, ומה ההבדל בין תהליך קבוע מראש לבין מערכת שמקבלת החלטות בזמן ריצה.

הנקודה החשובה היא לא רק לדעת להריץ דוגמה, אלא להבין את המבנה שמאחוריה. אם מבינים את המבנה, אפשר אחר כך להחליף מודל, להוסיף כלי, לחבר מקור מידע חדש, לשנות את דרך השליפה, או לבנות תהליך עבודה אחר לגמרי.

במהלך השיעור נתקדם בכמה שכבות.

קודם נבין מהו AI Agent. נראה שהוא אינו רק מודל שפה, אלא מערכת שמחברת בין בקשת המשתמש, מודל שפה, הנחיות, זיכרון, כלים ותוצאה. נלמד גם מה Agent אינו, כדי לא להשתמש במונח הזה לכל דבר שמערב LLM.

לאחר מכן נפרק את הארכיטקטורה לרכיבים. נבין מה תפקיד ה-LLM, מהו Memory, מהו Tool, ומהן Instructions. כל רכיב כזה נראה פשוט בפני עצמו, אבל החיבור ביניהם הוא מה שיוצר מערכת שימושית.

בהמשך נלמד את ההבדל בין Workflow מובנה לבין Agent Loop. זה אחד ההבדלים החשובים ביותר בבניית מערכות AI מעשיות. Workflow מובנה מתאים כאשר אנחנו יודעים מראש את השלבים. Agent Loop מתאים יותר כאשר המערכת צריכה להחליט בזמן ריצה מה לעשות בהמשך. ההבדל הזה משפיע על אמינות, בדיקות, תחזוקה, סיכון ועלות.

אחרי זה נכיר דפוסי עבודה נפוצים לבניית מערכות Agentic. לא כל מערכת צריכה להיות Agent חופשי. לפעמים הפתרון הנכון הוא Prompt Chaining. לפעמים Routing. לפעמים Parallelization. לפעמים Orchestrator-workers. לפעמים Evaluator-optimizer. ולפעמים באמת צריך Agent Loop דינמי יותר.

בחלק המעשי נבנה את ההבנה דרך קוד.

נתחיל מהקבצים הבסיסיים של סביבת העבודה. לפני שקוראים קוד מורכב, חשוב לדעת מה תפקיד כל קובץ בפרויקט. נבין אילו ספריות צריך, מה תפקיד קובץ הדוגמה, ואיך הקבצים מתחברים אחד לשני.

אחר כך נבנה בסיס RAG. זה השלב שבו לוקחים מסמכי טקסט, מחלקים אותם ליחידות קטנות יותר, יוצרים להן embeddings, ושומרים אותן ב-vector store. המטרה היא לא רק “לבנות מאגר”, אלא להבין איך טקסט רגיל הופך למידע שאפשר לחפש בו לפי משמעות.

לאחר מכן נבנה RAG Chatbot. כאן נראה איך שאלה של משתמש הופכת לחיפוש במאגר, איך מתקבל context רלוונטי, ואיך ה-context הזה נשלח למודל יחד עם השאלה. נלמד גם למה צריך היסטוריית שיחה, ואיפה היא יכולה לעזור או להזיק.

בשלב הבא נעבור ל-Stock Agent. זו כבר דוגמה שמראה מעבר חשוב: המערכת לא רק שולפת מידע ממסמכים, אלא מפעילה כלי חיצוני. המשתמש שואל על מניה, ה-Agent מזהה שצריך מידע עדכני, מפעיל tool שמביא נתונים, ואז מחזיר תשובה.

נוסיף גם חלק חשוב על עבודה עם סוכן קוד. המטרה היא ללמוד איך לבנות קוד כזה בצורה הדרגתית: איך לנסח משימה, איך לבקש קובץ מסוים, איך להגדיר פונקציות, איך לבקש טיפול בשגיאות, ואיך לא לתת לסוכן הקוד לשנות יותר מדי דברים בבת אחת.

בהמשך נראה איך אותה ארכיטקטורה יכולה לעבוד גם עם מודל בענן וגם עם מודל מקומי. זה חשוב כי מערכת טובה לא אמורה להיות תלויה לגמרי בספק מודל אחד. אם הקוד בנוי נכון, אפשר להחליף את שכבת המודל בלי לשבור את הכלים, הזיכרון ותהליך העבודה.

בסוף נתרגל הרחבות מעשיות: מעבר למודל מקומי, הוספת כלים חדשים, בדיקת שינוי מחיר מניה לאורך זמן, בניית ממשק משתמש, ובניית Workflow דו-שלבי. התרגול נועד להעביר את הקורא מקריאה להבנה פעילה: לא רק להבין מה הקוד עושה, אלא לדעת איך להרחיב אותו בעצמו.

המטרה הסופית היא שתסיים את השיעור עם תמונה ברורה: Agent הוא לא מילת קסם ולא מוצר מוכן. זה דפוס בנייה של מערכת. ברגע שמבינים את הדפוס, אפשר לבנות מערכות AI הרבה יותר שימושיות, אבל גם יותר מבוקרות, צפויות ובטוחות.
