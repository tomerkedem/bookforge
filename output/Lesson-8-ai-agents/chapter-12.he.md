# בניית Stock Agent עם Tool

עד עכשיו בנינו מערכת RAG שמקבלת שאלה, מחפשת מידע במסמכים, ומחזירה תשובה על בסיס context. זו מערכת חשובה מאוד, אבל היא עדיין בעיקר מערכת של שליפה וניסוח.

בחלק הזה נעשה צעד נוסף: נבנה Agent שמסוגל להפעיל כלי חיצוני.

במקום להסתמך רק על מסמכים שנשמרו מראש, הסוכן יוכל לגשת למקור מידע חיצוני בזמן הריצה, להביא נתונים עדכניים, ואז להשתמש בהם כדי לענות למשתמש.

הקובץ המרכזי בחלק הזה הוא:

```bash
stock_agent.py
```

הדוגמה שנבנה היא Stock Agent: סוכן שמקבל שאלה על מניה, מזהה שצריך מידע פיננסי, מפעיל tool שמביא נתונים על המניה, ואז מחזיר תשובה מסודרת למשתמש.

זה מעבר חשוב מאוד בהבנת Agents. כאן המודל כבר לא רק “עונה”. הוא מקבל החלטה להשתמש בכלי, מפעיל אותו, מקבל תוצאה, ומשתמש בתוצאה כדי לבנות תשובה.

## מה אנחנו בונים

אנחנו בונים Agent פשוט שמטרתו לענות על שאלות בסיסיות על מניות.

לדוגמה, המשתמש יכול לשאול:

```bash
What is the current price of MSFT?
```

או:

```bash
Give me market data for Apple stock.
```

במקום שהמודל ינסה לענות מתוך הידע הפנימי שלו, הוא צריך להבין שזו שאלה שדורשת נתון חיצוני ועדכני. מחיר מניה הוא לא מידע שכדאי להמציא, והוא גם לא מידע שמתאים להישמר מראש במסמך סטטי. זה מידע שמשתנה כל הזמן.

לכן הסוכן משתמש בכלי.

הזרימה הכללית נראית כך:

User question

↓

Agent understands the request

↓

Agent chooses the stock tool

↓

Tool fetches market data

↓

Agent receives the result

↓

Agent returns a clear answer

במילים פשוטות, המשתמש לא מפעיל את הכלי בעצמו. הוא לא צריך לדעת איך קוראים לפונקציה, איזה API עומד מאחוריה, או איך מביאים נתוני מניה. הוא שואל בשפה טבעית, וה-Agent מחליט איך לבצע את הפעולה.

זה ההבדל המרכזי בין פונקציה רגילה לבין Agent עם Tools.

בפונקציה רגילה, אנחנו כותבים במפורש:

```python
get_stock_info("MSFT")
```

אבל ב-Agent, המשתמש כותב:

```bash
What is the current price of Microsoft stock?
```

והסוכן צריך להבין לבד שהכוונה היא כנראה ל-MSFT, להפעיל את הכלי המתאים, ואז להסביר את התוצאה.

ה-Agent מורכב משלושה רכיבים מרכזיים:

```bash
LLM + Tools + Instructions
```

ה-LLM אחראי להבין את הבקשה ולנסח תשובה.

ה-Tools מאפשרים לסוכן לבצע פעולות או להביא מידע חיצוני.

ה-Instructions מגדירות לסוכן איך להתנהג, מתי להשתמש בכלי, ואיך להחזיר תשובה למשתמש.

בדוגמה שלנו, הכלי המרכזי הוא פונקציה שמביאה נתוני מניה. בהמשך נראה איך מגדירים אותה כ-tool, איך היא משתמשת ב-yfinance, ואיך מחברים אותה ל-Agent.



## הגדרת Tool

כדי ש-Agent יוכל לבצע פעולה חיצונית, צריך לתת לו כלי שהוא יודע להפעיל. במקרה שלנו, הכלי הוא פונקציה שמקבלת סימול של מניה ומחזירה עליה מידע.

בקובץ stock_agent.py הפונקציה נקראת:

```python
get_stock_info
```

הפונקציה מוגדרת כ-tool בעזרת decorator:

```python
@tool
def get_stock_info(symbol: str) -> str:
```

הסימון @tool אומר ל-LangChain:

הפונקציה הזאת אינה רק פונקציית Python רגילה. זו פונקציה שה-Agent רשאי להפעיל כחלק מתהליך החשיבה והפעולה שלו.

הפונקציה מקבלת פרמטר אחד:

```python
symbol: str
```

הפרמטר הזה הוא ticker symbol של מניה. לדוגמה:

```bash
AAPL
MSFT
TSLA
NVDA
```

כאשר המשתמש שואל על מניה, ה-Agent צריך להבין איזה ticker מתאים לשאלה, ואז להפעיל את הכלי עם אותו סימול.

לדוגמה, אם המשתמש שואל:

```bash
What is the current price of Apple stock?
```

ה-Agent אמור להבין שהסימול של Apple הוא כנראה:

```bash
AAPL
```

ואז להפעיל את הכלי כאילו קרא לפונקציה:

```python
get_stock_info("AAPL")
```

בתוך הפונקציה מתבצעת קודם כל בדיקה בסיסית של הקלט:

```python
symbol = symbol.strip().upper()
if not symbol:
    return "Error: Please provide a stock ticker symbol."
```

הקריאה ל-strip() מסירה רווחים מיותרים, והקריאה ל-upper() הופכת את הסימול לאותיות גדולות. זה חשוב כי ticker symbols נכתבים בדרך כלל באותיות גדולות, כמו MSFT ולא msft.

אם המשתמש לא סיפק סימול בכלל, הפונקציה מחזירה הודעת שגיאה ברורה במקום להמשיך לקרוא ל-API עם ערך ריק.

זו נקודה חשובה מאוד בבניית tools:

כלי טוב לא מניח שהקלט תמיד תקין. הוא בודק את הקלט, מטפל במקרים בעייתיים, ומחזיר תשובה שה-Agent יכול להבין ולהציג למשתמש.

ה-tool מחזיר מחרוזת טקסט רגילה. כלומר, הוא לא מחזיר אובייקט מורכב או מסך גרפי, אלא תשובה כתובה שאפשר להעביר חזרה ל-Agent.

במילים פשוטות, ה-tool הוא הגשר בין השפה הטבעית של המשתמש לבין פעולה אמיתית בקוד.

המשתמש כותב שאלה רגילה.

ה-Agent מבין שצריך נתוני מניה.

ה-tool מבצע את העבודה הטכנית.

וה-Agent משתמש בתוצאה כדי לנסח תשובה ברורה.



## שימוש ב-yfinance

אחרי שהגדרנו את הפונקציה כ-tool, צריך להבין מאיפה הכלי מביא את המידע. במקרה שלנו, הכלי משתמש בספרייה בשם yfinance.

yfinance מאפשרת להביא נתוני שוק על מניות לפי ticker symbol. כלומר, במקום שאנחנו נכתוב בעצמנו חיבור ישיר ל-API פיננסי, הספרייה נותנת לנו דרך פשוטה לבקש מידע על מניה.

בתוך get_stock_info, הקוד יוצר אובייקט מניה כך:

```python
ticker = yf.Ticker(symbol)
info = ticker.info
```

המשתנה symbol הוא הסימול שה-Agent העביר לכלי. לדוגמה:

```bash
AAPL
MSFT
NVDA
TSLA
```

כאשר קוראים ל-yf.Ticker(symbol), אנחנו בעצם אומרים:

צור אובייקט שמייצג את המניה הזאת, כדי שנוכל לשלוף עליה נתונים.

לאחר מכן, ticker.info מחזיר מילון גדול של נתונים. מתוך המילון הזה אפשר לקחת רק את השדות שמעניינים אותנו.

לדוגמה, הכלי מנסה להביא את המחיר הנוכחי:

```python
price = info.get("currentPrice") or info.get("regularMarketPrice") or info.get("previousClose")
```

השורה הזאת בנויה בצורה חכמה. היא לא מניחה שתמיד יהיה שדה אחד קבוע. היא מנסה קודם את currentPrice, ואם הוא לא קיים, מנסה את regularMarketPrice, ואם גם הוא לא קיים, משתמשת ב-previousClose.

זה חשוב, כי נתונים פיננסיים לא תמיד מגיעים באותה צורה עבור כל מניה, כל בורסה או כל זמן ביום.

בנוסף למחיר, הכלי שולף גם נתונים נוספים כאשר הם זמינים:

```python
name = info.get("longName") or info.get("shortName") or symbol
day_high = info.get("dayHigh")
day_low = info.get("dayLow")
volume = info.get("volume")
currency = info.get("currency", "")
```

הנתונים האלה מאפשרים להחזיר תשובה עשירה יותר:

```bash
Company name
Current price
Currency
Day high
Day low
Volume
```

לדוגמה, במקום להחזיר רק:

```bash
MSFT price is 430
```

אפשר להחזיר תשובה מסודרת יותר:

```bash
Microsoft Corporation
Current price: 430 USD
Day range: 425 - 433
Volume: 21,000,000
```

ה-Agent מקבל את התוצאה הזאת מה-tool, ואז יכול לנסח אותה בשפה טבעית למשתמש.

חשוב להבין את חלוקת התפקידים:

```bash
yfinance  →  brings raw market data
Tool      →  formats the data
Agent     →  explains the result to the user
```

כלומר, yfinance אינה ה-Agent. היא רק מקור הנתונים.

ה-tool הוא שכבת הקוד שמביאה ומארגנת את הנתונים.

ה-Agent הוא הרכיב שמחליט מתי להשתמש בכלי ואיך להציג את התוצאה.

זו נקודה מרכזית בבניית Agents: כלי חיצוני לא חייב להיות מורכב. לפעמים tool הוא פשוט פונקציה קטנה שעוטפת ספרייה קיימת, אבל מבחינת ה-Agent היא פותחת יכולת חדשה לגמרי.



## הנחיות שימוש בכלי

אחרי שהגדרנו tool שמסוגל להביא נתוני מניה, צריך להסביר ל-Agent מתי להשתמש בו. הכלי עצמו לא מספיק. ה-Agent צריך לקבל הנחיות שמגדירות באילו מצבים עליו להפעיל את הכלי, ובאילו מצבים לא.

ב-stock_agent.py ההנחיות מוגדרות בתוך SYSTEM_PROMPT:

```python
SYSTEM_PROMPT = """
You are a helpful assistant.
When the user asks about a stock price, quote, or market data,
use the get_stock_info tool with the relevant ticker symbol.
Summarize the result clearly for the user.
"""
```

ההנחיה הזאת אומרת לסוכן כמה דברים חשובים.

ראשית, הוא צריך להיות עוזר מועיל:

```bash
You are a helpful assistant.
```

זו הגדרת התנהגות כללית. היא לא קשורה דווקא למניות, אלא לאופי התשובה.

לאחר מכן מגיע החלק החשוב יותר:

```bash
When the user asks about a stock price, quote, or market data,
use the get_stock_info tool with the relevant ticker symbol.
```

כאן אנחנו מגדירים לסוכן כלל פעולה:

כאשר המשתמש שואל על מחיר מניה, quote או נתוני שוק, אל תענה מהזיכרון הפנימי של המודל. השתמש בכלי.

זה חשוב במיוחד בגלל שמחירי מניות הם מידע משתנה. מודל שפה לא יודע בהכרח מה המחיר הנוכחי של מניה, וגם אם הוא “זוכר” מחיר כלשהו, ייתכן שהוא כבר לא נכון.

לדוגמה, אם המשתמש שואל:

```bash
What is the current price of NVDA?
```

ה-Agent צריך להבין שזו לא שאלה כללית. זו שאלה שדורשת נתון עדכני. לכן עליו להפעיל את get_stock_info.

לעומת זאת, אם המשתמש שואל:

```bash
What is a stock?
```

לא בהכרח צריך להפעיל את הכלי. זו שאלה מושגית שאפשר לענות עליה בלי להביא נתוני שוק בזמן אמת.

החלק האחרון בהנחיה הוא:

```bash
Summarize the result clearly for the user.
```

ה-tool מחזיר נתונים, אבל המשתמש לא בהכרח רוצה לראות פלט טכני. הוא רוצה תשובה ברורה. לכן ה-Agent צריך לקחת את תוצאת הכלי ולנסח אותה בצורה מובנת.

אפשר לחשוב על ההנחיות כעל מדיניות עבודה:

```python
If the user asks for live market data:
    use the stock tool

If the user asks a general finance question:
    answer normally

After using the tool:
    explain the result clearly
```

זו נקודה חשובה בבניית Agents. הכלים נותנים יכולת, אבל ההנחיות קובעות את ההתנהגות. בלי הנחיות טובות, המודל עלול להשתמש בכלי כשלא צריך, או לא להשתמש בו דווקא כשכן צריך.

במערכת אמיתית, כדאי להרחיב את ההנחיות אפילו יותר. למשל, אפשר להוסיף שה-Agent לא נותן ייעוץ השקעות, לא ממליץ לקנות או למכור מניה, ומציג נתונים בלבד. כך הופכים את השימוש בכלי לבטוח וברור יותר.

## יצירת Agent

אחרי שיש לנו LLM, tool והנחיות, אפשר לחבר אותם יחד ל-Agent.

החיבור העקרוני נראה כך:

```bash
LLM + Tools + Instructions = Agent
```

כל אחד מהרכיבים האלה אחראי על חלק אחר במערכת.

ה-LLM אחראי להבין את הבקשה של המשתמש, להחליט מה צריך לעשות, ולנסח תשובה.

ה-Tools נותנים לסוכן יכולת לבצע פעולות מחוץ למודל. במקרה שלנו, הכלי מאפשר להביא נתוני מניה דרך yfinance.

ה-Instructions מגדירות לסוכן מתי להשתמש בכלי ואיך להחזיר את התשובה למשתמש.

ב-stock_agent.py יצירת ה-Agent מתבצעת בתוך הפונקציה:

```python
def build_agent():
```

בתוך הפונקציה קודם בודקים שקיים API key:

```python
if not os.getenv("ANTHROPIC_API_KEY"):
    raise EnvironmentError("ANTHROPIC_API_KEY is not set.")
```

הבדיקה הזאת חשובה כי בלי מפתח API אין אפשרות לקרוא למודל. עדיף לעצור את הריצה עם הודעה ברורה, במקום לקבל שגיאה לא מובנת בשלב מאוחר יותר.

לאחר מכן מגדירים את המודל:

```python
model = init_chat_model(
    "anthropic:claude-haiku-4-5-20251001",
    temperature=0,
)
```

הערך temperature=0 אומר למודל להיות יותר יציב ופחות יצירתי. במקרה של Agent שמפעיל כלי ומחזיר נתוני מניה, זו בחירה נכונה. אנחנו לא רוצים תשובה דמיונית או ניסוח חופשי מדי. אנחנו רוצים התנהגות עקבית.

לאחר מכן יוצרים את ה-Agent:

```python
agent = create_agent(
    model=model,
    tools=[get_stock_info],
    system_prompt=SYSTEM_PROMPT,
)
```

כאן מתבצע החיבור המרכזי.

המודל מקבל את היכולת לחשוב ולנסח.

רשימת הכלים נותנת לו פעולות שהוא יכול להפעיל.

ה-system prompt מגדיר את כללי ההתנהגות.

שימו לב שה-tools מועברים כרשימה:

```python
tools=[get_stock_info]
```

גם אם יש כרגע רק כלי אחד, המבנה הוא רשימה כי Agent יכול לעבוד עם כמה כלים. בהמשך אפשר להוסיף כלים נוספים, למשל כלי שמביא חדשות, כלי שמחזיר המלצות אנליסטים, או כלי שבודק שינוי במחיר לאורך זמן.

בשלב הזה כבר לא מדובר רק בצ’אטבוט. יש לנו מערכת שיכולה לקבל שאלה, להבין שהיא דורשת פעולה חיצונית, לבחור כלי מתאים, להפעיל אותו, ולעבד את התוצאה.

זה בדיוק הרעיון המרכזי של Agent עם Tools:

המודל לא מחזיק את כל המידע בתוכו, אלא יודע מתי להשתמש ביכולת חיצונית כדי לענות טוב יותר.

## הרצת שאלה

אחרי שה-Agent נבנה, אפשר להריץ עליו שאלה אמיתית.

המשתמש לא צריך לקרוא ישירות לפונקציה get_stock_info. הוא פשוט כותב שאלה בשפה טבעית, וה-Agent מחליט אם צריך להפעיל את הכלי.

לדוגמה:

```bash
What is the current price of MSFT?
```

הזרימה הפנימית נראית כך:

User question

↓

Agent

↓

Tool call

↓

Tool result

↓

Final answer

כלומר, המשתמש רואה שיחה פשוטה, אבל מאחורי הקלעים מתרחשים כמה שלבים.

ב-stock_agent.py יש פונקציה שמקבלת את ה-Agent ואת שאלת המשתמש:

```python
def query_agent(agent, user_input: str) -> str:
```

בתוך הפונקציה מפעילים את ה-Agent כך:

```python
result = agent.invoke(
    {"messages": [{"role": "user", "content": user_input}]}
)
```

הקלט שנשלח ל-Agent הוא הודעת משתמש רגילה. ה-Agent מקבל את ההודעה, קורא את ההנחיות, בודק אילו כלים זמינים לו, ומחליט איך לענות.

אם השאלה קשורה למחיר מניה או לנתוני שוק, הוא אמור להפעיל את הכלי get_stock_info.

לדוגמה, אם המשתמש שואל:

```bash
Give me the quote for NVDA.
```

ה-Agent אמור להבין ש-NVDA הוא ticker symbol, להפעיל את הכלי, לקבל ממנו נתונים, ואז להחזיר תשובה מסודרת.

אפשר לדמיין את זה כך:

```bash
User:
Give me the quote for NVDA.

Agent decision:
This requires market data.

Tool call:
get_stock_info("NVDA")

Tool result:
Company name, price, currency, day range, volume.
Final answer:
Here is the current market data for NVIDIA...
```

לאחר שה-Agent מסיים את העבודה, הקוד מחזיר את ההודעה האחרונה שנוצרה:

```python
return result["messages"][-1].content
```

ברוב המקרים, זו תהיה התשובה הסופית למשתמש.

ב-command-line, השאלה מתקבלת בלולאה רגילה:

```python
user_input = input("\nYou: ").strip()
```

אם המשתמש כתב quit או exit, התוכנית נעצרת. אחרת, היא שולחת את הקלט ל-Agent ומדפיסה את התשובה:

```python
answer = query_agent(agent, user_input)
print(f"\nAssistant: {answer}")
```

מבחינת המשתמש, זו נראית כמו שיחת צ’אט רגילה. אבל מבחינת המערכת, זה כבר Agent אמיתי: הוא לא רק מחזיר טקסט, אלא יכול להפעיל פעולה חיצונית בזמן הריצה.

זו הנקודה שבה רואים את ההבדל בין צ’אטבוט לבין Agent עם tool.

צ’אטבוט רגיל מנסה לענות.

Agent יכול לבדוק, להביא מידע, ואז לענות.

## טיפול בשגיאות

כאשר בונים Agent שמפעיל כלי חיצוני, חשוב לזכור שהכלי לא תמיד יחזיר תשובה תקינה. בניגוד לשיחה רגילה עם מודל, כאן יש תלות במקור מידע חיצוני, בקלט של המשתמש, בזמינות הרשת, ובאיכות הנתונים שמוחזרים מה-API.

לכן tool טוב חייב לדעת לטפל בשגיאות.

המקרה הפשוט ביותר הוא ticker ריק. למשל, אם המשתמש שואל:

```bash
What is the current stock price?
```

אבל לא מציין על איזו מניה הוא מדבר, אין לכלי סימול שאפשר לשלוח ל-yfinance.

במקרה כזה עדיף להחזיר הודעה ברורה:

```bash
Please provide a stock ticker symbol.
```

וזה בדיוק הרעיון בבדיקה הבסיסית:

```python
symbol = symbol.strip().upper()
if not symbol:
    return "Error: Please provide a stock ticker symbol."
```

טעות נוספת היא ticker שלא קיים. לדוגמה:

```bash
Give me the price of ABCXYZ123.
```

יכול להיות ש-yfinance לא יחזיר נתונים שימושיים עבור הסימול הזה. במצב כזה הכלי לא צריך להחזיר תשובה שנראית תקינה. הוא צריך להסביר שלא נמצאו נתונים מתאימים.

אפשר לנסח הודעה בסגנון:

```bash
I could not find market data for this ticker symbol.
```

מקרה נוסף הוא כשל בקריאה ל-API. גם אם הקוד נכון, השירות החיצוני עלול להיות לא זמין, איטי, או להחזיר שגיאה זמנית. לכן כדאי לעטוף את הקריאה ב-try/except.

לדוגמה:

```python
try:
    ticker = yf.Ticker(symbol)
    info = ticker.info
except Exception as e:
    return f"Error: Could not fetch stock data for {symbol}."
```

במערכת לימודית אפשר להחזיר הודעה קצרה. במערכת אמיתית, עדיף גם לכתוב את השגיאה ללוג פנימי, כדי שמפתח יוכל לבדוק מה קרה בלי לחשוף למשתמש פרטים טכניים מיותרים.

צריך גם לטפל במצב שבו חלק מהנתונים חסרים. למשל, ייתכן שיש שם חברה אבל אין מחיר נוכחי, או שיש מחיר אבל אין נפח מסחר. לכן הקוד לא צריך להניח שכל שדה תמיד קיים.

במקום לגשת ישירות לשדה כך:

```python
price = info["currentPrice"]
```

עדיף להשתמש ב-get:

```python
price = info.get("currentPrice") or info.get("regularMarketPrice") or info.get("previousClose")
```

כך הכלי מנסה למצוא ערך חלופי אם השדה הראשון חסר.

עוד מקרה שחשוב לחשוב עליו הוא timeout. כאשר כלי תלוי באינטרנט או בשירות חיצוני, ייתכן שהתגובה תתעכב. אם אין טיפול בזה, המשתמש עלול לחכות בלי להבין מה קורה. במערכת אמיתית כדאי להגדיר timeout, להחזיר הודעה ברורה, ולאפשר ניסיון חוזר.

העיקרון החשוב הוא שה-Agent לא אמור לקרוס בגלל tool שנכשל. אם הכלי לא הצליח להביא מידע, הסוכן צריך לקבל תוצאה ברורה ולהסביר למשתמש מה קרה.

אפשר לחשוב על זה כך:

```bash
Bad tool behavior:
crash / unclear technical error / empty answer

Good tool behavior:
clear error message → Agent explains the issue → user knows what to fix
```

טיפול טוב בשגיאות הופך את ה-Agent למערכת אמינה יותר. לא בגלל שהוא תמיד מצליח, אלא בגלל שגם כשהוא נכשל, הוא נכשל בצורה מובנת ומבוקרת.

## ההבדל בין RAG Chatbot לבין Stock Agent

אחרי שבנינו גם RAG Chatbot וגם Stock Agent, חשוב להבין את ההבדל ביניהם. בשני המקרים יש לנו מערכת שמקבלת שאלה מהמשתמש ומחזירה תשובה, אבל הדרך שבה התשובה נוצרת שונה מאוד.

RAG Chatbot נועד לענות על בסיס מסמכים קיימים. הוא לא מחפש מידע חדש באינטרנט ולא מפעיל פעולה חיצונית בזמן אמת. במקום זה, הוא מקבל שאלה, מחפש ב-Vector Store את ה-chunks הכי רלוונטיים, מכניס אותם ל-context, ואז שולח אותם ל-LLM כדי שינסח תשובה.

הזרימה שלו נראית כך:

User question 
 ↓ 
Vector Store 
 ↓ 
Relevant chunks 
 ↓ 
Context 
 ↓ 
LLM 
 ↓ 
Answer

לעומת זאת, Stock Agent נועד להפעיל כלי חיצוני. כאשר המשתמש שואל על מחיר מניה או נתוני שוק, המידע לא אמור להגיע ממסמך ישן או מהזיכרון של המודל. זה מידע משתנה, ולכן צריך להביא אותו בזמן אמת.

הזרימה שלו נראית כך:

User question 
 ↓ 
Agent decides which tool to use 
 ↓ 
Tool call 
 ↓ 
External data source 
 ↓ 
Tool result 
 ↓ 
LLM 
 ↓ 
Final answer

במילים פשוטות, RAG Chatbot שולף מידע מתוך מאגר ידע שנבנה מראש. Stock Agent מפעיל פעולה בזמן אמת כדי להביא מידע מבחוץ.

ההשוואה ביניהם נראית כך:

<div dir="rtl">

| **מערכת** | **מקור המידע** | **הפעולה המרכזית** |
| --- | --- | --- |
| **RAG Chatbot** | מסמכים | שליפה לפי משמעות |
| **Stock Agent** | API חיצוני | הפעלת כלי בזמן אמת |

</div>

לדוגמה, אם המשתמש שואל:

```bash
What does the document say about ChromaDB?
```

זו שאלה שמתאימה ל-RAG Chatbot. התשובה אמורה להגיע מתוך מסמך שנשמר במאגר.

אבל אם המשתמש שואל:

```bash
What is the current price of MSFT?
```

זו שאלה שמתאימה ל-Stock Agent. התשובה דורשת מידע עדכני, ולכן הסוכן צריך להפעיל כלי שמביא נתוני שוק.

ההבדל הזה חשוב מאוד בתכנון מערכות AI. אם מקור הידע הוא מסמכים, נהלים, מדריכים, קבצי תמיכה או מאגר ידע פנימי, RAG הוא פתרון טבעי. אם צריך להביא נתון חי, לקרוא ל-API, לבדוק סטטוס, לבצע פעולה או לעבוד מול מערכת חיצונית, צריך לחשוב בכיוון של Agent עם tools.

אפשר גם לשלב בין השניים. למשל, Agent יכול קודם להשתמש ב-RAG כדי למצוא נוהל מתאים, ואז להפעיל tool כדי לבדוק נתון במערכת חיצונית.

לדוגמה:

User asks a support question 
 ↓ 
Agent searches internal documentation 
 ↓ 
Agent finds the relevant procedure 
 ↓ 
Agent calls an external system 
 ↓ 
Agent returns a grounded answer

כך RAG נותן לסוכן זיכרון חיצוני, ו-Tools נותנים לו יכולת פעולה.

כאשר משלבים אותם נכון, מקבלים מערכת חזקה יותר: מערכת שלא רק יודעת לענות לפי ידע קיים, אלא גם יודעת לבדוק מידע, לבצע פעולות, ולעבוד מול מקורות חיצוניים בזמן אמת.
