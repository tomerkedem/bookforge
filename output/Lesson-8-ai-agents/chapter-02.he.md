# איך Agent בנוי

לפני שנכנסים לרכיבים בנפרד, כדאי לראות את התמונה הכללית. Agent הוא לא רכיב יחיד, אלא מערכת שמחברת כמה חלקים: בקשת משתמש, מודל שפה, זיכרון, כלים, הנחיות ותוצאה.

## מבט כללי

אפשר לחשוב על Agent כעל שכבת ניהול שיושבת במרכז. המשתמש לא מדבר ישירות עם כל הכלים ולא בוחר בעצמו מה צריך לקרות. הוא שולח בקשה, וה-Agent מחליט איך לטפל בה.

הזרימה הבסיסית נראית כך:

```bash
User Request
     ↓
AI Agent
  ↔ LLM
  ↔ Memory
  ↔ Tools
  ↑
Instructions
     ↓
Answer or Action
```

המשתמש מתחיל את התהליך עם בקשה. זו יכולה להיות שאלה פשוטה, משימה פתוחה, בקשה לניתוח מסמך, או פעולה שצריך לבצע.

ה-Agent מקבל את הבקשה ובודק מה נדרש כדי לטפל בה. לפעמים מספיק לשלוח את הבקשה ל-LLM ולקבל תשובה. לפעמים צריך לצרף מידע מהזיכרון. לפעמים צריך להפעיל כלי חיצוני. לפעמים צריך לשלב כמה צעדים.

ה-LLM משמש כמנוע ההבנה וההסקה. הוא עוזר להבין את הבקשה, לנסח תשובה, ולעיתים גם לבחור את הצעד הבא.

ה-Memory מאפשר למערכת להשתמש בהקשר קודם או במידע חיצוני. זה יכול להיות היסטוריית שיחה, מסמכים, בסיס נתונים או vector store.

ה-Tools מאפשרים למערכת לפעול מחוץ לטקסט: להביא מידע עדכני, לקרוא ל-API, לבצע חיפוש, להריץ קוד או לבצע פעולה במערכת אחרת.

ה-Instructions מגדירות את כללי ההתנהגות: איך לענות, מתי להשתמש בכלים, מתי לא לנחש, ומה לעשות כשאין מספיק מידע.

הדבר החשוב בתרשים הוא להבין שה-LLM אינו כל המערכת. הוא רכיב חשוב מאוד, אבל ה-Agent הוא זה שמארגן את העבודה סביבו. הוא מחליט איזה מידע לשלוח, מתי להיעזר בזיכרון, מתי להפעיל כלי, ואיך להחזיר למשתמש תוצאה ברורה.

## תפקיד ה-Agent

ה-Agent הוא שכבת הניהול של המערכת. הוא לא מחליף את ה-LLM, אלא משתמש בו כחלק מתהליך רחב יותר.

כאשר משתמש שולח בקשה, ה-Agent צריך להחליט איך לטפל בה. לפעמים התשובה פשוטה ואפשר להעביר את הבקשה ישירות למודל. במקרים אחרים צריך לצרף הקשר, לשלוף מידע ממסמכים, להפעיל כלי, או לבצע כמה שלבים לפני שמחזירים תשובה.

אפשר לחשוב על ה-Agent כעל רכיב ששואל בכל פעם:

```bash
What does this request require?
```

הוא צריך לנהל כמה החלטות:

- מה לשלוח ל-LLM.

- איזה מידע נוסף לצרף.

- האם צריך להשתמש בזיכרון.

- האם צריך להפעיל כלי.

- איך לפרש את התוצאה שחזרה.

- האם אפשר להחזיר תשובה, או שצריך שלב נוסף.

לדוגמה, אם המשתמש שואל:

```bash
Explain what embeddings are.
```

ייתכן שאין צורך בכלי. ה-Agent יכול לשלוח את הבקשה ל-LLM ולקבל הסבר.

אבל אם המשתמש שואל:

```bash
What does our document say about embeddings?
```

כאן כבר צריך לבדוק מקור מידע חיצוני. ה-Agent צריך להבין שלא מדובר בשאלה כללית, אלא בשאלה על מסמך מסוים. במקרה כזה הוא יכול להפעיל תהליך RAG, לשלוף context רלוונטי, ורק אז לשלוח את השאלה למודל.

ואם המשתמש שואל:

```bash
What is the current price of Apple stock?
```

ה-Agent צריך להבין שמדובר בנתון עדכני, ולכן יש צורך בכלי חיצוני שמביא נתוני מניה.

התפקיד המרכזי של ה-Agent הוא לא “לדעת הכול”, אלא לנהל את הדרך לתשובה. הוא מתאם בין הבקשה, ה-LLM, הזיכרון, הכלים וההנחיות, כדי שהמערכת תחזיר תוצאה מבוססת יותר ולא רק תשובה שנשמעת טוב.

## תפקיד ה-LLM

ה-LLM הוא מנוע השפה וההסקה של ה-Agent. הוא הרכיב שעוזר למערכת להבין בקשות שכתובות בשפה טבעית, לזהות כוונה, לנסח תשובות, להסביר תוצאות, ולעיתים גם לבחור מה הצעד הבא.

אם המשתמש כותב:

```bash
Can you explain this in simpler words?
```

ה-LLM יודע להבין שלא מדובר בשאלה חדשה לגמרי, אלא בבקשה לשכתוב או הסבר פשוט יותר של משהו שכבר נאמר.

אם המשתמש כותב:

```bash
Find the most relevant parts in my documents and answer the question.
```

ה-LLM יכול לעזור להבין את השאלה ולנסח תשובה, אבל הוא לא אמור לשלוף את המסמכים בעצמו. לשם כך יש צורך ברכיב חיפוש או RAG שמחזיר לו context מתאים.

בתוך Agent, ה-LLM יכול להשתתף בכמה פעולות:

- להבין את בקשת המשתמש.

- לזהות אם חסר מידע.

- לבחור ניסוח מתאים לשאלה או לתשובה.

- לפרש תוצאה שחזרה מכלי.

- לעזור לבחור כלי מתאים כאשר קיימים כמה כלים.

- להסביר למשתמש את התוצאה בצורה ברורה.

חשוב לזכור שה-LLM אינו מקור אמת לכל דבר. הוא טוב מאוד בשפה ובהסקה, אבל כאשר צריך נתון מדויק, עדכני או פרטי, ה-Agent צריך לחבר אותו למקור מידע מתאים. אחרת המערכת עלולה להחזיר תשובה שנשמעת נכונה, אבל אינה מבוססת מספיק.

## תפקיד ה-Memory

Memory מאפשר ל-Agent לעבוד עם הקשר ולא להתחיל מאפס בכל בקשה. בלי זיכרון, כל שאלה עומדת בפני עצמה, והמערכת לא באמת יודעת מה נאמר קודם או איזה מידע כבר נאסף.

יש שני סוגים מרכזיים של זיכרון.

הראשון הוא זיכרון קצר טווח. זה המידע שנמצא בתוך השיחה הנוכחית או בתוך חלון ההקשר של המודל. לדוגמה, אם המשתמש אומר בתחילת השיחה:

```bash
I am building a RAG chatbot in Python.
```

ואחרי כמה הודעות שואל:

```bash
Can you show me a simple example?
```

ה-Agent יכול להבין שהדוגמה צריכה להיות קשורה ל-RAG Chatbot בפייתון, גם אם המשתמש לא חזר על כל הפרטים.

הסוג השני הוא זיכרון חיצוני. זה מידע שלא נמצא בהכרח בתוך השיחה, אלא נשמר או נשלף ממקור חיצוני:

- מסמכים

- קבצים

- מסד נתונים

- vector store

- היסטוריית פעולות

- מידע שנשמר על המשתמש או על המשימה

במערכת RAG, הזיכרון החיצוני מופיע בצורה ברורה: יש מסמכים, מחלקים אותם לקטעים, יוצרים להם embeddings, ושומרים אותם במאגר וקטורי. בזמן שאלה, ה-Agent יכול לשלוף את הקטעים הרלוונטיים ולהעביר אותם ל-LLM כחלק מה-context.

חשוב להבין ש-Memory אינו אומר שהמודל “זוכר הכול”. בדרך כלל, הזיכרון הוא מנגנון שהמערכת מנהלת סביב המודל. ה-Agent מחליט איזה מידע לשלוף, איזה מידע לצרף, ואיך להשתמש בו כדי לענות בצורה מדויקת יותר.



## תפקיד ה-Tools

Tools הם הדרך שבה Agent עושה משהו מעבר לכתיבת תשובה. הם מאפשרים למערכת להתחבר לעולם החיצוני: להביא מידע, להריץ פעולה, לבדוק נתונים, לקרוא ל-API או לעבוד מול מערכת אחרת.

לדוגמה, LLM יכול להבין את השאלה:

```bash
What is the current price of Apple stock?
```

אבל בלי כלי חיצוני הוא לא באמת יודע להביא את המחיר הנוכחי. בשביל זה צריך Tool שמתחבר למקור נתוני מניות ומחזיר תוצאה עדכנית.

Tool יכול להיות דבר ממש פשוט, כמו פונקציית חישוב:

```python
def calculate_total(price, quantity):
    return price * quantity
```

והוא יכול להיות גם כלי שמתחבר לשירות חיצוני:

```bash
get_stock_info("AAPL")
```

או כלי שמחפש במסמכים:

```bash
search_documents("What does the document say about ChromaDB?")
```

בתוך Agent, ה-Tools נותנים למערכת יכולת פעולה. במקום שהמודל ינחש, הוא יכול לבקש מהמערכת להפעיל כלי מתאים, לקבל תוצאה, ואז להשתמש ב-LLM כדי להסביר אותה למשתמש.

אבל Tools דורשים זהירות. כלי יכול להיכשל, להחזיר מידע חסר, להיות איטי, או לבצע פעולה רגישה. לכן חשוב להגדיר לכל כלי:

- מה הוא עושה.

- איזה קלט הוא מקבל.

- איזה פלט הוא מחזיר.

- מתי מותר להשתמש בו.

- איך מטפלים בשגיאה.

הכוח של Agent מגיע הרבה פעמים מהכלים שלו. ככל שהכלים מוגדרים טוב יותר, המערכת יכולה לפעול בצורה מדויקת ובטוחה יותר.



## תפקיד ה-Instructions

Instructions הן ההנחיות שמגדירות איך ה-Agent אמור להתנהג. הן לא הבקשה הנקודתית של המשתמש, אלא כללי העבודה של המערכת.

לדוגמה, המשתמש יכול לשאול:

```bash
What does this document say about ChromaDB?
```

אבל ה-Instructions יכולות להגדיר למערכת איך לענות:

```bash
Answer only based on the provided context.
If the context does not contain the answer, say that the information is not available.
Keep the answer concise.
```

ההנחיות עוזרות ל-Agent להבין גבולות חשובים:

- מתי להשתמש בכלים.

- מתי לענות לפי context בלבד.

- מתי לא לנחש.

- מתי לבקש הבהרה.

- איך לנסח תשובה.

- מה לעשות כשאין מספיק מידע.

- אילו פעולות דורשות אישור משתמש.

לדוגמה, במערכת שמחפשת במסמכים, Instruction חשוב יכול להיות:

```bash
If the retrieved context does not contain relevant information, do not invent an answer.
```

במערכת שמשתמשת בכלי מניות, Instruction יכול להיות:

```bash
Use the stock data tool only when the user asks about stock price, market data, or a company ticker.
```

חשוב להבין שהנחיות הן שכבת כיוון, לא מנגנון אבטחה מלא. המודל יכול לטעות, לפרש הוראה בצורה לא מדויקת, או לבחור פעולה לא מתאימה. לכן במערכת אמיתית לא מסתמכים רק על Instructions. משלבים אותן עם קוד, הרשאות, בדיקות, טיפול בשגיאות והגבלות ברורות על הכלים.

במילים פשוטות, Instructions אומרות ל-Agent איך להתנהג, אבל הארכיטקטורה היא זו שצריכה לוודא שההתנהגות נשארת מבוקרת.
