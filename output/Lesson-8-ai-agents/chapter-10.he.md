# בניית בסיס RAG

אחרי שהכרנו את הקבצים ואת סדר ההרצה, אפשר להתחיל לבנות את החלק הראשון של המערכת המעשית: בסיס ה-RAG.

המטרה של החלק הזה היא להבין איך טקסט רגיל הופך למאגר שאפשר לחפש בו לפי משמעות. זה השלב שמכין את הזיכרון החיצוני של המערכת. לפני שהצ’אטבוט יכול לענות לפי מסמכים, צריך קודם לטעון את המסמכים, לחלק אותם לחלקים קטנים, ליצור להם embeddings, ולשמור אותם בתוך vector store.

הקובץ המרכזי בחלק הזה הוא:

```bash
build_rag_db.py
```

## מה אנחנו בונים

אנחנו בונים מאגר ChromaDB שמכיל קטעים קטנים של מסמכים. כל קטע כזה נקרא chunk. לכל chunk יוצרים embedding, כלומר ייצוג מספרי של המשמעות שלו, ואז שומרים את הייצוגים האלה במאגר וקטורי.

הזרימה הכללית נראית כך:

```bash
Text files
   ↓
Load documents
   ↓
Split into chunks
   ↓
Create embeddings
   ↓
Store in ChromaDB
```

הרעיון הוא לא לשמור את הטקסט רק כטקסט רגיל, אלא להפוך אותו למשהו שאפשר לחפש בו בצורה סמנטית. כלומר, לא רק לפי התאמה מדויקת של מילים, אלא לפי משמעות.

לדוגמה, אם יש במסמך משפט על:

```bash
ChromaDB stores embeddings and supports semantic search.
```

והמשתמש שואל:

```bash
Where do we save the vectors?
```

חיפוש רגיל לפי מילים עלול לפספס את הקשר. אבל חיפוש סמנטי יכול להבין שהשאלה קשורה ל-vectors, embeddings ו-vector store.

בשלב הזה עדיין לא בונים צ’אטבוט. אנחנו רק מכינים את בסיס הידע. זה דומה להכנת אינדקס לפני שמתחילים לחפש. אם המאגר לא נבנה, ל-RAG Chatbot לא יהיה מאיפה לשלוף מידע.

בקובץ build_rag_db.py התהליך הזה מתבצע בצורה מסודרת: מוגדרת תיקיית שמירה ל-ChromaDB, מוגדר שם collection, מוגדר מודל embeddings, ואז נבנות פונקציות לטעינת מסמכים, חלוקה ל-chunks, בניית vector store וטעינת מאגר קיים.

## טעינת מסמכים

השלב הראשון בבניית בסיס RAG הוא טעינת המסמכים. לפני שאפשר לחלק טקסט ל-chunks, ליצור embeddings או לשמור מידע ב-ChromaDB, צריך להביא את הטקסט עצמו לתוך הקוד.

בקובץ build_rag_db.py קיימת פונקציה בשם:

```python
load_and_chunk_documents(data_dir: str = "data")
```

הפונקציה מקבלת שם של תיקייה, וברירת המחדל היא תיקייה בשם data. כלומר, הקוד מצפה למצוא קבצי טקסט בתוך תיקייה כזאת:

```bash
project_folder/
  data/
    file1.txt
    file2.txt
```

הקוד בודק קודם שהתיקייה קיימת. אם היא לא קיימת, הוא לא ממשיך בשקט ולא מחזיר תוצאה ריקה, אלא מעלה שגיאה ברורה:

```python
if not data_path.exists():
    raise FileNotFoundError(f"Data directory not found: {data_path}")
```

זו נקודה חשובה. במערכות לימודיות וגם במערכות אמיתיות, עדיף לקבל שגיאה ברורה מוקדם מאשר לגלות אחר כך שהצ’אטבוט לא מחזיר תשובות כי לא נטענו מסמכים בכלל.

אחרי שהקוד מוצא את התיקייה, הוא עובר על כל קבצי ה-.txt שבתוכה:

```python
for p in data_path.glob("*.txt"):
    loader = TextLoader(str(p), encoding="utf-8")
    docs.extend(loader.load())
```

כאן משתמשים ב-TextLoader, שתפקידו לקרוא קובץ טקסט ולהפוך אותו לאובייקט Document של LangChain. האובייקט הזה שומר את תוכן המסמך, ובדרך כלל יכול לשמור גם metadata כמו שם הקובץ או מקור הטקסט.

גם כאן יש בדיקה חשובה: אם התיקייה קיימת אבל אין בה קבצי טקסט, הקוד מעלה שגיאה:

```python
if not docs:
    raise ValueError(f"No .txt files found in {data_path}")
```

המשמעות היא שהקוד לא מניח שהכול תקין. הוא בודק את תנאי ההתחלה לפני שהוא ממשיך לשלב הבא. זה הרגל נכון מאוד בעבודה עם RAG, כי הרבה בעיות במערכת מתחילות כבר בשלב טעינת המסמכים: תיקייה לא נכונה, קבצים חסרים, encoding לא מתאים, או פורמט שלא נטען כמו שצריך.

בשלב הזה עדיין לא נוצר מאגר וקטורי. רק טענו את המסמכים והכנו אותם להמשך. השלב הבא יהיה לחלק את המסמכים ל-chunks קטנים יותר, כדי שאפשר יהיה לחפש בהם בצורה יעילה ומדויקת יותר.

## חלוקה ל-Chunks

אחרי שטענו את המסמכים, השלב הבא הוא לחלק אותם לחלקים קטנים יותר שנקראים chunks.

הסיבה פשוטה: בדרך כלל לא נרצה לשלוח מסמך שלם למודל בכל פעם שהמשתמש שואל שאלה. מסמך יכול להיות ארוך מדי, להכיל הרבה מידע לא רלוונטי, ולבזבז מקום יקר בתוך חלון ההקשר.

במקום זה, מחלקים את המסמך לקטעים קטנים. בזמן שאלה, המערכת מחפשת רק את הקטעים שהכי קרובים במשמעות לשאלה של המשתמש.

לדוגמה, אם יש מסמך ארוך על LangChain, ChromaDB ו-RAG, והמשתמש שואל:

```bash
What is ChromaDB used for?
```

אין צורך לשלוח למודל את כל המסמך. מספיק לשלוף את ה-chunks שמדברים על ChromaDB.

בקובץ build_rag_db.py החלוקה מתבצעת באמצעות RecursiveCharacterTextSplitter:

```python
splitter = RecursiveCharacterTextSplitter(
    chunk_size=400,
    chunk_overlap=80,
    length_function=len,
)
```

הפרמטר chunk_size=400 אומר שכל chunk יהיה בערך עד 400 תווים. זה לא אומר שכל chunk תמיד יהיה בדיוק באותו אורך, אלא שזה גודל היעד לחלוקה.

הפרמטר chunk_overlap=80 אומר שתהיה חפיפה של 80 תווים בין chunk אחד לבא אחריו. החפיפה חשובה כי לפעמים רעיון מתחיל בסוף chunk אחד וממשיך בתחילת chunk אחר. בלי overlap, המערכת עלולה לחתוך משפט או רעיון באמצע ולאבד הקשר.

אפשר לחשוב על זה כך:

```bash
Chunk 1: [.................]
Chunk 2:              [.................]
Chunk 3:                           [.................]
```

החפיפה עוזרת לשמור על רצף בין הקטעים.

אחרי שמגדירים את ה-splitter, הקוד מפעיל:

```python
return splitter.split_documents(docs)
```

בשלב הזה המסמכים המקוריים כבר לא נשארים כיחידות גדולות. הם הופכים לרשימה של chunks קטנים יותר, שכל אחד מהם יכול לעבור embedding ולהישמר במאגר הווקטורי.

זו אחת ההחלטות החשובות ביותר בבניית RAG. אם ה-chunks גדולים מדי, כל תוצאה תחזיר יותר מדי מידע ועלולה לבלבל את המודל. אם הם קטנים מדי, כל chunk עלול לא להכיל מספיק הקשר כדי לענות טוב. לכן chunk_size ו-chunk_overlap הם לא רק פרטים טכניים, אלא חלק מהאיכות של מערכת ה-RAG.

**יצירת Embeddings**

אחרי שהמסמכים חולקו ל-chunks, צריך להפוך כל chunk לייצוג מספרי. הייצוג הזה נקרא embedding.

Embedding הוא דרך לייצג משמעות של טקסט כמספרים. במקום שהמחשב יראה רק מילים, הוא מקבל וקטור שמייצג את המשמעות הכללית של המשפט או הקטע.

לדוגמה, שני משפטים יכולים להשתמש במילים שונות אבל לדבר על רעיון דומה:

```bash
ChromaDB stores embeddings for semantic search.
A vector database saves text representations and helps retrieve similar content.
```

בחיפוש רגיל לפי מילים, ייתכן שלא תהיה התאמה חזקה בין המשפטים. אבל בחיפוש סמנטי, embeddings יכולים לעזור לזהות שהם קשורים לאותו רעיון: שמירת ייצוגים וקטוריים וחיפוש לפי משמעות.

בקובץ build_rag_db.py מוגדר מודל embeddings כך:

```python
EMBEDDING_MODEL = "sentence-transformers/all-MiniLM-L6-v2"
```

ובהמשך נוצרת פונקציה שמחזירה את אובייקט ה-embeddings:

```python
def get_embeddings() -> HuggingFaceEmbeddings:
    return HuggingFaceEmbeddings(
        model_name=EMBEDDING_MODEL,
        model_kwargs={"device": "cpu"},
    )
```

המשמעות היא שהמערכת משתמשת במודל מקומי דרך HuggingFace כדי להפוך טקסטים לווקטורים. במקרה הזה ההרצה מוגדרת על CPU, ולכן היא לא דורשת GPU.

אחרי שלכל chunk יש embedding, אפשר לשמור אותו ב-vector store. בזמן שאלה, גם השאלה של המשתמש תהפוך ל-embedding, ואז המערכת תוכל למצוא את ה-chunks שהכי קרובים אליה במשמעות.

זה הבסיס של RAG: לא מחפשים רק מילים זהות, אלא קטעים שדומים במשמעות לשאלה של המשתמש.

## שמירה ב-Vector Store

אחרי שיצרנו embeddings לכל ה-chunks, צריך לשמור אותם במקום שמאפשר חיפוש מהיר ויעיל. המקום הזה נקרא Vector Store.

בפרויקט הזה נעשה שימוש ב-ChromaDB. זהו מאגר שמתמחה באחסון embeddings ובחיפוש לפי דמיון סמנטי. אפשר לחשוב עליו כמנוע חיפוש שמבין משמעות, ולא רק התאמה מדויקת של מילים.

בקובץ build_rag_db.py מוגדרים שני ערכים חשובים:

```python
CHROMA_PERSIST_DIR = Path(__file__).parent / "chroma_db"
COLLECTION_NAME = "rag_docs"
```

COLLECTION_NAME הוא השם של אוסף המסמכים שנשמר במאגר. אפשר לחשוב על collection כמו אוסף מסודר של chunks ו-embeddings ששייכים לאותו תחום ידע.

לדוגמה:

```bash
rag_docs
company_policies
technical_docs
support_articles
```

CHROMA_PERSIST_DIR הוא הנתיב שבו ChromaDB ישמור את המאגר על הדיסק. במקרה הזה, המאגר יישמר בתיקייה בשם chroma_db.

הקוד שבונה את המאגר נראה כך:

```python
return Chroma.from_documents(
    documents=chunks,
    embedding=get_embeddings(),
    persist_directory=str(CHROMA_PERSIST_DIR),
    collection_name=COLLECTION_NAME,
)
```

בשלב הזה קורים כמה דברים יחד: ה-chunks נכנסים למאגר, לכל chunk נוצר או משויך embedding, כל המידע נשמר תחת collection מסודר, והמאגר נכתב לתיקייה קבועה בדיסק.

למה חשוב לשמור את המאגר לדיסק?

אם לא נשמור את המאגר, בכל הרצה נצטרך לבצע מחדש את כל התהליך:

```bash
Load documents → Split into chunks → Create embeddings → Build vector store
```

כאשר עובדים עם מעט מסמכים זה אולי לא נורא. אבל בפרויקט אמיתי, עם הרבה מסמכים, יצירת embeddings מחדש בכל הרצה יכולה להיות איטית ומיותרת.

כאשר שומרים את המאגר לדיסק, אפשר לבנות אותו פעם אחת בלבד, ולאחר מכן פשוט לטעון אותו מחדש.

<img src="/Lesson-8-ai-agents/assets/image-06.png" alt="image-06.png" width="710" height="222" />


בשלב הזה בסיס ה-RAG כבר קיים. המסמכים הוטענו, חולקו ל-chunks, הומרו ל-embeddings, ונשמרו במאגר שאפשר לחפש בו לפי משמעות. השלב הבא יהיה להבין איך טוענים את המאגר הקיים ומשתמשים בו מתוך הצ’אטבוט.

## טעינת Vector Store קיים

אחרי שבנינו את המאגר ושמרנו אותו לדיסק, לא נרצה לבנות אותו מחדש בכל פעם שהצ’אטבוט עולה. במקום זה, נרצה לטעון את ה-Vector Store הקיים ולהשתמש בו לחיפוש.

ב-build_rag_db.py יש פונקציה נפרדת לטעינה:

```python
def load_vectorstore() -> Chroma:
    """Load the persisted ChromaDB vector store."""
    if not CHROMA_PERSIST_DIR.exists():
        raise FileNotFoundError(
            f"Vector store not found at {CHROMA_PERSIST_DIR}. Run build_rag_db.py first."
        )
    return Chroma(
        persist_directory=str(CHROMA_PERSIST_DIR),
        embedding_function=get_embeddings(),
        collection_name=COLLECTION_NAME,
    )
```

הפונקציה הזאת עושה דבר חשוב: היא קודם בודקת אם תיקיית המאגר קיימת. אם לא, היא מחזירה שגיאה ברורה שמסבירה שצריך להריץ קודם את build_rag_db.py.

זו בדיקה חשובה, כי בלי המאגר הצ’אטבוט לא יכול לבצע retrieval. אין לו מאיפה לשלוף chunks רלוונטיים.

אם המאגר כן קיים, הפונקציה יוצרת אובייקט Chroma שמצביע על התיקייה שכבר נשמרה בדיסק:

```python
persist_directory=str(CHROMA_PERSIST_DIR)
```

בנוסף, היא משתמשת באותו מודל embeddings:

```python
embedding_function=get_embeddings()
```

זה חשוב מאוד. כאשר מחפשים במסד וקטורי, השאלה של המשתמש צריכה לעבור embedding באותה שיטה שבה נוצרו ה-embeddings של המסמכים. אם משתמשים במודל embeddings אחר בזמן החיפוש, התוצאות עלולות להיות פחות טובות או לא עקביות.

הפונקציה הזאת מפרידה בין שני מצבים:

Build vector store once 
 ↓ 
Load vector store many times

הבנייה מתבצעת כאשר מוסיפים או מעדכנים מסמכים. הטעינה מתבצעת בכל פעם שהצ’אטבוט צריך להשתמש במאגר הקיים.

זו הפרדה נכונה בתכנון מערכת RAG. שלב הבנייה הוא שלב הכנה. שלב הטעינה הוא חלק מהריצה השוטפת של האפליקציה.

## טעויות נפוצות

בבניית בסיס RAG יש כמה טעויות שחוזרות הרבה. רובן לא קשורות ל-LLM עצמו, אלא להכנת המאגר.

טעות ראשונה היא לשכוח ליצור את תיקיית data. הקוד מצפה למצוא בתוכה קבצי טקסט. אם התיקייה לא קיימת, או אם היא נמצאת במקום לא נכון, שלב טעינת המסמכים ייכשל.

```bash
project_folder/
  data/
    sample_docs.txt
```

טעות שנייה היא להריץ את הצ’אטבוט לפני שבונים את המאגר. rag_chatbot.py מצפה למצוא Vector Store קיים בתיקיית chroma_db. אם לא מריצים קודם את build_rag_db.py, אין מאגר לטעון ואין מאיפה לשלוף מידע.

הסדר הנכון הוא:

```bash
python build_rag_db.py
python rag_chatbot.py
```

טעות שלישית היא לבחור chunks גדולים מדי. אם כל chunk מכיל יותר מדי מידע, ה-retriever עלול להחזיר קטעים עמוסים ולא ממוקדים. במקרה כזה ה-LLM מקבל context רחב מדי, וקשה לו להבין מה באמת רלוונטי לשאלה.

גם chunks קטנים מדי יכולים להיות בעיה. אם חותכים את הטקסט לחלקים קצרים מדי, כל chunk עלול לאבד הקשר. למשל, תשובה שדורשת משפט לפני ומשפט אחרי עלולה להיחתך לשני חלקים נפרדים.

טעות נוספת היא לחשוב ש-Vector Store הוא מודל. הוא לא. ה-Vector Store לא “מבין” כמו LLM ולא מנסח תשובות. הוא מאגר חיפוש שמחזיר קטעים דומים במשמעות לשאלה. את התשובה הסופית עדיין מנסח ה-LLM על בסיס ה-context שנשלף.

חשוב גם לזכור שאם המסמכים משתנים, המאגר לא תמיד מתעדכן לבד. אם מוסיפים קבצים חדשים או משנים תוכן קיים, צריך לבנות מחדש את ה-Vector Store או להוסיף מנגנון עדכון מסודר.

במילים פשוטות, איכות ה-RAG מתחילה הרבה לפני השאלה של המשתמש. היא מתחילה באיכות המסמכים, בגודל ה-chunks, במודל ה-embeddings, ובדרך שבה שומרים וטוענים את המאגר.
