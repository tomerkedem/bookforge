# Workflow מובנה מול Agent Loop

אחרי שהבנו את רכיבי הבסיס של Agent, אפשר לעבור לשאלה חשובה יותר: איך המערכת מחליטה מה לעשות בפועל?

יש מערכות שבהן סדר הפעולות ידוע מראש. הן עוברות שלב אחרי שלב, בצורה מסודרת וצפויה. לעומתן, יש מערכות שבהן הסוכן מחליט בזמן ריצה מה הצעד הבא, לפי המצב, התשובה שקיבל או הכלי שהופעל.

ההבדל הזה משפיע מאוד על תכנון המערכת, על רמת הגמישות שלה, על קלות הבדיקה, ועל רמת הסיכון.

## מהו Workflow מובנה

Workflow מובנה הוא תהליך שבו השלבים ידועים מראש. המערכת לא צריכה “להחליט מחדש” בכל רגע מה לעשות, כי המסלול כבר מוגדר.

לדוגמה, תהליך פשוט לסיכום מסמך יכול להיראות כך:

```bash
Load document
   ↓
Split into sections
   ↓
Summarize each section
   ↓
Combine summaries
   ↓
Return final summary
```

במקרה כזה, גם אם משתמשים ב-LLM בכל אחד מהשלבים, עדיין מדובר בתהליך מובנה. המערכת יודעת מה השלב הראשון, מה השלב הבא, ומתי התהליך מסתיים.

Workflow כזה מתאים במיוחד כאשר המשימה ברורה. למשל:

- סיכום מסמך.

- בדיקת איכות של תשובה.

- יצירת מייל לפי תבנית.

- תרגום טקסט.

- הפקת דוח בשלבים.

- בניית תשובה לפי context שנשלף מראש.

היתרון הגדול של Workflow מובנה הוא צפיות. קל יותר להבין מה המערכת עושה, לבדוק כל שלב בנפרד, לאתר תקלה, ולשפר חלק מסוים בלי לשנות את כל המערכת.

לדוגמה, אם שלב הסיכום עובד טוב אבל שלב האיחוד חלש, אפשר לשפר רק את שלב האיחוד. אין צורך לפרק את כל התהליך.

החיסרון הוא ש-Workflow מובנה פחות גמיש. אם המשימה משתנה בזמן אמת, אם חסר מידע, או אם צריך לבחור בין כמה דרכי פעולה, תהליך קבוע מראש עלול להיות מוגבל מדי.

לכן Workflow מובנה הוא בחירה טובה כאשר אנחנו יודעים מראש מה צריך לקרות. הוא פחות מתאים כאשר המערכת צריכה לחקור, לבדוק כמה כיוונים, להפעיל כלים לפי מצב, או להחליט בעצמה מה הצעד הבא.

## מתי Workflow מתאים

Workflow מתאים כאשר אנחנו יודעים מראש מה סדר הפעולות הרצוי. במצב כזה אין סיבה לתת לסוכן חופש מלא להחליט בכל רגע מה לעשות. עדיף להגדיר תהליך ברור, לבדוק אותו היטב, ולוודא שכל שלב עושה בדיוק את מה שהוא אמור לעשות.

לדוגמה, אם אנחנו בונים תהליך שמייצר מסמך מקצועי, אפשר להגדיר שלבים קבועים:

```bash
Create outline
   ↓
Write draft
   ↓
Review quality
   ↓
Improve wording
   ↓
Return final document
```

כאן אין צורך שהמערכת “תחקור” בכל פעם מה לעשות. השלבים ידועים מראש, וה-LLM יכול לעזור בתוך כל שלב: לבנות ראשי פרקים, לכתוב טיוטה, לבדוק ניסוח או לשפר בהירות.

Workflow מתאים במיוחד לתהליכים עסקיים ברורים. למשל, מערכת שמקבלת פנייה מלקוח, בודקת סוג פנייה, מחפשת מידע רלוונטי, מנסחת תשובה, ואז שולחת אותה לאישור. אם הארגון יודע מה התהליך הנכון, עדיף לא להפוך אותו ל-Agent חופשי מדי.

הוא מתאים גם לבדיקות איכות. למשל, אפשר לבנות תהליך שבו תשובה נוצרת בשלב ראשון, ואז עוברת בדיקה בשלב שני:

```bash
Generate answer
   ↓
Check if answer is based on context
   ↓
Fix if needed
   ↓
Return final answer
```

זה שימושי מאוד במערכות RAG, כי לא מספיק לקבל תשובה יפה. צריך לבדוק שהיא באמת מבוססת על המידע שנשלף.

היתרון הגדול של Workflow הוא יציבות. קל יותר להבין אותו, להסביר אותו, לבדוק אותו ולשפר אותו. אם משהו נכשל, אפשר לזהות באיזה שלב הבעיה קרתה.

לכן, כאשר המשימה ברורה, חוזרת על עצמה, וצריכה להיות צפויה, Workflow הוא בדרך כלל הבחירה הראשונה. לא צריך להתחיל מ-Agent Loop אם תהליך פשוט ומוגדר מראש פותר את הבעיה בצורה טובה יותר.

## מהו Agent Loop

Agent Loop הוא תהליך דינמי שבו הסוכן לא מקבל מראש רשימה סגורה של שלבים, אלא מחליט בזמן ריצה מה לעשות בהמשך.

המחזור הבסיסי נראה כך:

```bash
observe → think → act → observe → think → act
```

כלומר, הסוכן קודם מתבונן במצב: מה המשתמש ביקש, איזה מידע כבר קיים, מה חסר, ומה קרה בשלב הקודם.

אחר כך הוא “חושב” מה כדאי לעשות: האם לענות, האם לחפש מידע, האם להפעיל כלי, האם לשאול שאלת הבהרה, או האם לעצור.

לאחר מכן הוא פועל: מפעיל כלי, שולף מידע, מבצע פעולה או מחזיר תשובה.

אחרי הפעולה הוא שוב מתבונן בתוצאה, ומחליט אם צריך עוד שלב.

לדוגמה, אם המשתמש מבקש:

```bash
Find out whether this company is worth following and explain why.
```

לא תמיד יודעים מראש כמה שלבים יידרשו. הסוכן יכול להתחיל בחיפוש מידע, לראות שחסר נתון פיננסי, להפעיל כלי נוסף, לבדוק חדשות, להשוות מקורות, ואז להחליט שיש מספיק מידע כדי לענות.

זה היתרון של Agent Loop: הוא גמיש. הוא מתאים למצבים שבהם המשימה פתוחה, משתנה, או לא מוגדרת עד הסוף.

אבל יש לזה מחיר. ככל שהסוכן מחליט יותר לבד, קשה יותר לצפות את כל הצעדים מראש. לכן Agent Loop דורש גבולות ברורים: אילו כלים מותר להפעיל, כמה צעדים מותר לבצע, מתי לעצור, ומתי לבקש אישור מהמשתמש.



## מתי Agent Loop מתאים

Agent Loop מתאים כאשר אי אפשר לדעת מראש את כל השלבים. אם המשימה פתוחה, משתנה, או תלויה בתוצאות שמתקבלות תוך כדי עבודה, תהליך קבוע מראש עלול להיות מוגבל מדי.

לדוגמה, בקשה כזאת אינה תמיד מתאימה ל-Workflow פשוט:

```bash
Research this topic and tell me what I should pay attention to.
```

כאן לא ברור מראש כמה מקורות צריך לבדוק, אילו שאלות יתעוררו תוך כדי, או האם המידע הראשון שיימצא יספיק. הסוכן צריך לפעול בשלבים, לבדוק תוצאות, ולהחליט מה הצעד הבא.

Agent Loop מתאים במיוחד למשימות כמו:

- חיפוש מידע דינמי.

- בדיקת כמה מקורות.

- פתרון בעיות שלא ידועות מראש.

- ניתוח מצב שבו חסרים נתונים.

- שימוש בכמה כלים לפי הצורך.

- תהליך שבו לא יודעים מראש כמה צעדים יידרשו.

לדוגמה, משתמש יכול לבקש:

```bash
Check why this RAG chatbot gives weak answers.
```

הסוכן לא יודע מראש מה הבעיה. ייתכן שהבעיה היא ב-chunk size, ב-embeddings, בכמות ה-context, ב-prompt, ב-retriever, או במסמכים עצמם. הוא צריך לבדוק שלב אחר שלב, לראות מה התוצאה, ואז להחליט לאן להמשיך.

זה בדיוק המקום שבו Agent Loop יכול לעזור. הוא מאפשר למערכת להתקדם לפי מה שהיא מגלה תוך כדי עבודה.

אבל גם כאן צריך להיזהר. Agent Loop לא אומר שהסוכן עושה מה שהוא רוצה בלי גבולות. צריך להגדיר מגבלות ברורות: מספר צעדים מקסימלי, כלים מותרים, מתי לעצור, מתי לבקש אישור, ואיך לדווח למשתמש מה נעשה.

Workflow מובנה מתאים כאשר אנחנו יודעים מה צריך לקרות. למשל, אם המערכת צריכה לקבל מסמך, לסכם אותו, לבדוק את איכות הסיכום ולהחזיר תשובה, אפשר להגדיר את השלבים מראש.

Agent Loop מתאים כאשר הדרך אל התשובה לא ידועה מראש. למשל, אם צריך לחקור בעיה, לבדוק כמה מקורות, להפעיל כלים שונים לפי התוצאה, או להחליט בכל שלב מה חסר.

הבחירה הנכונה אינה לפי מה שנשמע מתקדם יותר, אלא לפי אופי המשימה. אם התהליך ברור, עדיף להתחיל מ-Workflow. אם המשימה פתוחה ודורשת החלטות תוך כדי עבודה, Agent Loop יכול להתאים יותר.

בפועל, הרבה מערכות טובות משלבות בין השניים: יש להן מסגרת עבודה ברורה, ובתוכה נקודות מסוימות שבהן ה-LLM עוזר לבחור את הצעד הבא.
