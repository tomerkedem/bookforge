# הכרת סביבת העבודה והקבצים

אחרי שהבנו את הרעיונות המרכזיים של Agents, אפשר לעבור לחלק המעשי. לפני שנכנסים לקוד עצמו, חשוב להבין מה אנחנו עומדים לבנות, אילו קבצים קיימים, ומה התפקיד של כל קובץ.

המטרה כאן היא לא לרוץ ישר להרצה, אלא לבנות תמונה מסודרת של הפרויקט. כאשר יודעים מה כל קובץ עושה, קל יותר להבין את הקוד, לתקן תקלות, ולהרחיב את המערכת בהמשך.

## מה נבנה

בחלק המעשי נבנה שתי מערכות קטנות, שכל אחת מדגימה יכולת אחרת בעולם של AI Agents.

המערכת הראשונה היא **RAG Chatbot**. זו מערכת שמקבלת שאלה מהמשתמש, מחפשת מידע רלוונטי במסמכים, מעבירה את המידע הזה ל-LLM, ומחזירה תשובה שמבוססת על ה-context שנשלף.

הזרימה הכללית שלה נראית כך:

User question 
 ↓ 
Search relevant documents 
 ↓ 
Retrieve context 
 ↓ 
Send context + question to LLM 
 ↓ 
Return answer

המערכת הזאת מלמדת איך נותנים ל-LLM זיכרון חיצוני. במקום שהמודל יענה רק מהידע הפנימי שלו, הוא מקבל מידע ממסמכים שאנחנו סיפקנו.

המערכת השנייה היא **Stock Agent עם כלי חיצוני**. זו מערכת שמקבלת שאלה על מניה, מזהה שצריך מידע עדכני, מפעילה כלי שמביא נתוני מניה, ואז מחזירה תשובה מסודרת למשתמש.

הזרימה הכללית שלה נראית כך:

User question 
 ↓ 
Agent understands that stock data is needed 
 ↓ 
Agent calls stock tool 
 ↓ 
Tool returns market data 
 ↓ 
Agent explains the result

שתי המערכות משלימות זו את זו. ה-RAG Chatbot מדגים עבודה עם זיכרון חיצוני ומסמכים. ה-Stock Agent מדגים עבודה עם Tool שמביא מידע ממקור חיצוני בזמן אמת.

ביחד, הן נותנות בסיס טוב להבנת Agent מעשי: מערכת שמבינה בקשה, משתמשת במידע או בכלים לפי הצורך, ומחזירה תשובה שאינה מבוססת רק על ניחוש של המודל.

## סדר הקבצים

כדי להבין את החלק המעשי בצורה נכונה, כדאי ללמוד את הקבצים לפי סדר הבנייה שלהם, ולא לפי סדר אקראי.

הסדר הנכון הוא:

```bash
requirements.txt
sample_docs.txt
build_rag_db.py
rag_chatbot.py
stock_agent.py
```

הסיבה לסדר הזה פשוטה: לפני שמריצים קוד, צריך להבין אילו ספריות נדרשות. אחר כך צריך להבין מהו מקור המידע שעליו נעבוד. רק לאחר מכן אפשר לבנות את בסיס ה-RAG, להשתמש בו בצ’אטבוט, ולבסוף לעבור לדוגמה של Agent שמפעיל כלי חיצוני.

הקובץ הראשון, requirements.txt, מגדיר את הספריות של הפרויקט. הוא כולל ספריות לעבודה עם LangChain, ChromaDB, Anthropic, Ollama, yfinance, FastAPI ו-Jinja2. הספריות האלה רומזות כבר על מה שנבנה בהמשך: RAG, מודלים, כלים חיצוניים, ואפשרות להרחיב את המערכת לממשק Web.

אחריו מגיע sample_docs.txt. זהו מקור ידע קטן לדוגמה. הוא מאפשר לנו להבין את רעיון ה-RAG בלי להתחיל ממסמכים גדולים ומורכבים. בקובץ כזה אפשר לראות איך טקסט רגיל הופך בהמשך לידע שניתן לחיפוש ושליפה.

לאחר מכן מגיע build_rag_db.py. זה הקובץ שבונה את בסיס ה-RAG: הוא טוען מסמכים, מחלק אותם ל-chunks, יוצר embeddings, ושומר אותם ב-ChromaDB. זה שלב ההכנה. בלי השלב הזה, לצ’אטבוט אין מאגר ידע לשלוף ממנו מידע.

רק אחרי שבנינו את המאגר, עוברים ל-rag_chatbot.py. הקובץ הזה משתמש ב-vector store שנבנה קודם, שולף ממנו context רלוונטי, ומעביר אותו ל-LLM יחד עם שאלת המשתמש.

בסוף נלמד את stock_agent.py. כאן כבר לא מדובר רק בשליפה ממסמכים, אלא בסוכן שמשתמש בכלי חיצוני כדי להביא מידע על מניות. זו דוגמה טובה למעבר מ-RAG Bot למערכת Agentic יותר.

## תפקיד כל קובץ

אחרי שמבינים את סדר הקבצים, אפשר להבין מה כל אחד מהם עושה בתוך הפרויקט.

requirements.txt הוא קובץ ההתקנות של הפרויקט. הוא מגדיר אילו ספריות צריך להתקין כדי שהקוד יעבוד. במקרה שלנו הוא כולל ספריות לעבודה עם LangChain, ChromaDB, Anthropic, Ollama, yfinance, FastAPI, Jinja2 ועוד. כלומר, כבר מתוך הקובץ הזה אפשר להבין שהפרויקט משלב RAG, עבודה עם מודלים, כלים חיצוניים ואפשרות להרחבה לממשק Web.

sample_docs.txt הוא מקור ידע קטן לדוגמה. הוא מכיל טקסטים קצרים שמסבירים מושגים כמו LangChain, ChromaDB, RAG ו-Claude. המטרה שלו היא לתת לנו מסמכים פשוטים שאפשר להכניס למערכת RAG, במקום להתחיל מיד עם חומר גדול ומורכב.

build_rag_db.py הוא הקובץ שבונה את בסיס ה-RAG. הוא טוען קבצי טקסט, מחלק אותם ל-chunks, יוצר embeddings, ושומר את התוצאה ב-ChromaDB. זהו שלב ההכנה של מערכת ה-RAG. לפני שהצ’אטבוט יכול לענות לפי מסמכים, צריך קודם לבנות את המאגר שבו הוא יחפש.

rag_chatbot.py הוא הקובץ שמפעיל את הצ’אטבוט עצמו. הוא טוען את ה-vector store שנבנה קודם, יוצר retriever, שולף context רלוונטי לפי שאלת המשתמש, ומעביר את השאלה יחד עם ה-context ל-LLM. בנוסף, הוא שומר היסטוריית שיחה פשוטה כדי לתמוך בשאלות המשך.

stock_agent.py הוא הדוגמה של Agent שמשתמש בכלי חיצוני. במקום לשלוף מידע ממסמכים, הוא מגדיר tool בשם get_stock_info, שמשתמש ב-yfinance כדי להביא מידע על מניה לפי ticker symbol. כאן רואים את המעבר ממערכת שעונה לפי context למערכת שמפעילה כלי כדי להביא נתון חיצוני.

כך מתקבלת תמונה מסודרת של הפרויקט: קודם מתקינים ספריות, אחר כך מכינים מקור ידע, בונים מאגר RAG, מפעילים צ’אטבוט שמחפש במסמכים, ולבסוף בונים Agent שמפעיל כלי חיצוני.



## איך להריץ

אחרי שמבינים מה תפקיד כל קובץ, אפשר לראות את סדר ההרצה של הפרויקט.

השלב הראשון הוא התקנת הספריות:

```bash
pip install -r requirements.txt
```

הפקודה הזאת מתקינה את כל התלויות שהפרויקט צריך, כמו LangChain, ChromaDB, ספריות למודלים, yfinance, FastAPI ו-Jinja2. בלי השלב הזה, הקוד עלול להיכשל כבר ב-import הראשון.

אחרי ההתקנה, בונים את בסיס ה-RAG:

```bash
python build_rag_db.py
```

הקובץ הזה צריך לרוץ לפני הצ’אטבוט, כי הוא מכין את המאגר שבו הצ’אטבוט יחפש אחר כך. הוא טוען את המסמכים, מחלק אותם ל-chunks, יוצר embeddings ושומר את התוצאה ב-ChromaDB.

רק אחרי שהמאגר נבנה, אפשר להריץ את הצ’אטבוט:

```bash
python rag_chatbot.py
```

בשלב הזה המערכת טוענת את ה-vector store הקיים, מקבלת שאלות מהמשתמש, שולפת context רלוונטי, ומעבירה אותו ל-LLM כדי לקבל תשובה.

לבסוף, אפשר להריץ את דוגמת ה-Agent עם כלי חיצוני:

```bash
python stock_agent.py
```

כאן המערכת לא מחפשת במסמכים, אלא משתמשת בכלי שמביא מידע על מניה. זו דוגמה טובה להבנת ההבדל בין RAG Chatbot לבין Agent שמפעיל Tool.

סדר ההרצה המלא הוא:

```bash
pip install -r requirements.txt
python build_rag_db.py
python rag_chatbot.py
python stock_agent.py
```

חשוב לשים לב: אם מריצים את rag_chatbot.py לפני build_rag_db.py, הצ’אטבוט לא ימצא את מאגר ה-RAG. קודם בונים את הזיכרון החיצוני, ורק אחר כך משתמשים בו.
