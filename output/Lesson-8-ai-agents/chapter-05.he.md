# Tools - איך הסוכן מבצע פעולות

עד עכשיו ראינו שהסוכן יכול להבין בקשות, להשתמש ב-LLM, ולשלוף מידע מזיכרון או ממערכת RAG. אבל כדי שסוכן יהיה באמת שימושי, הוא צריך לפעמים לעשות יותר מאשר לענות. הוא צריך להתחבר למערכות אחרות, לקרוא מידע, לבצע חישוב, להריץ פעולה או להפעיל שירות חיצוני.

כאן נכנסים ה-Tools. הם הגשר בין ה-LLM לבין העולם שמחוץ לטקסט.

## מהו Tool

Tool הוא פונקציה או שירות שה-Agent יכול להפעיל כחלק מהעבודה שלו.

במקום שהמודל רק ינסח תשובה, הוא יכול לבקש מהמערכת להפעיל כלי מסוים. הכלי מבצע פעולה מוגדרת, מחזיר תוצאה, ואז ה-Agent משתמש בתוצאה הזאת כדי להמשיך את התהליך או להחזיר תשובה למשתמש.

Tool יכול להיות משהו ממש פשוט. למשל פונקציה שמחשבת סכום:

```python
def calculate_total(price, quantity):
    return price * quantity
```

Tool יכול גם להיות חיבור לשירות חיצוני:

```python
get_stock_info("AAPL")
```

או חיפוש במסמכים:

```python
search_documents("What does the document say about ChromaDB?")
```

או פעולה במערכת אחרת:

```python
create_support_ticket(issue_description)
```

הנקודה החשובה היא שלכל Tool צריך להיות תפקיד ברור. הסוכן צריך לדעת מה הכלי עושה, איזה קלט הוא מקבל, ומה הפלט שהוא מחזיר.

לדוגמה, כלי בשם get_stock_info יכול לקבל סימול מניה כמו AAPL, לפנות לשירות נתוני שוק, ולהחזיר מחיר נוכחי, מטבע, טווח יומי ונפח מסחר.

במצב כזה, ה-LLM לא צריך לנחש את מחיר המניה. הוא צריך להבין שהשאלה דורשת נתון עדכני, ואז להשתמש בכלי המתאים.

אפשר לחשוב על Tool כמו יכולת חיצונית שהוספנו לסוכן. ה-LLM מספק את ההבנה והניסוח, אבל ה-Tool מספק פעולה מוגדרת בעולם האמיתי או במערכת חיצונית.

## למה צריך Tools

LLM לבדו לא באמת מבצע פעולות. הוא יכול להסביר איך לבצע פעולה, לכתוב קוד שמבצע פעולה, או לתאר מה צריך לקרות, אבל הוא לא ניגש בעצמו למערכת חיצונית בלי שמחברים אותו לכלי מתאים.

לדוגמה, אם המשתמש שואל:

```bash
What is the current price of Apple stock?
```

ה-LLM יכול להבין שהשאלה קשורה למניית Apple, ואפילו לזהות שהסימול הוא כנראה AAPL. אבל הוא לא אמור לנחש את המחיר הנוכחי. כדי לענות נכון, צריך Tool שמביא נתוני מניה ממקור חיצוני.

אותו רעיון נכון גם במקרים אחרים.

אם המשתמש שואל:

```bash
What does this document say about LangChain?
```

צריך כלי שמחפש במסמכים או במערכת RAG.

אם המשתמש שואל:

```bash
What is the status of order 12345?
```

צריך כלי שמתחבר למערכת הזמנות.

אם המשתמש מבקש:

```bash
Create a support ticket for this issue.
```

צריך כלי שיודע לפתוח קריאה במערכת המתאימה.

בלי Tools, המודל נשאר בעיקר בעולם הטקסט. הוא יכול להסביר, לנמק, לנסח ולסכם, אבל הוא לא באמת בודק נתונים חיים, לא קורא API, לא שולף ממסד נתונים, ולא מבצע פעולה במערכת חיצונית.

Tools הם מה שמאפשרים לסוכן להפוך ממערכת שמדברת על פעולה למערכת שמשתתפת בפעולה. ה-LLM מבין מה המשתמש רוצה, וה-Tool מספק את היכולת המעשית לבצע או לבדוק את מה שנדרש.



## דוגמאות Tools

Tool יכול להיות כמעט כל פעולה מוגדרת שהמערכת יודעת לבצע. לפעמים זה כלי שמחזיר מידע, לפעמים כלי שמבצע פעולה, ולפעמים כלי שעוזר לעבד נתונים.

דוגמה בסיסית היא חיפוש במסמכים. כאשר המשתמש שואל שאלה על תוכן של קובץ, הסוכן יכול להפעיל כלי חיפוש שמחזיר את הקטעים הרלוונטיים:

```python
search_documents("What does the document say about embeddings?")
```

דוגמה נוספת היא קריאה ל-API. אם המשתמש שואל על סטטוס הזמנה, הסוכן יכול להפעיל כלי שמתחבר למערכת הזמנות:

```python
get_order_status(order_id="12345")
```

במקרה של מניות, כלי יכול להביא מידע עדכני על חברה:

```python
get_stock_info("AAPL")
```

Tool יכול גם לשלוף מידע ממסד נתונים:

```python
get_customer_details(customer_id="987")
```

או לבצע חישוב:

```bash
calculate_shipping_price(weight=2.4, country="Israel")
```

במערכות מתקדמות יותר, כלי יכול ליצור קובץ, לשלוח מייל, לפתוח קריאה, להריץ בדיקות קוד או לעדכן רשומה במערכת.

חשוב לשים לב שלכל כלי יש רמת סיכון שונה. כלי שמחפש במסמכים או מחשב מחיר הוא יחסית בטוח. כלי ששולח מייל, מוחק קובץ או מעדכן נתונים במערכת דורש הרבה יותר בקרה.

לכן כשמתכננים Tools לסוכן, לא מספיק לשאול “מה הכלי יכול לעשות?”. צריך לשאול גם “מה עלול לקרות אם הכלי יופעל בטעות?”.

## איך Agent בוחר Tool

כדי ש-Agent יוכל להשתמש בכלי, הוא צריך לדעת אילו כלים קיימים ומה כל כלי עושה. בדרך כלל המערכת מעבירה ל-LLM תיאור קצר של הכלים הזמינים, כולל שם הכלי, המטרה שלו, והקלט שהוא מצפה לקבל.

לדוגמה, כלי מניות יכול להיות מתואר כך:

```python
get_stock_info(symbol)
Use this tool when the user asks about stock price, quote, or market data.
Input: stock ticker symbol, for example AAPL or MSFT.
```

כאשר המשתמש שואל:

```bash
What is the current price of Apple stock?
```

ה-LLM יכול להבין שהשאלה קשורה למחיר מניה, לזהות שהסימול הוא כנראה AAPL, ולבחור את הכלי get_stock_info.

הזרימה נראית כך:

```bash
User question
     ↓
LLM understands the intent
     ↓
Agent selects the relevant tool
     ↓
Tool runs and returns data
     ↓
LLM explains the result
```

חשוב שהכלים יהיו מוגדרים בצורה ברורה. אם תיאור הכלי כללי מדי, המודל עלול להשתמש בו במקרים לא מתאימים. אם יש כמה כלים דומים, צריך להסביר היטב מתי להשתמש בכל אחד מהם.

לדוגמה, אם יש שני כלים:

```python
search_documents(query)
get_stock_info(symbol)
```

ה-Agent צריך להבין ששאלה על מסמך פנימי שייכת ל-search_documents, ושאלה על מחיר מניה שייכת ל-get_stock_info.

בחלק מהמקרים הבחירה בכלי יכולה להתבצע על ידי ה-LLM. במקרים אחרים עדיף להשתמש בלוגיקה רגילה בקוד. למשל, אם הבקשה כוללת מספר הזמנה, אפשר להפעיל כלי סטטוס הזמנה לפי כלל ברור. לא תמיד צריך לתת למודל להחליט הכול.

הגישה הטובה היא לשלב: לתת ל-LLM להבין שפה וכוונה, אבל לשמור בקוד את הגבולות החשובים. כך הסוכן יכול להיות גמיש, ועדיין להישאר צפוי ובטוח יותר.



## גבולות ובטיחות

Tools נותנים ל-Agent כוח אמיתי, ולכן צריך להגדיר להם גבולות ברורים. ככל שכלי יכול לעשות פעולה משמעותית יותר, כך חשוב יותר לשלוט מתי ואיך מפעילים אותו.

יש הבדל גדול בין כלי שקורא מידע לבין כלי שמשנה מידע.

לדוגמה, כלי כזה יחסית בטוח:

```python
search_documents(query)
```

הוא רק מחפש ומחזיר מידע.

גם כלי כזה בדרך כלל בטוח יחסית:

```python
get_stock_info(symbol)
```

הוא מביא נתוני מניה, אבל לא מבצע פעולה כספית.

לעומת זאת, כלים כאלה כבר דורשים הרבה יותר זהירות:

```python
send_email(to, subject, body)
delete_file(path)
update_customer_record(customer_id, data)
create_payment(amount, recipient)
```

כאן טעות יכולה לגרום לנזק אמיתי: הודעה תישלח לאדם הלא נכון, קובץ יימחק, נתון ישתנה, או פעולה כספית תתבצע בטעות.

לכן במערכת Agentic טובה צריך להגדיר כמה שכבות הגנה:

- אילו כלים זמינים לסוכן.

- מתי מותר להשתמש בכל כלי.

- אילו פעולות דורשות אישור משתמש.

- אילו הרשאות נדרשות.

- מה קורה אם הכלי נכשל.

- כמה זמן מחכים לפני timeout.

- איך מתעדים את הפעולה בלוגים.

לדוגמה, אם המשתמש מבקש לשלוח מייל, לא תמיד נכון לשלוח אותו מיד. עדיף שה-Agent יכין טיוטה, יציג אותה למשתמש, ורק אחרי אישור מפורש יבצע שליחה.

```bash
User request
   ↓
Agent drafts the email
   ↓
User confirms
   ↓
Tool sends the email
```

כך הסוכן עדיין עוזר, אבל לא מקבל חופש בלתי מוגבל.

העיקרון החשוב הוא פשוט: ככל שכלי משפיע יותר על העולם החיצוני, כך צריך יותר בקרה. Tools הם מה שהופך Agent למערכת שימושית, אבל גם מה שמחייב תכנון אחראי.
