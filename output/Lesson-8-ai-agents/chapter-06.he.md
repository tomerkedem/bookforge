# Instructions - איך מגדירים התנהגות

אחרי שראינו איך Agent משתמש ב-LLM, בזיכרון ובכלים, צריך להבין איך מכוונים את ההתנהגות שלו. מערכת Agentic לא אמורה לפעול בצורה חופשית לגמרי. היא צריכה לדעת איך לענות, מתי להשתמש בכלים, מתי לא לנחש, ומתי לעצור.

כאן נכנסות ה-Instructions. הן שכבת ההכוונה של המערכת.

## מהן Instructions

Instructions הן הנחיות שמגדירות איך המערכת אמורה להתנהג.

הן יכולות לקבוע את סגנון התשובה, את גבולות הפעולה, את הדרך שבה משתמשים בכלים, ואת מה שהמערכת צריכה לעשות כאשר חסר מידע.

לדוגמה, במערכת RAG נרצה להגדיר הנחיה כזאת:

```bash
Answer only based on the provided context.
If the context does not contain the answer, say that the information is not available.
```

המשמעות היא שהמערכת לא אמורה לענות לפי ידע כללי כאשר השאלה אמורה להתבסס על מסמכים. אם אין context מתאים, עדיף להחזיר תשובה זהירה מאשר להמציא.

ב-Agent שמשתמש בכלי מניות, אפשר להגדיר הנחיה אחרת:

```bash
Use the stock data tool only when the user asks about stock price, market data, or a company ticker.
```

כאן ההנחיה עוזרת למערכת להבין מתי הכלי רלוונטי ומתי לא.

Instructions יכולות גם להגדיר מגבלות בטיחות:

```bash
Do not perform actions that change external systems without explicit user confirmation.
```

זו הנחיה חשובה במיוחד כאשר ל-Agent יש כלים שיכולים לשלוח מייל, לעדכן נתונים, ליצור קבצים או לבצע פעולה במערכת חיצונית.

במילים פשוטות, Instructions הן לא התשובה עצמה. הן הכללים שמכוונים את הדרך שבה ה-Agent מגיע לתשובה או לפעולה.



## ההבדל בין Prompt לבין Instructions

כדאי להפריד בין שני דברים שנראים דומים, אבל יש להם תפקיד שונה: Prompt ו-Instructions.

Prompt הוא מה שהמשתמש מבקש עכשיו.

לדוגמה:

```bash
Explain what RAG is.
```

או:

```bash
What is the current price of Apple stock?
```

או:

```bash
Summarize this document in simple Hebrew.
```

אלה בקשות נקודתיות. הן משתנות משיחה לשיחה ומהודעה להודעה.

Instructions הן הכללים הקבועים יחסית שמכוונים את המערכת. הן לא שייכות רק לבקשה אחת, אלא מגדירות איך ה-Agent אמור להתנהג לאורך כל העבודה.

לדוגמה:

```bash
Answer clearly and concisely.
Use tools only when needed.
Do not invent information if the context is missing.
Ask for confirmation before performing external actions.
```

אפשר לחשוב על זה כך:

```python
Prompt = what the user wants now
Instructions = how the system should behave
```

אם המשתמש שואל שאלה על מסמך, ה-Prompt הוא השאלה עצמה. ה-Instructions יכולות להגדיר שהמערכת חייבת לענות רק לפי המידע שנשלף מהמסמך.

אם המשתמש שואל על מניה, ה-Prompt הוא השאלה על המניה. ה-Instructions יכולות להגדיר שמותר להשתמש בכלי מניות רק כאשר השאלה באמת קשורה למחיר, quote או market data.

ההפרדה הזאת חשובה כי היא מאפשרת לבנות מערכת עקבית יותר. המשתמש יכול לשאול הרבה שאלות שונות, אבל ההתנהגות הבסיסית של ה-Agent נשארת יציבה: לא לנחש, להשתמש בכלים רק כשצריך, לבקש אישור לפני פעולה רגישה, ולהחזיר תשובה ברורה.



## דוגמאות להנחיות

Instructions טובות צריכות להיות ברורות, קצרות ומעשיות. הן לא אמורות להיות אוסף כללי של משפטים יפים, אלא כללים שעוזרים למערכת לדעת איך להתנהג במצבים אמיתיים.

לדוגמה, במערכת שעונה לפי מסמכים, אפשר להגדיר:

```bash
Answer only based on the provided context.
```

הנחיה כזאת חשובה כאשר אנחנו לא רוצים שהמודל יענה לפי ידע כללי. אם המטרה היא לענות לפי מסמכי החברה, לפי קובץ שנטען, או לפי מידע שנשלף מ-RAG, התשובה צריכה להיות מבוססת על אותו context.

אפשר להוסיף גם הנחיה למקרה שבו אין מספיק מידע:

```bash
If the context does not contain relevant information, say that you do not have enough information.
```

זו הנחיה חשובה מאוד, כי היא מלמדת את המערכת לא להמציא תשובה כאשר המקור לא באמת תומך בה.

ב-Agent שמשתמש בכלי מניות, אפשר להגדיר:

```bash
Use the stock data tool only when the user asks about stock price, market data, or a company ticker.
```

כך המערכת מבינה שהכלי לא מיועד לכל שאלה על חברה, אלא רק לשאלות שבהן באמת צריך נתוני שוק.

במערכת שיכולה לבצע פעולות חיצוניות, חשוב להגדיר גבול ברור:

```bash
Do not perform actions that modify external systems without explicit user confirmation.
```

כלומר, מותר להכין טיוטה, להציע פעולה או להסביר מה צריך לעשות, אבל לא לבצע פעולה שמשנה מערכת חיצונית בלי אישור ברור.

אפשר גם להגדיר סגנון תשובה:

```bash
Keep the answer short, clear, and practical.
```

הנחיה כזאת לא קשורה לכלים או לזיכרון, אלא לאיכות התקשורת עם המשתמש. Agent טוב לא רק מבצע פעולה נכונה, אלא גם מסביר אותה בצורה שהמשתמש יכול להבין.

בפועל, Instructions טובות משלבות שלושה סוגי כללים: איך לענות, מתי להשתמש בכלים, ומה לא לעשות. ככל שה-Agent מקבל יותר יכולת פעולה, כך ההנחיות צריכות להיות מדויקות יותר.



## למה Instructions לא מספיקות

Instructions הן חשובות, אבל אסור להתייחס אליהן כאל מנגנון שמבטיח התנהגות מושלמת. בסוף, המודל עדיין מפרש טקסט. הוא יכול לטעות, להבין הנחיה בצורה לא מדויקת, או לבחור פעולה שלא התכוונו אליה.

לדוגמה, אפשר להגדיר:

```bash
Do not invent information.
```

זו הנחיה טובה, אבל היא לא מבטיחה שהמודל לעולם לא יטעה. אם ה-context חלקי, אם השאלה לא ברורה, או אם המודל מפרש משהו לא נכון, עדיין יכולה להופיע תשובה לא מדויקת.

אותו דבר נכון גם לגבי כלים. אפשר לכתוב:

```bash
Use the stock tool only for stock market questions.
```

אבל במערכת אמיתית עדיף לא להסתמך רק על המשפט הזה. כדאי להוסיף גם לוגיקה בקוד שמגבילה מתי הכלי זמין, בודקת את הקלט, מטפלת בשגיאות, ומונעת שימוש לא מתאים.

במיוחד כאשר יש כלים שמבצעים פעולות חיצוניות, Instructions לבדן לא מספיקות. אם כלי יכול לשלוח מייל, לעדכן רשומה, למחוק קובץ או לבצע פעולה רגישה, צריך להוסיף שכבות הגנה אמיתיות:

- בדיקת הרשאות.

- אישור משתמש לפני פעולה.

- validation לקלט.

- allowlist לפעולות מותרות.

- timeout.

- logging.

- טיפול ברור בשגיאות.

אפשר לחשוב על זה כך:

```bash
Instructions guide the model.
Code enforces the rules.
```

הנחיות עוזרות למודל להבין איך להתנהג, אבל הקוד הוא זה שצריך לאכוף את הגבולות החשובים. Agent טוב משלב בין שניהם: Instructions שמכוונות את ההתנהגות, וארכיטקטורה שמוודאת שהמערכת נשארת בטוחה, צפויה ומבוקרת.
