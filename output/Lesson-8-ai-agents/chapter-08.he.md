# דפוסי עבודה לבניית מערכות Agentic

אחרי שהבנו את ההבדל בין Workflow מובנה לבין Agent Loop, אפשר להכיר כמה דפוסי עבודה נפוצים לבניית מערכות Agentic.

המטרה כאן היא לא לזכור שמות, אלא להבין מתי כל דפוס מתאים. לא כל מערכת צריכה להיות Agent חופשי שמחליט הכול לבד. לפעמים הפתרון הנכון הוא שרשרת שלבים פשוטה. לפעמים צריך לנתב בקשה למסלול מתאים. לפעמים כדאי להריץ כמה משימות במקביל. ולפעמים באמת צריך סוכן שמחליט בכל שלב מה לעשות בהמשך.

## Prompt Chaining

Prompt Chaining הוא דפוס שבו מפרקים משימה לרצף של שלבים. כל שלב מקבל את הפלט של השלב הקודם, מעבד אותו, ומעביר תוצאה לשלב הבא.

במקום לבקש מה-LLM לבצע משימה גדולה בבת אחת, מחלקים אותה למשימות קטנות וברורות יותר.

לדוגמה, נניח שאנחנו רוצים ליצור תשובה איכותית ואז לתרגם אותה. אפשר לבנות תהליך כזה:

```bash
draft → review → rewrite → translate → final answer
```

- בשלב הראשון המודל כותב טיוטה.

- בשלב השני הוא בודק אם הטיוטה ברורה ומדויקת.

- בשלב השלישי הוא משפר את הניסוח.

- בשלב הרביעי הוא מתרגם.

- ובסוף מתקבלת תשובה סופית.

היתרון של Prompt Chaining הוא שליטה. במקום לתת למודל משימה גדולה ומעורפלת, אנחנו מגדירים לו תהליך ברור. כל שלב עושה דבר אחד, ולכן קל יותר להבין איפה נוצרה בעיה.

לדוגמה, אם התרגום יצא טוב אבל הטיוטה המקורית הייתה חלשה, הבעיה היא לא בשלב התרגום אלא בשלב הכתיבה. אם הטיוטה טובה אבל הניסוח הסופי מסורבל, אפשר לשפר רק את שלב השכתוב.

Prompt Chaining מתאים במיוחד למשימות שאפשר לפרק לשלבים טבעיים:

- כתיבת מסמך.

- סיכום ואז שכתוב.

- יצירת תשובה ואז בדיקת איכות.

- ניסוח מייל ואז קיצור שלו.

- יצירת קוד ואז הסבר שלו.

- יצירת תוכן ואז תרגום.

לדוגמה, בתהליך לימודי אפשר להשתמש בשרשרת כזאת:

```bash
explain concept → add example → add code → simplify language
```

כך לא מבקשים מהמודל “כתוב לי הכול מושלם”. במקום זה בונים את התוצאה בהדרגה.

הטעות הנפוצה היא ליצור שרשרת ארוכה מדי. אם יש יותר מדי שלבים, התהליך נעשה איטי, יקר וקשה לתחזוקה. לפעמים שניים או שלושה שלבים מספיקים. המטרה היא לא להרבות בשלבים, אלא לחלק את המשימה במקום שבו החלוקה באמת משפרת את האיכות.

## Routing

Routing הוא דפוס שבו המערכת מקבלת בקשה ומנתבת אותה למסלול המתאים. במקום שכל בקשה תעבור באותו תהליך, ה-Agent או ה-Workflow בודקים מה סוג הבקשה, ואז מפעילים את המסלול הנכון.

לדוגמה:

```bash
billing question → Billing flow
technical question → Technical flow
document question → RAG flow
general question → General assistant
```

נניח שמשתמש כותב:

```bash
I was charged twice for the same order.
```

זו כנראה שאלה שקשורה לחיוב, ולכן כדאי לשלוח אותה למסלול Billing.

אם המשתמש כותב:

```bash
The chatbot cannot find answers in my documents.
```

זו כבר בעיה טכנית, ולכן מתאים לשלוח אותה למסלול Technical.

ואם המשתמש כותב:

```bash
What does the uploaded file say about ChromaDB?
```

כאן מתאים להפעיל מסלול RAG, כי התשובה צריכה להגיע ממסמכים.

היתרון של Routing הוא שכל מסלול יכול להיות ממוקד יותר. במקום לבנות Prompt אחד ענק שמנסה לטפל בכל סוגי הבקשות, בונים כמה מסלולים קטנים וברורים. כל מסלול מקבל את ההנחיות, הכלים וה-context שמתאימים לו.

זה גם עוזר להקטין רעש. שאלה על מסמכים לא צריכה לקבל את כל ההנחיות של מערכת חיובים. שאלה טכנית לא צריכה לקבל context של שירות לקוחות. ככל שהמסלול ממוקד יותר, קל יותר לשפר אותו.

Routing מתאים במיוחד כאשר יש כמה סוגי בקשות שחוזרים על עצמם, וכל סוג דורש טיפול שונה.

הטעות הנפוצה היא להסתמך על Routing לא מדויק. אם המערכת מסווגת בקשה למסלול לא נכון, כל התהליך שאחר כך עלול להיכשל. לכן כדאי להגדיר קטגוריות ברורות, לתת דוגמאות, ובמקרים לא בטוחים לבחור מסלול כללי או לשאול את המשתמש שאלת הבהרה.

## Parallelization

Parallelization הוא דפוס שבו מריצים כמה משימות במקביל במקום לבצע אותן אחת אחרי השנייה. המטרה היא לחסוך זמן, לחלק משימה גדולה לחלקים קטנים יותר, או לקבל כמה תוצאות שאפשר להשוות ביניהן.

יש שני שימושים נפוצים לדפוס הזה.

הראשון הוא **Sectioning** - חלוקה לחלקים. למשל, אם יש לנו מסמך ארוך, אפשר לחלק אותו לכמה חלקים ולסכם כל חלק בנפרד:

```bash
chapter 1 → summary 1
chapter 2 → summary 2
chapter 3 → summary 3
        ↓
final summary
```

במקום לשלוח את כל המסמך בבת אחת, כל חלק מטופל בנפרד. אחר כך מאחדים את התוצאות לתשובה אחת.

השימוש השני הוא **Voting** - הרצת אותה משימה כמה פעמים והשוואת התוצאות. לדוגמה, אפשר לבקש מכמה קריאות LLM לבדוק את אותה תשובה, ואז לבחור את הבדיקה החוזרת או המשכנעת ביותר.

```bash
same task → answer A
same task → answer B
same task → answer C
        ↓
compare and choose
```

Parallelization מתאים כאשר המשימה ניתנת לחלוקה, או כאשר רוצים לשפר אמינות באמצעות כמה תוצאות שונות.

לדוגמה, אם רוצים לסכם ספר, אפשר לחלק אותו לפרקים. אם רוצים לבדוק איכות של תשובה, אפשר להריץ כמה בדיקות עצמאיות. אם רוצים לנתח קוד גדול, אפשר לחלק אותו לקבצים או למודולים.

הטעות הנפוצה היא להשתמש ב-Parallelization גם כשאין באמת עצמאות בין החלקים. אם כל שלב תלוי בתוצאה של השלב הקודם, אי אפשר להריץ אותם במקביל בצורה פשוטה. במקרה כזה עדיף להשתמש ב-Prompt Chaining או Workflow רגיל.

## Orchestrator-workers

Orchestrator-workers הוא דפוס שבו יש רכיב מרכזי שמנהל את העבודה, ולצידו כמה workers שמבצעים חלקים שונים של המשימה.

ה-Orchestrator מקבל את הבקשה הכללית, מפרק אותה לתת-משימות, מחליט מי צריך לבצע כל חלק, ואז מאחד את התוצאות לתשובה אחת.

אפשר לתאר את זה כך:

User task 
 ↓ 
Orchestrator 
 ↓ 
Worker 1 → result 1 
Worker 2 → result 2 
Worker 3 → result 3 
 ↓ 
Final combined answer

לדוגמה, אם המשתמש מבקש:

```bash
Review this project and tell me what should be improved.
```

ה-Orchestrator יכול לחלק את העבודה לכמה כיוונים:

- Worker אחד בודק את מבנה הקבצים.

- Worker שני בודק איכות קוד.

- Worker שלישי בודק תלויות וספריות.

- Worker רביעי בודק האם חסרות הוראות הרצה.

אחרי שכל worker מחזיר תוצאה, ה-Orchestrator מאחד את הכול לתשובה אחת מסודרת.

הדפוס הזה מתאים למשימות מורכבות, במיוחד כאשר לא רוצים שכל העבודה תתבצע בקריאה אחת גדולה ל-LLM. במקום להעמיס על מודל אחד את כל ההקשר וכל ההחלטות, מחלקים את המשימה לחלקים ברורים יותר.

היתרון הוא גמישות. ה-Orchestrator יכול להחליט איך לחלק את העבודה לפי הבקשה. במשימה אחת הוא יפעיל שני workers, ובמשימה אחרת חמישה.

הטעות הנפוצה היא ליצור יותר מדי workers בלי צורך אמיתי. אם המשימה פשוטה, החלוקה רק תוסיף מורכבות, זמן ועלות. Orchestrator-workers מתאים כאשר המשימה באמת מורכבת מספיק כדי להצדיק חלוקה וניהול.

## Evaluator-optimizer

Evaluator-optimizer הוא דפוס שבו רכיב אחד מייצר תשובה, ורכיב אחר בודק אותה ומציע שיפור. במקום להסתפק בפלט הראשון של המודל, מוסיפים שלב ביקורת שמנסה לזהות חולשות, חוסר דיוק או ניסוח לא מספיק ברור.

הזרימה יכולה להיראות כך:

generate answer 
 ↓ 
evaluate answer 
 ↓ 
improve answer 
 ↓ 
final answer

לדוגמה, אם המערכת כותבת תשובה לפי מסמכים, אפשר להוסיף שלב שבודק:

- האם התשובה באמת מבוססת על ה-context.

- האם היא עונה על השאלה.

- האם יש בה מידע שלא הופיע במקור.

- האם הניסוח ברור.

- האם חסרה הסתייגות כאשר אין מספיק מידע.

במקום להחזיר מיד את התשובה הראשונה, המערכת נותנת לה לעבור בדיקה ושיפור.

הדפוס הזה מתאים במיוחד כאשר איכות הפלט חשובה. למשל: תשובות מקצועיות, סיכומים, בדיקת קוד, מסמכים עסקיים, או מערכות RAG שבהן חשוב לוודא שהתשובה לא ממציאה מידע.

הטעות הנפוצה היא לחשוב שה-Evaluator תמיד צודק. גם הוא יכול לטעות, במיוחד אם הקריטריונים לא ברורים. לכן חשוב להגדיר מראש מה בדיוק בודקים: דיוק, בהירות, התאמה ל-context, שלמות, סגנון או עמידה בהנחיות.



## Agent Loop

Agent Loop הוא הדפוס שבו הסוכן מחליט בכל שלב מה לעשות בהמשך. במקום לעבור במסלול קבוע מראש, הוא פועל במחזוריות: בודק את המצב, חושב מה חסר, מפעיל פעולה, בודק את התוצאה, ואז מחליט אם להמשיך.

המחזור הבסיסי נראה כך:

```bash
observe → think → act → observe → think → act
```

לדוגמה, המשתמש מבקש:

```bash
Find out why my RAG chatbot gives weak answers.
```

זו לא משימה עם פתרון אחד ידוע מראש. ייתכן שהבעיה היא בחלוקה ל-chunks, בבחירת embeddings, בכמות ה-context, באיכות המסמכים, ב-prompt, או בדרך שבה ה-retriever מחזיר תוצאות.

Agent Loop יכול לפעול כך:

observe the problem 
 ↓ 
check the retrieval logic 
 ↓ 
inspect chunk size 
 ↓ 
review the prompt 
 ↓ 
suggest changes 
 ↓ 
ask whether to apply them

היתרון הוא גמישות. הסוכן לא חייב לדעת מראש מה הבעיה. הוא יכול להתקדם לפי מה שהוא מגלה.

אבל זה גם הדפוס שדורש הכי הרבה בקרה. אם לא מגדירים גבולות, הסוכן עלול לבצע יותר מדי צעדים, להפעיל כלים לא מתאימים, או להמשיך לחפש גם כשכבר יש תשובה טובה מספיק.

לכן Agent Loop מתאים בעיקר למשימות פתוחות: חקירה, דיבוג, חיפוש מידע, תכנון, בדיקת מערכת או עבודה שבה לא יודעים מראש כמה צעדים יידרשו.

במערכת טובה כדאי להגדיר מראש מגבלות פשוטות: כמה צעדים מותר לבצע, אילו כלים זמינים, מתי לעצור, ומתי צריך אישור מהמשתמש.





## טבלה מסכמת

אחרי שמכירים את הדפוסים המרכזיים, קל יותר לבחור את המבנה הנכון לפי סוג המשימה. אין דפוס אחד שהוא תמיד הטוב ביותר. הבחירה תלויה ברמת הגמישות שצריך, ברמת הצפיות שרוצים, ובשאלה האם התהליך ידוע מראש או משתנה תוך כדי עבודה.

<div dir="rtl">

| **דפוס** | **מתאים **ל | **רמת גמישות** | **רמת צפיות** |
| --- | --- | --- | --- |
| **Prompt Chaining** | תהליך בשלבים | בינונית | גבוהה |
| **Routing** | סוגי בקשות שונים | בינונית | גבוהה |
| **Parallelization** | משימות ניתנות לחלוקה | בינונית | גבוהה |
| **Orchestrator-workers** | משימות מורכבות | גבוהה | בינונית |
| **Evaluator-optimizer** | שיפור איכות | בינונית | בינונית |
| **Agent Loop** | משימות פתוחות | גבוהה מאוד | נמוכה |

Prompt Chaining מתאים כאשר יודעים מראש מה סדר השלבים. Routing מתאים כאשר יש כמה סוגי בקשות וכל סוג צריך טיפול אחר. Parallelization מתאים כאשר אפשר לחלק את העבודה לחלקים עצמאיים. Orchestrator-workers מתאים כאשר יש משימה מורכבת שצריך לפרק באופן גמיש. Evaluator-optimizer מתאים כאשר איכות התוצאה חשובה וצריך שלב ביקורת. Agent Loop מתאים כאשר הדרך לפתרון לא ידועה מראש.

בפועל, הרבה מערכות לא משתמשות בדפוס אחד בלבד. מערכת יכולה להתחיל ב-Routing, להפעיל RAG במסלול אחד, להשתמש ב-Prompt Chaining במסלול אחר, ולהוסיף Evaluator שבודק את התשובה לפני שהיא חוזרת למשתמש. הבחירה הנכונה היא לא לבחור את הדפוס הכי מתקדם, אלא את הדפוס שהופך את המערכת לברורה, אמינה וקלה לבדיקה.
