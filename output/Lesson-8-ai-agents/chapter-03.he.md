# LLM - מנוע החשיבה של הסוכן

אחרי שראינו את המבנה הכללי של Agent, אפשר להתמקד ברכיב המרכזי שמאפשר לו להבין שפה, להסיק מסקנות ולנסח תשובות: ה-LLM.

חשוב להבין את התפקיד שלו בצורה מדויקת. ה-LLM הוא רכיב חזק מאוד, אבל הוא לא כל המערכת. הוא לא הזיכרון, לא הכלים, לא מסד הנתונים ולא מנגנון האבטחה. הוא מנוע השפה והחשיבה שה-Agent מפעיל בתוך תהליך רחב יותר.

## מה ה-LLM עושה

ה-LLM הוא הרכיב שמאפשר ל-Agent לעבוד עם שפה טבעית. המשתמש לא חייב לכתוב פקודה מדויקת או לבחור כפתור מתוך רשימה. הוא יכול לכתוב בקשה רגילה, לפעמים קצרה, לפעמים לא מדויקת, וה-LLM עוזר להבין למה הוא מתכוון.

לדוגמה, המשתמש יכול לכתוב:

```bash
Can you explain this code in simple words?
```

ה-LLM מבין שזו לא בקשה להריץ את הקוד, אלא להסביר אותו. הוא גם מבין שההסבר צריך להיות פשוט, ולא טכני מדי.

ב-Agent, ה-LLM יכול לעזור בכמה פעולות מרכזיות:

- להבין שפה טבעית.

- לזהות את כוונת המשתמש.

- לסווג את סוג הבקשה.

- לפרק משימה לתת-שלבים.

- לנסח תשובה ברורה.

- להסביר תוצאה שחזרה מכלי.

- להחליט, במקרים מסוימים, מה הצעד הבא.

לדוגמה, אם המשתמש שואל:

```bash
What does this document say about vector databases?
```

ה-LLM יכול להבין שהשאלה קשורה למסמך ולא לידע כללי. אבל הוא לא אמור להמציא את תוכן המסמך. ה-Agent צריך לשלוף את המידע הרלוונטי ממקור חיצוני, ורק אז ה-LLM ינסח תשובה לפי ה-context שקיבל.

בדוגמה אחרת, אם המשתמש כותב:

```bash
Check if this stock went up or down this year.
```

ה-LLM יכול להבין שצריך השוואה בין שני נתונים לאורך זמן. הוא יכול לעזור לזהות שצריך כלי שמביא נתוני מניה, ואחרי שהכלי מחזיר תוצאה, להסביר אותה למשתמש.

הכוח של ה-LLM הוא לא בכך שהוא יודע הכול, אלא בכך שהוא יודע לעבוד עם שפה, להבין כוונה, ולחבר בין מידע לבין תשובה ברורה. בתוך Agent, זו יכולת קריטית, אבל היא צריכה לעבוד יחד עם זיכרון, כלים והנחיות.

## למה זה חשוב

משתמשים לא תמיד כותבים בקשות בצורה מסודרת. הם לא תמיד יודעים איזה כלי צריך להפעיל, איזה מידע חסר, או איך לנסח את המשימה בצורה טכנית. הם פשוט כותבים את מה שהם צריכים בשפה טבעית.

לדוגמה:

```bash
Can you check if this answer is based on the documents?
```

זו לא פקודה טכנית. המשתמש לא אמר “הפעל retriever”, לא אמר “בדוק context”, ולא אמר “השווה בין התשובה לבין המקורות”. אבל ה-LLM יכול להבין שהכוונה היא לבדוק האם התשובה נשענת על מידע קיים ולא על ניחוש.

בדוגמה אחרת, המשתמש יכול לכתוב:

```bash
This explanation is too complicated. Make it easier for beginners.
```

כאן אין צורך בכלי חיצוני. ה-LLM מבין שמדובר בשכתוב ובהורדת רמת המורכבות של ההסבר.

לעומת זאת, אם המשתמש כותב:

```bash
What does our file say about LangChain?
```

ה-LLM צריך להבין שהשאלה אינה כללית. המשתמש לא שואל “מה זה LangChain”, אלא מבקש תשובה לפי קובץ מסוים. במקרה כזה צריך לשלוף מידע מהקובץ או ממאגר RAG, ורק אחר כך לנסח תשובה.

זו הסיבה שה-LLM חשוב כל כך בתוך Agent. הוא מאפשר למערכת לעבוד עם בקשות אנושיות, לא רק עם פקודות מובנות מראש.

בלי LLM, היינו צריכים להגדיר לכל תרחיש פקודה קשיחה, למשל:

```python
RUN_DOCUMENT_SEARCH(query="LangChain")
```

אבל משתמשים לא עובדים כך. הם שואלים, מתקנים, ממשיכים, מתייחסים למה שנאמר קודם, ולעיתים נותנים משימות לא מדויקות. ה-LLM מגשר בין השפה האנושית לבין הפעולות שהתוכנה יודעת לבצע.

עם זאת, חשוב לזכור שה-LLM לא צריך לעשות הכול לבד. התפקיד שלו הוא להבין, לפרש, להסביר ולעזור בתכנון. כאשר צריך מידע מדויק, נתון עדכני או פעולה חיצונית, ה-Agent צריך לחבר אותו לכלים ולמקורות מידע מתאימים.



## מה ה-Agent שולח ל-LLM

ה-LLM לא עובד לבד. הוא מקבל מה-Agent חבילת מידע שמטרתה לעזור לו להבין את המשימה ולהחזיר תשובה טובה יותר.

במקרה הפשוט ביותר, ה-Agent שולח רק את בקשת המשתמש:

```bash
User: Explain what an AI Agent is.
```

אבל במערכת אמיתית, בדרך כלל נשלח למודל יותר מזה. ה-Agent יכול לצרף כמה סוגי מידע:

- בקשת המשתמש.

- הנחיות מערכת.

- היסטוריית שיחה.

- מידע שנשלף ממערכת RAG.

- רשימת כלים זמינים.

- תוצאות של פעולות קודמות.

לדוגמה, אם המשתמש שואל:

```bash
What does the document say about ChromaDB?
```

ה-Agent לא אמור לשלוח למודל רק את השאלה. הוא צריך קודם לשלוף את הקטעים הרלוונטיים מהמסמכים, ואז לשלוח למודל משהו בסגנון:

```python
System instructions:
Answer only based on the provided context.

Context:
ChromaDB is an open-source embedding database...

User question:
What does the document say about ChromaDB?
```

במצב כזה, ה-LLM לא נדרש לנחש. הוא מקבל context רלוונטי ויכול לנסח תשובה שמבוססת על המידע שנשלף.

במקרה של Agent עם כלים, ה-Agent יכול לשלוח גם תיאור של הכלים הזמינים. לדוגמה:

```python
Available tool:
get_stock_info(symbol) - returns current stock information for a ticker symbol.
```

כאשר המשתמש שואל על מחיר מניה, ה-LLM יכול להבין שהכלי הזה רלוונטי. לאחר שהכלי מחזיר תוצאה, ה-Agent יכול לשלוח למודל גם את הפלט של הכלי, כדי שהמודל יסביר אותו בשפה ברורה.

לכן חשוב להבין שהשאלה האמיתית אינה רק “איזה מודל נבחר”, אלא גם “איזה מידע אנחנו שולחים למודל”. אותו LLM יכול להחזיר תשובה חלשה אם הוא מקבל מעט מדי הקשר, ותשובה הרבה יותר טובה אם ה-Agent מספק לו את ההנחיות, המקורות, הכלים והתוצאות הנכונות.

## דוגמה פשוטה

נניח שהמשתמש שואל:

```bash
What is the current stock price of Apple?
```

ה-LLM יכול להבין כמה דברים מתוך השאלה:

- המשתמש שואל על חברה בשם Apple.

- הכוונה היא כנראה למניה של Apple.

- הסימול הרלוונטי הוא בדרך כלל AAPL.

- מדובר בנתון עדכני, לא בשאלה תאורטית.

אבל כאן מגיעה נקודה חשובה: ה-LLM לא אמור לענות מהידע הפנימי שלו. מחיר מניה הוא מידע משתנה, ולכן תשובה שמבוססת רק על הזיכרון של המודל עלולה להיות שגויה.

במקרה כזה, תפקיד ה-LLM הוא לא “לדעת את המחיר”, אלא להבין שצריך להשתמש בכלי.

הזרימה יכולה להיראות כך:

```bash
User asks about Apple stock price
        ↓
LLM understands this requires current data
        ↓
Agent uses stock data tool
        ↓
Tool returns the current price
        ↓
LLM explains the result clearly
```

רואים כאן את ההפרדה החשובה בין הבנה לבין אימות נתונים. ה-LLM מבין את הבקשה ומנסח את התשובה, אבל הנתון עצמו מגיע מהכלי.

אותו עיקרון נכון גם בשאלות על מסמכים. אם המשתמש שואל:

```bash
What does our document say about ChromaDB?
```

ה-LLM יכול להבין שהשאלה קשורה למסמך, אבל הוא לא אמור להמציא מה כתוב בו. ה-Agent צריך לשלוף את הקטעים הרלוונטיים מהמסמך, ורק אז ה-LLM יכול להסביר אותם.

במילים פשוטות: בתוך Agent, ה-LLM הוא לא תמיד מקור התשובה. לפעמים הוא המוח שמבין מה צריך לעשות, והכלים או הזיכרון הם אלה שמביאים את המידע שעליו התשובה מתבססת.

## טעויות נפוצות

הטעות הראשונה היא לחשוב שה-LLM תמיד יודע נתונים עדכניים. בפועל, מודל שפה לא אמור להיות מקור אמין למחירי מניות, מזג אוויר, סטטוס משלוח, נתונים פיננסיים, מידע ממערכות פנימיות או כל דבר שמשתנה בזמן אמת.

אם המשתמש שואל:

```bash
What is the current price of Apple stock?
```

תשובה טובה לא אמורה להתבסס על ניחוש של המודל. צריך להפעיל כלי מתאים שמביא את הנתון ממקור חיצוני.

טעות נוספת היא לתת ל-LLM חופש פעולה גדול מדי. ככל שהמודל מחליט יותר דברים בעצמו, כך חשוב יותר להגדיר גבולות: אילו כלים מותר להפעיל, מתי צריך אישור משתמש, מה עושים במקרה של שגיאה, ואילו פעולות אסור לבצע אוטומטית.

לדוגמה, יש הבדל גדול בין כלי שקורא נתונים לבין כלי שמשנה נתונים. קריאת מחיר מניה היא פעולה יחסית בטוחה. שליחת מייל, מחיקת קובץ, פתיחת קריאה או עדכון רשומה במערכת דורשים הרבה יותר בקרה.

טעות שלישית היא לא לבדוק את הפלט. גם אם ה-LLM ניסח תשובה יפה, זה לא אומר שהתשובה נכונה. במערכת טובה צריך לבדוק אם התשובה מבוססת על context, אם הכלי החזיר תוצאה תקינה, ואם אין סתירה בין הנתונים לבין ההסבר.

טעות רביעית היא לבלבל בין ניסוח תשובה לבין אימות נתונים. ה-LLM מצוין בניסוח. הוא יודע להסביר, לפשט, לסכם ולארגן מידע. אבל אם צריך לוודא נתון, צריך מקור מתאים: מסמך, API, מסד נתונים, כלי חישוב או מערכת חיצונית.

זו הבחנה חשובה מאוד בבניית Agents. ה-LLM יכול לעזור למערכת להבין מה צריך לעשות, אבל הוא לא תמיד המקום שממנו צריכה להגיע האמת העובדתית.
