# פרק 2: תשתית הבסיס המודרנית (Reference Stack)

## שכבות המערכת (The System Layers)

בספר הנדסה טוב, הקוד לא אמור לרוץ "באוויר" או להסתמך על שירותי ענן יקרים שמשתנים כל שני וחמישי. כדי שתוכלו לבנות, לפרק ולהבין את המערכת באמת, אנחנו נבנה **Reference Stack** מלא. כולל סביבת ייחוס סטנדרטית שתלווה אותנו לאורך כל הספר.

הסטאק הזה נבחר בקפידה כדי לייצג את ה-State of the Art של 2026: הוא מודרני, מבוסס קוד פתוח (בליבת התשתית), וסקיילבילי. הוא ירוץ אצלכם על המחשב המקומי בתוך Docker, אבל הוא בנוי בדיוק כמו מערכת Production אמיתית.

**שכבת התשתית (Infrastructure Layer)**

הבסיס המוצק שעליו הכל עומד. כאן אנחנו לא ממציאים את הגלגל, אלא משתמשים בכלים המוכחים ביותר בתעשייה:

- **Docker & Docker Compose:** הסביבה שלנו היא קונטיינרית לחלוטין. אין "זה עובד אצלי במחשב". אם זה רץ בקונטיינר, זה ירוץ גם בענן.

- **PostgreSQL 16 + pgvector/pgvectorscale:** סוס העבודה הוותיק, בתוספת יכולות וקטוריות מתקדמות. למה לא Vector DB ייעודי ישר על ההתחלה? כי ב-2026, Postgres כבר עושה עבודה מדהימה בניהול וקטורים לצד דאטה רגיל (Metadata), וזה מפשט את הארכיטקטורה משמעותית.

- **Apache Kafka 3.7 (KRaft Mode):** מערכת העצבים לאירועים בזמן אמת. אנחנו משתמשים בגרסת KRaft המודרנית (ללא Zookeeper המסורבל), מה שהופך את ההרמה שלה לקלה ומהירה הרבה יותר.

- **Redis:** זיכרון לטווח קצר (Caching), ניהול Sessions מהיר, ומניעת כפילויות.

**שכבת איכות הנתונים (Data Quality Layer)**

השומרים בשער. אלו הכלים שמוודאים שהדאטה שנכנס למערכת הוא לא "זבל":

- **Great Expectations 1.x:** הסטנדרט דה-פקטו לבדיקות דאטה. נשתמש בו כדי להגדיר "חוזים" (Contracts) ולוודא שהמידע עומד בהם לפני שהוא מגיע למודל.

- **Pydantic v2:** הספרייה הכי חשובה בפייתון המודרני. היא תגדיר את הסכמות (Schemas) שלנו בקוד, תוודא טיפוסים (Types), ותספק ולידציה מהירה וחזקה לכל אובייקט מידע.

- **JSON Schema:** השפה המשותפת שבה נגדיר את המבנה של הנתונים, כך שגם המודל וגם הקוד יבינו בדיוק מה צפוי להגיע.

**שכבת הבינה המלאכותית (AI Layer)**

המוח של המערכת. כאן נשלב בין העוצמה של הענן לפרטיות של המקומי:

- **המודל הענני (Cloud Model):** נשתמש ב-API גנרי תואם OpenAI (כמו GPT-5-class) למשימות מורכבות של הסקה (Reasoning).

- **המודל המקומי (Local Model):** נריץ מודל קוד פתוח (כמו Llama-3/4 או Mistral) דרך Ollama. זה יאפשר לנו לבנות רכיבים שעובדים גם ללא אינטרנט, ופרטיים לחלוטין.

- **Embeddings:** נשתמש במודל sentence-transformers קל ומהיר ליצירת וקטורים באופן מקומי, עם אפשרות למעבר ל-API ענני למשימות כבדות יותר.

**שכבת הניטור (Observability Layer)**

העיניים של המערכת. בלי זה, אנחנו טסים בחושך:

- **Prometheus:** אוסף המטריקות הסטנדרטי. הוא יאסוף נתונים טכניים (CPU, Memory) וגם עסקיים (כמות טוקנים, זמן תגובה של מודל, מספר שגיאות ולידציה).

- **Grafana:** הלוח המחוונים (Dashboard) שיציג לנו את הכל בצורה ויזואלית יפה וברורה.

- **Structured Logging:** לא עוד סתם print לטרמינל. נבנה לוגים בפורמט JSON שמאפשרים חיפוש וניתוח (כמו ב-ELK Stack, אבל פשוט יותר).

זוהי הנבחרת שלנו. בפרק הבא נבין **למה** בחרנו דווקא בהם, ואיך הם מתחברים יחד למכונה משומנת אחת.



## עקרונות התכנון של התשתית

לפני שקופצים לקוד, חשוב להבין **למה** בחרנו דווקא בסט הכלים הזה, ואיך לחשוב עליו כתפיסה הנדסית ולא כרשימת טכנולוגיות אקראית. המטרה כאן היא לא "להכריח" אתכם לעבוד בדיוק עם אותם כלים, אלא לתת מסגרת חשיבה שתאפשר להחליף רכיבים בלי לפרק את כל המערכת.

**יציבות לפני טרנדים**

עולם ה AI מתחדש מהר, אבל מערכות פרודקשן זקוקות ליציבות. לכן הסטאק בספר נשען על רכיבים שהוכיחו את עצמם לאורך זמן:

- **PostgreSQL** כבסיס נתונים ראשי, עם הרחבות לעבודה וקטורית במקום עוד שירות צד שלישי מיותר.

- **Kafka** כעמוד שדרה לסטרימינג, בגרסת KRaft המודרנית, במקום פתרונות אקזוטיים שקשה למצוא להם מפעילים מנוסים.

**הרעיון הפשוט:** עדיף לבחור טכנולוגיה קצת פחות נוצצת, אבל כזו שיהיה לכם קל למצוא לה דוגמאות, מדריכים וקהילה פעילה גם בעוד חמש שנים.

**אי-תלות בספק בודד**

היום שירות ענן אחד מוביל, מחר יוביל אחר. אם כל המערכת שלכם צמודה ל API ייחודי של ספק יחיד, אתם מאבדים חופש תנועה. לכן התכנון כאן שם דגש על:

- שימוש בפרוטוקולים ופורמטים סטנדרטיים (HTTP, JSON, SQL).

- שכבות הפשטה ברורות בין הקוד העסקי לבין ספקי המודלים, הווקטורים והניטור.

**המבחן הפשוט:** אם תרצו להחליף למשל את Chroma ב-Qdrant, או את Prometheus בפלטפורמה מנוהלת - תצטרכו לשנות קונפיגורציה וקצת קוד סביב, אבל לא לכתוב את כל המערכת מחדש.

**קוד פתוח כברירת מחדל**

התשתית מבוססת ככל האפשר על כלים פתוחים:

- קל יותר להבין מה קורה "מתחת למכסה המנוע".

- אפשר להריץ את הכל מקומית, בלי חשבון ענן ובלי הפתעות בחשבונית.

- קל לעבור בין סביבות: לוקאלי, סטייג'ינג, פרודקשן.

זה לא אומר שלא משתמשים בשירותים מנוהלים או ב-SaaS, אבל ברירת המחדל בספר היא שכל רכיב קריטי יהיה ניתן להרצה אצלכם, בקוד פתוח, כדי שתוכלו ללמוד עליו לעומק.

**איך מחליפים רכיב בלי לשבור את המערכת?**

כדי להפוך את התשתית לגמישה, כל רכיב מוגדר לפי **התפקיד** שלו, לא לפי השם המסחרי. כמה דוגמאות:

- "אחסון וקטורים" - יכול להיות pgvector, Qdrant או Pinecone. כל עוד הוא יודע לקבל embedding, לשמור אותו, ולהחזיר תוצאות חיפוש - המערכת תעבוד.

- "איסוף מטריקות" - יכול להיות Prometheus או שירות ענן אחר. מה שחשוב הוא שתקבלו חיווי עקבי על זמן תגובה, עלויות ושגיאות.

- "מודל ענני" - יכול להיות GPT 5 היום ומודל אחר מחר, כל עוד יש לו API תואם לצורת הקריאה שבחרתם.

בפרקים הבאים, בכל מקום שנשתמש בכלי ספציפי, נדגיש גם את **החוזה** שהוא מממש - כדי שתדעו מה אפשר להחליף, ואיך לעשות זאת בצורה בטוחה.

## סביבת העבודה: GitHub ו Docker

כדי שהספר הזה יהיה לא רק מעניין אלא גם **שמיש**, כל הרעיונות והדוגמאות חייבים לרוץ אצלך מקומית, בלי מאבק אינסופי בהתקנות ותלויות. בשביל זה נגדיר סביבת עבודה ברורה אחת, שתשמש כבסיס לכל הפרקים.

**מבנה הריפו (GitHub Repository)**

הריפו של הספר בנוי כך שכל פרק מקבל "מגרש משחקים" משלו, אבל כולם חולקים תשתית משותפת:

```bash
ai-data-engineering-playbook/

│ │ ├── patterns/

│ │ ├── configs/
│ │ └── lab/

│ └── ...

│ ├── pipelines/

```

הרעיון פשוט:

- לכל פרק יש תיקייה עם קוד ודוגמאות.

- כל התשתיות הכבדות חיות תחת docker/, כדי שלא תצטרך להרים הכל מחדש בכל פרק.

- תיקיית templates/ מחזיקה קטעי קוד ותצורה שנמחזר כמה פעמים לאורך הספר.

**שלוש סביבות Docker**

במקום להעמיס עליך מערכת מלאה מהיום הראשון, נבנה את הסביבה בשלושה שלבים:

1. **foundation/** - בסיס

 - Postgres

 - שירות להרצת קוד (למשל אפליקציית פייתון פשוטה או Jupyter)

 - מספיק כדי לתרגל Contracts, בדיקות דאטה ו Pipelines ראשונים.

1. **streaming/** - הוספת סטרימינג

 - כל מה שיש ב-foundation

 - Kafka (ב KRaft) וצרכנים פשוטים

 - כאן נתנסה בזרימת אירועים בזמן אמת וב Validation "על הדרך".

2. **production/** - הסטאק המלא

 - כל מה שיש ב-streaming

 - Prometheus, Grafana

 - מודל מקומי (דרך Ollama)

 - רכיבי עזר נוספים לפי הצורך

 - זו הסביבה שבה נריץ את הקייס המלא של LogiSmart.



**Quick Start - הרצה ראשונה**

כדי שכל הדוגמאות בספר יעבדו אצלכם בלי כאב ראש, נתחיל בסביבה הכי בסיסית שאפשר:

PostgreSQL + ממשק נוח לדאטה (Adminer), רצים בתוך Docker.

**שלב 1: שכפול הריפו**

בטרמינל (או PowerShell ב Windows):

```bash
git clone https://github.com/tomkedem/ai-data-engineering-playbook.git
cd ai-data-engineering-playbook
```

**שלב 2: מעבר לתיקיית foundation**

```bash
cd docker/foundation
```

בתיקייה הזו כבר מחכה לכם הקובץ:

docker-compose.yml

עם ההגדרה הבסיסית הבאה:

```yaml
services:
  postgres:
    image: postgres:16
    container_name: ai_book_postgres
    environment:
      POSTGRES_USER: ai_user
      POSTGRES_PASSWORD: ai_password
      POSTGRES_DB: ai_db
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  adminer:
    image: adminer:latest
    container_name: ai_book_adminer
    ports:
      - "8080:8080"
    depends_on:
      - postgres

volumes:
  postgres_data:
```

אין צורך לערוך אותו בשלב זה.

**שלב 3: הרמת הסביבה**

ודאו ש Docker Desktop רץ, ואז:

docker compose up -d

אם הכול תקין, תראו ב Docker Desktop שני קונטיינרים רצים: postgres ו adminer.

**שלב 4: התחברות למסד הנתונים**

פתחו דפדפן וגלשו לכתובת:

http://localhost:8080

מלאו את השדות ב Adminer:

- מערכת :(System) PostgreSQL

- שרת :( Server) postgres

- שם משתמש: ai_user

- סיסמה: ai_password

- מסד נתונים :( Database) ai_db

לחצו על "התחבר". 
אם אתם רואים את מסד הנתונים ai_db עם סכמה public ריקה - ההקמה הצליחה.

ל-commit (עם הקובץ + הוראות ההרצה):

text

chore: add foundation docker environment

מכאן והלאה, בכל פעם שנלמד דפוס חדש (Contracts, Streaming, RAG, Agents), נוסיף עליו שכבה חדשה - אבל תמיד נשמור על אותה סביבת עבודה מוכרת.
