# Context, State וזרימות עבודה

## מהו Context במערכת MCP ולמה הוא לא רק טקסט

כשמתכנתים מתחילים לעבוד עם מודלים, לרוב הם חושבים על Context כעל הטקסט שנשלח למודל. היסטוריית השיחה, הפרומפט, תוצאות הכלים. אבל במערכת ,MCP Context הוא מושג רחב יותר ומורכב יותר.

Context במערכת MCP הוא כל המידע שהמודל צריך כדי לקבל החלטה נכונה בנקודת זמן מסוימת. זה כולל את הטקסט, אבל גם דברים שלא נכתבים במפורש.

**יש שלושה סוגי מידע שמרכיבים את ה-Context:**

1. **מידע טקסטואלי** הוא מה שרוב המתכנתים מכירים. בקשת המשתמש, היסטוריית השיחה, תוצאות כלים שנקראו, ומסמכים שנשלפו. זה המידע שנראה בבירור בחלון ה-Context של המודל.

2. **מידע על יכולות זמינות** הוא כולל את רשימת הכלים והמשאבים שה-Host חושף למודל בהקשר הנוכחי. המודל לא יכול לקרוא לכלי שלא מופיע ברשימה הזו, גם אם הוא יודע שהכלי קיים. מה שמופיע ברשימה הוא חלק מה-Context.

3. **מידע תפעולי** הוא ההקשר שלעיתים קרובות חסר ויוצר בעיות. מי המשתמש שמבקש את הפעולה, בסביבה מה פועלים, אילו משאבים נעולים כרגע, ומה המצב של המערכת ברגע הזה. מידע תפעולי שלא מועבר למודל גורם לו להניח הנחות, וכבר ראינו בפרק 2 לאן הנחות מובילות.

ההבנה ש-Context הוא לא רק טקסט משנה את האופן שבו מתכננים מערכת. במקום לחשוב "איזה טקסט לשלוח למודל", חושבים "מה המודל צריך לדעת כדי לפעול נכון", ומוודאים שכל המידע הזה מגיע אליו בצורה מבוקרת.



## מהו State ולמה צריך להיזהר איתו

State הוא כל מידע שצריך להישמר בין קריאה אחת לקריאה הבאה. זה יכול להיות תוצאה של כלי שהמודל יצטרך בהמשך, החלטה שהתקבלה בשלב קודם, או מצב של תהליך שמתפרס על פני מספר שלבים.

ההבדל בין Context ל-State הוא נקודת הזמן. Context הוא מה שקיים עכשיו, בקריאה הנוכחית. State הוא מה שצריך לעבור מקריאה אחת לשנייה.

**למה State הוא מקור לבעיות**

State שמנוהל בצורה לא מסודרת הוא אחד הגורמים הנפוצים ביותר לכשלים במערכות MCP.

**להלן מספר דפוסים שחוזרים שוב ושוב:**

**State שנשמר בלי תאריך תפוגה**

מידע שנשמר בתחילת Session ממשיך להיות בתוקף גם כשהוא כבר לא רלוונטי. המודל מקבל החלטות על בסיס מידע ישן.

**State שמשותף בין משתמשים שונים**

מידע שנשמר ברמת השרת ולא ברמת ה-Session עלול להיות גלוי למשתמשים שלא אמורים לראות אותו.

**State שלא מנוקה כשה-Session נגמר.**

מידע שנצבר לאורך זמן תופס זיכרון ועלול להכיל נתונים רגישים שאין צורך לשמור.

**State שמניח עקביות שלא קיימת.**

שרת שמניח שהמודל זוכר מה קרה לפני שלושה שלבים, בלי שהמידע הזה הועבר מפורשות.

**כלל הבסיס**

**כל State שנשמר צריך לענות על ארבע שאלות:**

1. מי יכול לגשת אליו

2. כמה זמן הוא תקף

3. מה קורה כשהוא פג תוקף

4. ומי אחראי לנקות אותו

State שלא עונה על ארבע השאלות האלה הוא State שיגרום לבעיות.



## Stateful מול Stateless: המחיר של כל בחירה

כשמתכננים מערכת MCP, אחת ההחלטות הארכיטקטוניות המשמעותיות ביותר היא האם השרת שומר State בין קריאות או לא. לכל בחירה יש יתרונות וחסרונות, והטעות הנפוצה היא לבחור על בסיס מה שנראה פשוט יותר למימוש, לא על בסיס מה שמתאים לצרכי המערכת.

**Stateless**

בגישה Stateless, כל קריאה לשרת עצמאית לחלוטין. היא מכילה את כל המידע שהיא צריכה ולא מסתמכת על מידע ששמור מקריאות קודמות. השרת לא זוכר כלום בין קריאה לקריאה.

היתרונות ברורים: פשטות מוחלטת בניהול השרת, קל להרחיב כי כל קריאה יכולה להתבצע על ידי כל instance של השרת, וקל לבדוק כי כל קריאה היא עולם בפני עצמו. החיסרון הוא שכל קריאה נושאת איתה יותר מידע, וכשיש תהליכים שמתפרסים על פני כמה שלבים, ה-Host צריך לנהל את ההקשר המלא ולהעביר אותו בכל קריאה.

**Stateful**

בגישה Stateful, השרת שומר מידע בין קריאות. Session שנפתח בתחילת אינטראקציה שומר נתונים שזמינים לכל קריאה שאחריו.

**היתרונות:** קריאות קצרות יותר כי לא צריך להעביר את כל ההקשר בכל פעם, ותמיכה טבעית בתהליכים ארוכי טווח. **החסרונות:** מורכבות ניהולית גבוהה יותר, קשה יותר להרחיב כי Session קשור ל-instance ספציפי של השרת, וצריך לנהל מחזור חיים של ה-State כולל ניקוי, תפוגה, ואבטחה.

**איך בוחרים**

**השאלה המרכזית היא:** האם יש תהליכים במערכת שמתפרסים על פני כמה קריאות ושלא ניתן לשחזר את ההקשר שלהם בקלות?

אם כן, Stateful עשוי להיות הבחירה הנכונה.

אם כל קריאה יכולה להיות עצמאית עם המידע שה-Host מעביר לה, Stateless הוא הבחירה הפשוטה והאמינה יותר.

בפועל, מערכות רבות משלבות את שתי הגישות: ליבת השרת Stateless, עם שכבת Session ייעודית שמנהלת את ה-State הנדרש בנפרד.



## Request Scope, Session Scope, Workflow Scope: מה חי כמה זמן

לא כל מידע צריך לחיות אותו פרק זמן. אחת הטעויות הנפוצות בניהול State היא לשמור את הכל באותה רמה, בלי להבחין בין מידע שרלוונטי לקריאה אחת בלבד לבין מידע שצריך לשרוד לאורך תהליך שלם.

יש שלוש רמות של חיי State שכדאי להכיר.

1. **Request Scope**

מידע ברמת Request חי רק בזמן הקריאה הספציפית ומשתחרר מיד אחריה. זה כולל פרמטרי הקלט של הקריאה, תוצאות ביניים שחושבו בזמן הטיפול בה, ומשאבים שנפתחו לצורך הקריאה כמו חיבורים למסד נתונים. מידע ברמה זו לא נשמר בשום מקום אחרי שהקריאה מסתיימת.

2. **Session Scope**

מידע ברמת Session חי לאורך כל האינטראקציה של משתמש אחד, מהרגע שה-Host מתחבר לשרת ועד שהחיבור נסגר. זה כולל פרטי זהות המשתמש, הרשאות שנבדקו בתחילת ה-Session, והעדפות שהמשתמש הגדיר. מידע ברמה זו נשמר בשרת או ב-Host לאורך ה-Session ומשתחרר כשה-Session נסגר.

3. **Workflow Scope**

מידע ברמת Workflow חי לאורך תהליך עבודה מוגדר שיכול להתפרס על פני כמה Sessions. לדוגמה, תהליך אישור בקשה שמתחיל ביום אחד וממשיך ביום השני, או תהליך עיבוד מסמך שנשמר בין הפעלות. מידע ברמה זו חייב להישמר במסד נתונים או בשכבת אחסון עמידה, עם תאריך תפוגה מוגדר ומנגנון ניקוי.

**למה ההבחנה חשובה בפועל**

שמירת מידע ברמה גבוהה יותר מהנדרש יוצרת בעיות מיותרות. מידע שצריך לחיות רק ברמת Request ששמור ברמת Session תופס זיכרון, עלול לגרום לדליפת מידע בין קריאות, ומקשה על הבנת מה המערכת עושה. לעומת זאת, מידע שצריך לחיות ברמת Workflow ששמור רק ברמת Session יאבד כשה-Session נסגר.

כשמגדירים State חדש במערכת, השאלה הראשונה צריכה להיות: לאיזו רמה המידע הזה שייך? התשובה קובעת איפה שומרים אותו, כמה זמן הוא תקף, ומי אחראי לנקות אותו.



## אילו נתונים כדאי לשמור ואילו לא

לא כל מידע שנוצר במהלך עבודת המערכת צריך להישמר. ההחלטה מה לשמור ומה לזרוק היא אחת ההחלטות החשובות בתכנון מערכת MCP, ולרוב מתכנתים לא מקדישים לה מספיק תשומת לב עד שמשהו משתבש.

**מה כדאי לשמור**

מידע שנדרש לשחזר תהליך אחרי כשל. אם ה-Session נסגר באמצע תהליך ארוך, מה צריך לשמור כדי שאפשר יהיה להמשיך ממקום שהפסקת ולא מההתחלה?

מידע שדרוש לביקורת. כל פעולה שמשנה מצב במערכת צריכה להשאיר עקבות: מי ביצע, מתי, עם אילו פרמטרים, ומה הייתה התוצאה. זה לא State לצורך פעולה, זה תיעוד לצורך בקרה.

מידע שיקר לחשב מחדש. אם שאילתה למסד נתונים לוקחת חמש שניות ותוצאתה לא משתנה תכופות, שמירתה ב-Cache לזמן קצר היא החלטה סבירה.

**מה לא כדאי לשמור**

מידע שאפשר לחשב מחדש בקלות. אם תוצאה נגזרת ישירות מהקלט ולא תלויה בשום מצב חיצוני, אין טעם לשמור אותה.

מידע רגיש שלא נדרש אחרי הקריאה. סיסמאות, טוקנים, ופרטי תשלום לא צריכים להישמר מעבר לרגע שהם נדרשים לביצוע הפעולה.

מידע שגילו חייב להיות מוגבל. נתון שמשתנה תכופות ושהמערכת צריכה לעבוד תמיד עם הגרסה העדכנית שלו לא אמור להישמר ב-Cache ארוך טווח.

**כלל פשוט לקבלת החלטה**

לפני שמוסיפים שמירה של מידע, שאל: מה יקרה אם המידע הזה לא יישמר? אם התשובה היא "כלום", אל תשמור. אם התשובה היא "הפעולה תיכשל" או "המשתמש יצטרך להתחיל מחדש", שמור ותגדיר כמה זמן.



## Tasks: סוג יכולת חדש לפעולות אסינכרוניות

עד עכשיו כל הכלים שדיברנו עליהם עבדו בצורה סינכרונית. המודל קורא ל-Tool, השרת מבצע את הפעולה, ומחזיר תוצאה. הכל קורה בתוך אותה קריאה.

אבל יש פעולות שלוקחות זמן רב. עיבוד קובץ גדול, הרצת דוח מורכב, שליחת אצווה של הודעות, או כל תהליך שנמשך דקות ולא שניות. עבור פעולות כאלה, הגישה הסינכרונית לא עובדת. המודל לא יכול לחכות דקות לתשובה, וה-Connection עלול להתנתק באמצע.

לכך נועדה היכולת(פרימיטיב) ה-Tasks שנוסף ל-MCP.

**איך Tasks עובד**

במקום שהמודל יחכה לתוצאה, הוא קורא ל-Tool שמחזיר מיד Task ID. המודל ממשיך הלאה ובשלב מאוחר יותר קורא ל-Tool אחר עם ה-Task ID כדי לבדוק את סטטוס התהליך ולקבל את התוצאה כשהיא מוכנה. זה הדפוס של call-now, fetch-later.

```python
@server.list_tools()
async def list_tools():
    return [
        Tool(
            name="start_report_generation",
            description="""Starts generating a report asynchronously.
            Returns a task_id immediately.
            Use get_task_status to check progress and retrieve the result.""",
            inputSchema={
                "type": "object",
                "properties": {
                    "report_type": {
                        "type": "string",
                        "enum": ["sales", "inventory", "customers"]
                    },
                    "date_from": {
                        "type": "string",
                        "format": "date"
                    },
                    "date_to": {
                        "type": "string",
                        "format": "date"
                    }
                },
                "required": ["report_type", "date_from", "date_to"]
            }
        ),
        Tool(
            name="get_task_status",
            description="""Checks the status of an async task.
            Returns status (pending, running, completed, failed) and result if completed.""",
            inputSchema={
                "type": "object",
                "properties": {
                    "task_id": {
                        "type": "string",
                        "description": "Task ID returned by an async operation"
                    }
                },
                "required": ["task_id"]
            }
        )
    ]

@server.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "start_report_generation":
        report_type = arguments.get("report_type")
        date_from = arguments.get("date_from")
        date_to = arguments.get("date_to")

        # Generate unique task ID
        task_id = f"task_{report_type}_{date_from}_{date_to}"

        # Store initial task state
        task_store[task_id] = {
            "status": "pending",
            "report_type": report_type,
            "created_at": datetime.now().isoformat()
        }

        # Start async processing in background
        asyncio.create_task(
            generate_report(task_id, report_type, date_from, date_to)
        )

        return [
            TextContent(
                type="text",
                text=json.dumps({
                    "task_id": task_id,
                    "status": "pending",
                    "message": "Report generation started. Use get_task_status to check progress."
                }, indent=2)
            )
        ]

    if name == "get_task_status":
        task_id = arguments.get("task_id")
        task = task_store.get(task_id)

        if not task:
            return [
                TextContent(
                    type="text",
                    text=f"Task not found: {task_id}"
                )
            ], True

        return [
            TextContent(
                type="text",
                text=json.dumps(task, indent=2)
            )
        ]

    raise ValueError(f"Tool not found: {name}")

async def generate_report(task_id: str, report_type: str, date_from: str, date_to: str):
    """Background task that generates the report."""
    try:
        task_store[task_id]["status"] = "running"

        # Simulate report generation
        await asyncio.sleep(5)
        result = {"rows": 150, "total_sales": 45000}

        task_store[task_id]["status"] = "completed"
        task_store[task_id]["result"] = result
        task_store[task_id]["completed_at"] = datetime.now().isoformat()

    except Exception as e:
        task_store[task_id]["status"] = "failed"
        task_store[task_id]["error"] = str(e)
```

**מה חשוב להגדיר בכל Task:**

כל Task צריך סטטוסים ברורים שהמודל מבין: pending, running, completed, failed. כל Task צריך תאריך יצירה ותאריך תפוגה, כדי שאפשר לנקות Tasks ישנים. וכל Task שנכשל צריך להחזיר הסבר ברור מה השתבש.

## איך זרימות עבודה נשברות כשאין מבנה ברור

זרימת עבודה היא רצף של פעולות שמתבצעות בסדר מוגדר כדי להשיג תוצאה. בניית הזמנה, עיבוד בקשת אשראי, אישור מסמך. כל אחת מהן כוללת מספר שלבים שתלויים זה בזה.

כשזרימת עבודה מתוכננת בצורה לא נכונה, היא נשברת בדרכים שקשה לאתר ועוד יותר קשה לתקן.

**כשל 1: תלות שלא מוצהרת**

מודל שמבצע שלב שני מניח שהשלב הראשון הצליח. אבל אם השלב הראשון נכשל בשקט, השלב השני מתבצע על בסיס נתונים לא נכונים. לדוגמה, מודל שמנסה לאשר הזמנה שלא נוצרה עדיין כי שלב היצירה נכשל בלי לדווח על כך.

**הפתרון:** כל Tool שמייצר נתון שצעדים הבאים תלויים בו צריך להחזיר מזהה ברור שה-Tool הבא מקבל כקלט. אם ה-Tool הראשון לא החזיר מזהה, ה-Tool השני לא אמור לרוץ.

**כשל 2: State חלקי**

זרימה שנקטעת באמצע משאירה את המערכת במצב חלקי. חלק מהפעולות בוצעו, חלק לא. מודל שמנסה להריץ את הזרימה מחדש עלול לבצע פעמיים את הפעולות שכבר הצליחו.

**הפתרון:** כל זרימת עבודה שמתפרסת על פני מספר שלבים צריכה לשמור את מצבה. כל שלב מסמן את עצמו כהושלם לפני שממשיכים לשלב הבא. אם הזרימה מופעלת מחדש, היא ממשיכה מהשלב האחרון שהושלם.

**כשל 3: אין נקודת כשל מוגדרת**

כשזרימה כוללת כמה Tools ואחד מהם נכשל, מה קורה? אם אין תשובה מוגדרת לשאלה הזו, כל כשל הופך לבעיה שצריך לפתור מחדש בכל פעם.

**הפתרון:** לכל זרימת עבודה צריכה להיות מדיניות כשל ברורה. האם עוצרים ומדווחים? האם מנסים שוב? האם מבצעים Rollback לשלבים שכבר הושלמו? המדיניות הזו לא יכולה להיות שאלה שהמודל מחליט עליה בזמן אמת.

## איך בונים רצף פעולות בלי לאבד שליטה

אחרי שהבנו איך זרימות עבודה נשברות, הגיע הזמן לדבר על איך בונים אותן נכון.

**הכלל המרכזי פשוט:** כל שלב בזרימה צריך להיות עצמאי, מוגדר, ובר אימות לפני שממשיכים לשלב הבא.

**עקרון 1: כל שלב מחזיר אישור מפורש**

Tool שהצליח מחזיר תוצאה שמאשרת מה בדיוק קרה. לא רק "הצלחה", אלא מזהה, סטטוס, וכל מידע שהשלב הבא יצטרך. Tool שנכשל מחזיר שגיאה ברורה שמפסיקה את הזרימה.

```python
@server.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "create_order":
        customer_id = arguments.get("customer_id")
        items = arguments.get("items", [])

        # Validate before creating
        if not items:
            return [
                TextContent(
                    type="text",
                    text="Cannot create order: items list is empty"
                )
            ], True

        order_id = order_service.create(customer_id, items)

        # Return explicit confirmation with all data the next step needs
        return [
            TextContent(
                type="text",
                text=json.dumps({
                    "success": True,
                    "order_id": order_id,
                    "customer_id": customer_id,
                    "items_count": len(items),
                    "next_step": "Use confirm_order with this order_id to proceed"
                }, indent=2)
            )
        ]
```



**עקרון 2: כל שלב בודק את תנאי הכניסה שלו**

Tool שמקבל מזהה מהשלב הקודם מאמת שהמזהה קיים ושה-Entity במצב הנכון לפני שהוא עושה כל דבר אחר.

```python
    if name == "confirm_order":
        order_id = arguments.get("order_id")

        # Verify preconditions before proceeding
        order = order_service.get(order_id)

        if not order:
            return [
                TextContent(
                    type="text",
                    text=f"Cannot confirm: order {order_id} does not exist"
                )
            ], True

        if order["status"] != "pending":
            return [
                TextContent(
                    type="text",
                    text=f"Cannot confirm: order {order_id} is already {order['status']}"
                )
            ], True

        order_service.confirm(order_id)

        return [
            TextContent(
                type="text",
                text=json.dumps({
                    "success": True,
                    "order_id": order_id,
                    "status": "confirmed",
                    "next_step": "Use process_payment with this order_id to complete the purchase"
                }, indent=2)
            )
        ]
```



**עקרון 3: הזרימה מתועדת בתיאור הכלים**

כל Tool שהוא חלק מזרימה מציין בתיאור שלו מה מגיע לפניו ומה אחריו. המודל שקורא ל-Capability Negotiation מבין מה הסדר הנכון בלי לנחש.

```python
Tool(
    name="confirm_order",
    description="""Confirms a pending order.
    Prerequisites: Order must exist and be in 'pending' status.
    Call create_order first to get a valid order_id.
    Next step: Call process_payment with the confirmed order_id.""",
    inputSchema={
        "type": "object",
        "properties": {
            "order_id": {
                "type": "string",
                "description": "Order ID returned by create_order"
            }
        },
        "required": ["order_id"]
    }
)
```

**עקרון 4: מדיניות כשל מוגדרת מראש**

לפני שבונים זרימה, מגדירים מה קורה כשכל שלב נכשל. האם עוצרים? האם מנסים שוב? האם מבצעים Rollback? התשובות האלה נכתבות בקוד, לא נשארות כשאלות פתוחות שהמודל יצטרך להחליט עליהן בזמן אמת.



## תרגול

בחר תהליך עבודה שאתה מכיר מהמערכות שאתה עובד עליהן. זה יכול להיות תהליך יצירת הזמנה, אישור בקשה, עיבוד מסמך, או כל רצף פעולות שכולל יותר משלב אחד.

**חלק א: מיפוי הזרימה**

רשום את השלבים של התהליך שבחרת וענה על השאלות הבאות לכל שלב:

1. מה הקלט שהשלב הזה צריך?

2. מה הפלט שהשלב הזה מחזיר לשלב הבא?

3. באיזה מצב חייבת להיות המערכת כדי שהשלב הזה יוכל לרוץ?

4. מה קורה אם השלב הזה נכשל?

**חלק ב: זיהוי State**

לכל מידע שעובר בין השלבים, ענה:

1. ברמת איזה Scope הוא שייך: Request, Session, או Workflow?

2. כמה זמן הוא צריך להיות תקף?

3. מי אחראי לנקות אותו?

**חלק ג: זיהוי סיכונים**

ענה על השאלות הבאות:

1. איזה שלב עלול לגרום לנזק אם יתבצע פעמיים?

2. איזה שלב עלול לקחת יותר זמן ממצופה ודורש טיפול אסינכרוני?

3. אם הזרימה נקטעת בין שלב שני לשלישי, מה מצב המערכת ואיך ממשיכים?

**חלק ד: עדכון הפרויקט**
